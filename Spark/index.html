PySpark performance

Spark internals · execution · tuning

Fourteen connected lessons on how Apache Spark actually runs your PySpark — told as one story. Your code becomes jobs, stages, and tasks across a driver and executors; joins and wide operations trigger the expensive shuffle; memory decides what runs, spills, or OOMs; and the final section teaches how to debug, size, and explain the plan like an FDE. Each lesson is a self-contained interactive page with a runnable notebook and a markdown companion.

How to use this track

Read it in order — it's one story, not a menu

The track follows a Spark job from how it runs to how you make it fast: read the plan, avoid the shuffle, fix skew, reuse work, tune memory and GC, then debug and size production jobs. Parts A → B → C → D build on each other, so the intended path is 01 through 14. Each page stands alone if you need to jump straight to a topic.

→ Full learning plan (suggested order, timings & self-check questions)

The one-line tuning method Slow Spark job? Read the plan (.explain()) and the Spark UI first. Then, in order: read less (partition pruning / DPP, column pruning) → avoid or repair the shuffle (broadcast the small side, bucket repeated joins, let AQE coalesce) → fix skew (AQE skew join, then salting) → reuse work (cache only reused DataFrames) → size memory and GC last. Always verify the change in .explain() / the Spark UI — never assume.
