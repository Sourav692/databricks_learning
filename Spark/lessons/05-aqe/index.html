The plan that
rewrites itself.

Adaptive Query Execution waits until a shuffle finishes, reads the real data sizes, and re-optimizes the rest of the plan — coalescing tiny partitions, splitting skewed ones, and flipping a sort-merge join to a broadcast at runtime.

Spark 3.2+ / DBR 7.3+ Advisory size 64 MB Skew 5× & 256 MB Verified Jun 2026 docs

What it is

Adaptive Query Execution (AQE) is Spark re-optimizing the physical plan at runtime, using the actual statistics it collects after each shuffle — instead of trusting the static, estimate-based plan Catalyst built before the job ran.

A non-adaptive plan is decided once, from table stats and file sizes that are often wrong (a filter shrinks a table; a groupBy makes uneven partitions; a join side turns out tiny after a WHERE). AQE waits until a shuffle stage finishes, reads the real per-partition sizes, and rewrites the rest of the plan. It does three things in open-source Spark (a fourth on Databricks):

  • Coalesce post-shuffle partitions — merge hundreds of tiny partitions into a few right-sized (~64 MB) ones.
  • Split skewed partitions — break a partition far bigger than its peers into balanced sub-partitions.
  • Switch the join strategy — flip a planned sort-merge join to a broadcast hash join when a side turns out small.
  • (Databricks only) Detect & propagate empty relations — prune branches fed by a zero-row stage.
The one rule to remember AQE needs real shuffle statistics to act, so it only re-optimizes at stage boundaries (after a shuffle). It is on by default — leave it on, and let it coalesce / split / switch before you hand-tune spark.sql.shuffle.partitions.

Why it matters

  • It fixes the two most common shuffle pathologies automatically — "too many tiny partitions" (200 default partitions → 200 wasteful tasks) and "one giant skewed partition" (a hot key → one straggler task) — with no code changes.
  • Static estimates are routinely wrong. Catalyst plans from stats gathered before filters run. A 5 GB dimension a WHERE shrinks to 8 MB is still planned as a sort-merge shuffle — until AQE sees the real 8 MB and broadcasts it.
  • It's the modern default answer to "my shuffle is slow." Before manual repartition, a hand-set shuffle.partitions, or salting, AQE has usually already coalesced, split, and switched for you.
  • It composes with the rest of the track. AQE's runtime join switch backs Lesson 02's broadcast joins; its skew split is the first defence before Lesson 08's salting; Photon runs underneath AQE's adaptive plan — complementary, not alternatives.

How it works — deep dive

How AQE re-optimizes: the framework

analogy A non-adaptive plan is a route printed before you leave; AQE is live GPS that reroutes you at every junction once it sees the actual traffic.

  • Mechanism: Spark splits the plan into query stages at shuffle (and broadcast) boundaries. It runs the leaf stages, collects exact map-output statistics (per-partition byte sizes), then re-optimizes the remaining plan with those real numbers before submitting the next stage — repeating at every boundary.
  • Why it needs a shuffle: the statistics AQE relies on are the shuffle's per-partition sizes. No shuffle → no runtime stats → nothing to adapt.
  • Trade-off: a tiny planning cost per boundary in exchange for plans built on truth. On by default since Spark 3.2.0 — AQE was introduced in 1.6.0 but only default-true in 3.2.0 (SPARK-33679). Don't conflate the two.

1 · AQE Coalesce — merge tiny shuffle partitions

analogy Instead of mailing 200 near-empty envelopes, AQE combines them into a few full ones — fewer trips, same contents.

  • Problem: spark.sql.shuffle.partitions = 200, so a shuffle on a modest result makes 200 partitions, many a few KB — 200 tasks, most doing nothing.
  • Mechanism: with coalescePartitions.enabled = true, AQE reads real post-shuffle sizes and merges adjacent partitions until each is about advisoryPartitionSizeInBytes (64 MB), never below minPartitionSize (1 MB).
  • Trade-off: slightly less parallelism (fewer, bigger tasks) — almost always a win; the 64 MB advisory keeps tasks reasonable even on big results.

use case A daily aggregation outputting a few hundred MB: without AQE it runs 200 tasks / writes 200 tiny files; with coalesce it runs ~5 balanced tasks and writes ~5 files.

2 · AQE Split Partitions — break up skew

analogy If one checkout lane has 500 shoppers and the others have 5, AQE opens extra lanes for that one queue so everyone finishes together.

  • Problem (skew): uneven keys mean a few post-shuffle partitions are enormous. The huge one becomes a straggler task — minutes long (maybe spilling/OOM) while every other task finished. The stage waits on it.
  • Mechanism: with skewJoin.enabled = true, AQE flags a partition skewed when it is both > skewedPartitionFactor (5.0) × the median and > skewedPartitionThresholdInBytes (256 MB). It then splits that partition into sub-partitions (replicating the matching side) so the join stays correct and balanced.
  • Why both conditions: the factor catches relative skew; the 256 MB floor avoids splitting partitions that are merely a bit above a tiny median.
  • Trade-off: some replication of the matching side, but one multi-minute straggler becomes parallel sub-tasks. Sort-merge joins only; skew it can't tame → salting (Lesson 08).

use case transactions ⋈ accounts where one "house" account holds 40% of rows — that account_id partition is 5× the median and well over 256 MB, so AQE splits it.

3 · AQE Joins Strategy — switch sort-merge → broadcast at runtime

analogy You packed for a long road trip (sort-merge), then learn the destination is round the corner — so you ditch the heavy plan and walk (broadcast).

  • Problem: Catalyst chose sort-merge from pre-execution estimates, but an upstream filter shrank one side below the broadcast bar — it's about to shuffle two big tables for nothing.
  • Mechanism: after the shuffle feeding the join, AQE sees the actual size of each side and, if one is now small, rewrites the join to a Broadcast Hash Join — using a runtime broadcast threshold (distinct from the static autoBroadcastJoinThreshold Catalyst used up front).
  • OSS vs Databricks: OSS uses the runtime switch tied to the standard threshold logic; Databricks exposes a higher dedicated switch spark.databricks.adaptive.autoBroadcastJoinThreshold = 30 MB (vs the OSS static 10 MB spark.sql.autoBroadcastJoinThreshold).
  • Trade-off: you swap the double shuffle for a broadcast — fires only when the real size is genuinely small, bounded by the small side fitting in memory.

use case events ⋈ dim_country where dim_country is "big" in the catalog but a WHERE active = true reduces it to a few MB — Catalyst plans sort-merge; AQE flips it to broadcast.

Databricks: the 4th feature + Photon Databricks lists a 4th AQE feature — detect & propagate empty relations (a zero-row stage prunes dependent branches). AQE is on by default since DBR 7.3 LTS. Photon ≠ AQE: Photon is the native execution engine; AQE is the runtime planner. They're complementary — keep both on.

The code you'll write (and how to verify it)

Track rule Every technique is paired with how to prove it worked — the .explain() plan node or the Spark-UI signal. Apply, then verify. Never assume.

Confirm AQE is engaged

python# AQE master switch — on by default since Spark 3.2.0 / DBR 7.3+.
spark.conf.get("spark.sql.adaptive.enabled")   # -> 'true'

# VERIFY: an adaptive plan shows AdaptiveSparkPlan + AQEShuffleRead, and after a run
# the header flips to isFinalPlan=true.
df.explain(mode="formatted")
#   == Physical Plan ==
#   AdaptiveSparkPlan isFinalPlan=true        <-- AQE is engaged ✅
#   +- == Final Plan ==
#      *(...) ... AQEShuffleRead coalesced

1 · Coalesce — tune the target and floor

python# Defaults shown explicitly (and to tune the target).
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")              # default true
spark.conf.set("spark.sql.adaptive.advisoryPartitionSizeInBytes", 64*1024*1024)  # 64 MB target
spark.conf.set("spark.sql.adaptive.coalescePartitions.minPartitionSize", 1*1024*1024)  # 1 MB floor

agg = sales.groupBy("store_id").sum("amount")   # a shuffle (wide dependency)

# VERIFY: post-shuffle partition count collapses from 200 toward a handful.
print(agg.rdd.getNumPartitions())   # AQE-coalesced count (e.g. ~5), not 200
agg.explain(mode="formatted")       # look for: AQEShuffleRead coalesced

2 · Skew join — the lever set

python# Defaults shown explicitly.
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")                  # default true
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionFactor", "5.0")     # > 5x median ...
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes",
               256*1024*1024)                                       # ... AND > 256 MB

joined = transactions.join(accounts, "account_id")   # sort-merge join on a skewed key

# VERIFY: the plan shows the skew was detected and split.
joined.explain(mode="formatted")    # look for: AQEShuffleRead ... skewed=true

3 · Join switch — OSS vs Databricks runtime threshold

python# OSS static threshold (used by Catalyst BEFORE runtime):
spark.conf.get("spark.sql.autoBroadcastJoinThreshold")            # -> '10485760' (10 MB)
# Databricks AQE RUNTIME switch can use a higher bar:
spark.conf.get("spark.databricks.adaptive.autoBroadcastJoinThreshold")  # -> 30 MB (Databricks)

joined = events.join(dim_country.where("active = true"), "country_code")

# VERIFY: the FINAL (post-run) plan flips SortMergeJoin -> BroadcastHashJoin.
joined.explain(mode="formatted")
#   AdaptiveSparkPlan isFinalPlan=true
#   +- == Final Plan ==  BroadcastHashJoin ...   <-- AQE switched it ✅
#   +- == Initial Plan == SortMergeJoin ...       <-- stale-estimate plan

Contrast: hand-tuning vs letting AQE decide

python# ❌ Old habit: guess a shuffle partition count globally for the whole session.
spark.conf.set("spark.sql.shuffle.partitions", 8)   # wrong for the next huge-result query

# ✅ Modern: leave AQE on; it sizes post-shuffle partitions per query from real stats.
spark.conf.set("spark.sql.shuffle.partitions", 200)                       # pre-AQE upper bound
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")  # coalesce per query
# VERIFY: getNumPartitions() adapts to each query's data size — no re-tuning needed.

Uses, edge cases & limitations

Uses

  • Leave AQE on as the default for every job — coalesce, split, switch with zero code changes; the first answer to a slow shuffle.
  • Stop hand-setting spark.sql.shuffle.partitions — let coalesce size partitions per query; keep the conf only as an upper bound.
  • First line of defence against skew — enable AQE skew join before manual salting (Lesson 08); salt only what AQE can't tame.
  • Safety net for join estimates — the runtime sort-merge → broadcast switch covers a filter shrinking a side after planning (the gap Lesson 02 warned about).

Edge cases

  • No shuffle, no adaptation. A map-only job (pure select/filter) has no shuffle stats — don't expect coalescing on a shuffle-free plan.
  • Skew that misses a threshold. A partition 5× the median but only 100 MB (< 256 MB) is not split — both conditions must hold. Residual skew → salt.
  • Skew split is sort-merge only. It doesn't split skew inside a broadcast or shuffle-hash join.
  • Reading the plan too early. explain() before the run may show isFinalPlan=false / the initial plan. The final adaptive plan is only certain after execution.
  • Forcing broadcast() can pre-empt AQE. A manual hint fixes the strategy up front and removes AQE's runtime choice — prefer letting AQE decide unless you have a known, capped small side.

Limitations

  • Planning-layer only. AQE rewrites the plan; it does not change execution (that's Tungsten/Photon). It won't add partition pruning (Lesson 07) or bucketing (Lesson 11) for you.
  • Version gating. adaptive.enabled defaults to true only since Spark 3.2.0 (introduced 1.6.0); on Databricks since DBR 7.3 LTS. Older runtimes may need it enabled explicitly.
  • OSS lists 3 features; Databricks lists 4. Empty-relation propagation is the Databricks-documented 4th feature — not an OSS AQE feature.
  • It adapts the remaining plan, not the past. Work in a completed stage isn't redone; AQE only optimizes downstream of the boundary it just observed.

Common mistakes & gotchas

  • Conflating "AQE since 1.6" with "default-on." Introduced 1.6.0, but default-true only in 3.2.0 (SPARK-33679).
  • Disabling AQE to "control" the plan. You throw away the runtime coalesce/skew-split/join-switch that usually beats hand-tuning. Keep it on; scope a broadcast() hint if one join needs fixing.
  • Hand-setting shuffle.partitions low then wondering why a big query is slow — a globally small count starves a large query of parallelism.
  • Expecting skew split when only one threshold is met. A partition must be > 5× median and > 256 MB. Many "AQE didn't fix my skew" cases are a hot partition under 256 MB → that's a salting job (Lesson 08).
  • Confusing the OSS 10 MB static threshold with the Databricks 30 MB runtime switch. spark.sql.autoBroadcastJoinThreshold (10 MB) is up-front; spark.databricks.adaptive.autoBroadcastJoinThreshold (30 MB) is the runtime switch.
  • Thinking Photon replaces AQE (or vice-versa). Photon is the execution engine; AQE is the runtime planner. Both should be on — different layers.
  • Reading explain() before the run and assuming the initial plan is final. Look for isFinalPlan=true or the Spark UI's post-run DAG.
