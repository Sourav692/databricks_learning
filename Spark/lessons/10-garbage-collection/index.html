When the JVM pauses,
your tasks pause.

Garbage collection reclaims unreferenced objects for you — but each collection is stop-the-world, freezing every Spark task on that executor. Here's why a full GC stalls a stage, and the levers that make pauses shorter and rarer.

Spark 3.2+ / DBR LTS memory.fraction 0.6 G1GC default Spark 4.0 / JDK 17 Verified Jun 2026 docs

What it is

Garbage Collection (GC) is the JVM automatically reclaiming the memory of objects your program no longer references — you never free() in Java/Scala, the GC does it. On a Spark executor that convenience shows up as pauses:

  • The executor heap is split into generations — a Young generation (where new objects are born) and an Old generation (where long-lived objects end up).
  • A minor GC cleans the Young generation — frequent but cheap. A major / full GC cleans the Old generation — rare but expensive.
  • A GC is stop-the-world: while it runs, every application thread — including all of Spark's task threads on that executor — freezes. That freeze is a GC pause.
The one rule to remember GC cost is proportional to the number of Java objects, not the number of bytes. Many tiny objects (boxed values, deserialized cached rows) = lots of work for the collector = long pauses. Fewer, larger objects = cheaper GC.

Why it matters

  • A GC pause is wasted cluster time — during a stop-the-world collection no task on that executor makes progress; you're paying for idle CPUs.
  • Spark stages wait on the slowest task. GC runs per executor JVM, so a long pause on one executor stalls its tasks while the rest of the cluster waits at the stage boundary. One GC-thrashing executor can dominate a stage's wall-clock.
  • Heavy caching is the usual trigger. Cached blocks are long-lived, so they pile into the Old generation; a full Old gen forces frequent, expensive full GCs — the "I cached everything and now it's slower" surprise.
  • Interviewers probe it as symptom-vs-cause: "GC time is high — what's happening and what do you change?" The answer is the chain: too many objects / Old gen too full → stop-the-world pauses → executor stalls → tune cache shape, object shape, spark.memory.fraction, or the collector.

How it works — deep dive

1 · What is Garbage Collection?

analogy GC is the cleaning crew at a restaurant. While they sweep the whole floor (a full GC), every diner steps outside and waits (stop-the-world) — the more plates and crumbs (objects), the longer everyone waits.

  • Mechanism: the JVM tracks which objects are still reachable from live variables. Anything unreachable is garbage; the collector reclaims it so new allocations have room. You never free memory manually.
  • Why it costs: the docs state GC cost is "proportional to the number of Java objects" walked, not the byte count. A list of a million boxed Integers is far costlier to collect than one int[] holding the same data.
  • Why Spark cares: Spark allocates a lot — every deserialized cached row, shuffle buffer, and intermediate record is a heap object. Object-heavy workloads make the collector work harder, which becomes pause time.

use case A job caching a wide DataFrame deserialized (one object per field per row) churns millions of objects; a serialized cache (one byte array per partition) collapses that to a handful of objects per partition — far less GC.

2 · The heap generations (Young + Old)

  • Mechanism: the heap splits into Young (= Eden + two Survivor spaces S0/S1) and Old. New objects go to Eden; a minor GC copies survivors into a Survivor space; objects that survive enough minor GCs are promoted to Old.
  • Why two generations: most objects die young, so collecting the small churny Young space frequently is cheap; the long-lived Old space needs the expensive collection only rarely.
  • Trade-off: Spark's cached blocks and broadcast data are long-lived → they land in Old. Cache too much and Old fills, forcing frequent full GCs — the worst case for pause time.

analogy Eden is the prep counter (constant fast turnover); the Old gen is the walk-in freezer (things that stick around). You wipe the counter constantly and cheaply; you only defrost the freezer occasionally — and defrosting shuts the kitchen.

3 · Minor, major & full GC — and the stop-the-world pause

  • Minor GC — collects the Young gen only. Frequent and cheap (Eden is small, most of it already dead).
  • Major / full GC — collects the Old gen (often the whole heap). Rare and expensive — it walks far more live objects.
  • Stop-the-world: during a collection the JVM suspends every application thread. On an executor that freezes all task threads — a minor GC is a short blip, a full GC a multi-second stall.
  • The tuning goal: you can't eliminate GC, only make it shorter and rarer — fewer objects (both collections get cheaper) and a less-full Old gen (fewer full GCs). Every lever targets one of those two.

4 · The executor's role in the GC cycle

use case One executor caches more than its share, its Old gen stays near-full, and it spends seconds in full GC every few tasks. Its tasks finish late, the stage can't complete until that straggler does, and the job's wall-clock balloons — even though the other executors were idle and healthy.

  • GC is per-JVM: each executor is its own JVM with its own heap, so GC runs independently inside each executor. There is no cluster-wide GC.
  • A pause stalls only that executor — but the stage waits for it. A stage completes only when its slowest task finishes, so a GC-bound executor becomes a straggler that holds up the whole stage.
  • What loads the Old gen: cached blocks (especially deserialized) and broadcast data are long-lived → more Old-gen pressure → more full GCs. This is the direct link between caching too much and GC pain.

5 · Seeing GC in the Spark UI (and turning on GC logs)

  • Stages tab: every task row reports "GC Time" next to its duration. If GC Time is a large fraction of task time (rule of thumb: >~10% is worth investigating), GC is your bottleneck.
  • Executors tab: shows a per-executor Task Time (GC Time) summary — use it to spot the one executor that's GC-thrashing.
  • Verbose GC logs: pass JVM flags via spark.executor.extraJavaOptions; the logs land in each worker's stdout.
Start-time only spark.executor.extraJavaOptions is read when the executor JVM starts, so set it in the cluster's Spark config (or at session creation) — you can't flip the collector or GC flags mid-session. On Databricks, set it under the cluster's Spark config.

Tune it — lower spark.memory.fraction

The unified region (cache + execution) is spark.memory.fraction × (heap − 300 MiB). Drag it down and watch the cache shrink, the Old-gen pressure ease, and the full-GC pauses get shorter — the docs' "better to cache fewer objects than to slow down task execution" trade-off, made visible.

The code you'll write (and how to verify it)

Track rule Every technique is paired with how to prove it worked — the Spark-UI GC Time signal or the executor's GC log. Apply, then verify. Never assume.

Turn on detailed GC logging

python# Set at cluster creation / session config — read when the executor JVM STARTS.
spark.conf.set(
    "spark.executor.extraJavaOptions",
    "-verbose:gc -XX:+PrintGCDetails -XX:+PrintGCTimeStamps"
)
# VERIFY: Spark UI → Executors → stdout. You'll see per-collection lines:
#   [GC (Allocation Failure) ...]   <- minor GC
#   [Full GC (Ergonomics) ...]      <- full GC (many of these = Old gen too full)

Lever 1 · Serialized caching — one byte-array per partition (JVM/Scala-side)

PySpark honesty check There is no deserialized-vs-serialized choice from PySpark. The PySpark StorageLevel enum exposes only MEMORY_ONLY / MEMORY_AND_DISK / DISK_ONLY (+ _2 / DISK_ONLY_3) — no _SER levels — because the docs say "in Python, stored objects will always be serialized with the Pickle library, so it does not matter whether you choose a serialized level." So cache() is persist(MEMORY_AND_DISK); running both changes nothing and GC Time won't move. The serialized-cache win is a JVM/Scala-side lever — from PySpark, reach for Lever 2 (memory.fraction) or Lever 4 (off-heap) instead. 
scala// Scala / Java (JVM-side) — where the serialized-cache GC win is real.
import org.apache.spark.storage.StorageLevel

// ❌ Deserialized JVM cache (MEMORY_AND_DISK): one Java object per field per row →
//    millions of long-lived objects in the Old gen → heavy full GC.
hotDf.persist(StorageLevel.MEMORY_AND_DISK)

// ✅ Serialized JVM cache: docs say it stores "only one object (a byte array) per RDD partition"
//    → orders of magnitude fewer objects to track.
hotDf.persist(StorageLevel.MEMORY_AND_DISK_SER)
hotDf.count()   // materialize the cache
python# PySpark equivalent — there is NO _SER level to switch to; Python already pickles every cache.
from pyspark import StorageLevel

hot_df.persist(StorageLevel.MEMORY_AND_DISK)  # same as hot_df.cache() — Python caches are pickled
hot_df.count()                                # materialize the cache

# VERIFY (PySpark): switching StorageLevel here changes nothing → expect NO GC-Time change.
# The serialized-cache GC drop only appears on JVM/Scala caches; from PySpark, use Lever 2 / Lever 4.

Lever 2 · Lower spark.memory.fraction when the Old gen is full

python# Unified region = spark.memory.fraction × (heap − 300 MiB). Default 0.6.
# If cached blocks keep the Old gen near-full and you see constant full GCs,
# give the cache LESS heap so fewer long-lived objects are pinned.
spark.conf.set("spark.memory.fraction", "0.4")   # shrink cache/exec → less Old-gen pressure

# Docs verbatim: "better to cache fewer objects than to slow down task execution."
# VERIFY: re-run and compare GC Time in the Stages tab before vs after; reset the conf after A/B.

Lever 3 · Use G1GC on large heaps

python# VERSION NOTE: since Spark 4.0.0 (JDK 17 default), G1GC is ALREADY the default collector.
# On older Spark / JDK 8, ParallelGC was default and G1GC was the recommended opt-in:
spark.conf.set(
    "spark.executor.extraJavaOptions",
    "-XX:+UseG1GC -XX:G1HeapRegionSize=16m"    # bump region size for very large heaps
)
# -Xmn tunes the young-gen size when many short-lived objects churn the Young gen.
# VERIFY: executor stdout shows "[GC pause (G1 Evacuation Pause) ...]" with shorter durations;
# GC Time in the Stages tab falls on large-heap, allocation-heavy stages.

Lever 4 · Off-heap — keep data out of the GC's reach

python# Off-heap memory lives OUTSIDE the JVM heap → the collector never scans it → no GC pressure.
spark.conf.set("spark.memory.offHeap.enabled", "true")
spark.conf.set("spark.memory.offHeap.size", str(2 * 1024 * 1024 * 1024))  # 2 GB, bytes (> 0)
# Shrink the JVM heap accordingly, or you trade GC for OOM (Lesson 04).
# VERIFY: the same cache/shuffle workload shows lower GC Time — those bytes left the heap.

Contrast: default cache vs lower memory.fraction (the real PySpark GC lever)

python# ❌ GC-heavy: cache at the default 0.6 fraction → a large cache region pins many long-lived
#    objects in the Old gen → frequent full GCs.
big.cache(); big.count()                       # == persist(MEMORY_AND_DISK); Python pickles it

# ✅ GC-friendly: SAME cache, but a smaller unified region so it pins less of the heap → fewer
#    long-lived objects in the Old gen. (memory.fraction is the lever a PySpark user can turn — the
#    _SER cache switch is JVM/Scala-side only.)
big.unpersist(blocking=True)
spark.conf.set("spark.memory.fraction", "0.4")
big.persist(StorageLevel.MEMORY_AND_DISK); big.count()
# Stages tab: with a smaller cache region the second run's GC Time should be a smaller fraction of
# task time. (The change here is memory.fraction, NOT a serialized-vs-deserialized switch.)

Uses, edge cases & limitations

Uses

  • Large-heap, cache-heavy, or allocation-heavy jobs where the Spark UI shows GC Time as a big fraction of task time — the canonical signal to reach for these levers.
  • "Caching made it slower" diagnosis — switch to serialized/off-heap cache or lower spark.memory.fraction rather than blindly adding memory.
  • Python-heavy pipelines — replace row-at-a-time UDFs with native expressions / vectorized (pandas) UDFs to cut per-row object churn.

Edge cases

  • A bigger heap can make full GCs worse — more Old-gen memory means each full GC walks more live objects → longer pauses. Fix is usually fewer objects, not more heap.
  • Skew amplifies GC — one huge skewed partition (Lesson 08) allocates far more than its peers on one executor, pushing that JVM into GC thrash. Fix the skew first.
  • Spill increases GC — the docs note spilling brings "increased garbage collection". Curing spill (Lesson 04) often cures the GC symptom too.
  • extraJavaOptions is start-time only — read when the executor JVM boots; you can't flip the collector mid-session.

Limitations

  • You can't eliminate GC — it's intrinsic to the JVM. You only make pauses shorter and rarer; GC tuning is a last-mile lever, not a first resort.
  • GC tuning is workload-specific — the right memory.fraction, young-gen size, and collector depend on object churn vs cache size; tune empirically from the GC logs.
  • Off-heap is partly experimental for caching and you must hand-balance JVM heap vs off-heap size — get it wrong and you trade GC for OOM.
  • Version gate: G1GC is the default only since Spark 4.0.0 (JDK 17); on older Spark/JDK 8 it was an opt-in over ParallelGC. State which runtime you mean.

Common mistakes & gotchas

  • Throwing more heap at high GC time. A larger Old gen often means longer full-GC pauses. Reduce the number of objects (serialize the cache, vectorize UDFs) before sizing up.
  • Caching everything "to be safe." Cached blocks are long-lived and pile into the Old gen → more full GCs. Cache only reused DataFrames (Lesson 06), serialized or off-heap, and unpersist() when done.
  • Confusing GC Time with compute time. A slow task with high GC Time isn't slow because the work is hard — it's stalled in stop-the-world pauses. Read the GC Time column, don't guess.
  • Setting extraJavaOptions at runtime and expecting it to take effect — it's applied at executor startup. Set it in the cluster's Spark config.
  • Assuming G1GC is always on. It's the default only from Spark 4.0 / JDK 17; older runtimes need -XX:+UseG1GC.
  • Forgetting off-heap shrinks the usable heap. Off-heap data avoids GC but you must reduce the JVM heap, or the container OOMs (Lesson 04).
  • Ignoring skew/spill as the real cause. High GC is often a symptom of a skewed partition or execution-memory spill — fix those first.
