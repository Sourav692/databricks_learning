Driver, executors
& the shuffle.

How one line of PySpark becomes a fleet of tasks across a cluster: the driver builds the plan, an action fires a job, shuffle boundaries cut it into stages, and one task runs per partition. Get this picture and every later lesson clicks into place.

Spark 3.2+ / DBR LTS shuffle.partitions 200 Verified Jun 2026 docs

What it is

A Spark application is one program (your notebook or job) that runs as a small society of processes:

  • Driver — the brain. Runs your code, holds the SparkSession, turns DataFrame code into a plan, and schedules work. Exactly one per application.
  • Executors — the muscle. JVM processes on worker nodes that run tasks and hold cached data. There are N of them.
  • Cluster manager — the landlord. Allocates the worker machines (Standalone / YARN / Kubernetes; on Databricks the platform manages it). It does not schedule your tasks — the driver does.

You also choose where you launch from — the edge / gateway node — and where the driver runs — the deployment mode (client vs cluster).

The one rule to remember Spark is lazy. Transformations only describe work; an action fires a job; the job is cut at shuffle boundaries into stages; each stage runs as tasks — one per partition. Job → stage → task → partition is how you read any Spark UI.

Why it matters

  • Every performance problem is a problem with this picture — too much on the driver, the wrong partition count, or an expensive shuffle between stages. You can't tune what you can't see.
  • The shuffle is the villain of the whole track. A shuffle = Spark moving rows across the network so equal keys land on the same executor; it writes files to local disk and reads them back over the network — the single most expensive operation Spark does. Lessons 02–11 are about avoiding, repairing, or surviving it.
  • Interviewers start here. "Walk me through what happens when you call df.count()" separates people who use Spark from people who understand it.

How it works — deep dive

1 · The three roles: driver, executors, cluster manager

analogy A restaurant kitchen — the driver is the head chef calling orders and plating the dish; the executors are the line cooks at their stations; the cluster manager is the owner deciding how many cooks you get.

  • Driver — runs user code, builds the logical→physical plan (Catalyst), cuts it into jobs/stages/tasks, schedules them, and collects results. It also builds the small side of a broadcast join. A single point — overload it and the whole app sinks (Lesson 03).
  • Executors — long-lived JVMs with fixed cores (task slots) and heap (Lesson 04). They run tasks, cache blocks, and write/read shuffle files. Parallelism is capped by partitions, not by raw core count.
  • Cluster manager — only allocates machines; it does not schedule tasks. On Databricks this is fully managed.

use case Sizing a cluster: a stage with 800 partitions on executors with 4 cores each needs 200 executor-cores to finish in one wave; fewer cores = more waves = longer wall-clock.

2 · The edge / gateway node

  • Definition: a machine that can submit to the cluster and talk to the cluster manager, but is not in the worker pool that runs tasks.
  • It holds the Spark client libraries + cluster config and bridges your laptop/CI host to the cluster. In client mode the driver runs here.
  • On Databricks the notebook front-end plays this role and your commands are interpreted on the cluster's driver node. The concept still matters when you submit from CI/CD, Airflow, or a bastion host.

analogy The edge node is the drive-thru window — you order there and it relays to the kitchen, but no food is cooked at the window.

3 · Deployment modes: client vs cluster

The deployment mode answers one question: where does the driver run?

  • Client mode — driver on the submitting / edge node. YARN: "the driver runs in the client process, and the application master is only used for requesting resources from YARN." If the client dies, the app dies. Best for interactive notebooks / REPL.
  • Cluster mode — driver runs inside the cluster. YARN: "the Spark driver runs inside an application master process… and the client can go away after initiating the application." Best for production fire-and-forget batch.
  • On Databricks notebooks, the driver always runs on the cluster's driver node — it maintains the SparkContext, interprets notebook commands, and runs the Spark master. On a single-node cluster the driver node is both master and worker.

use case An Airflow DAG running nightly ETL → cluster mode (the worker hands off and moves on). A data scientist in a REPL → client mode (results stream to the terminal).

4 · Lazy evaluation, transformations vs actions

  • Lazy: Spark records what you want but does nothing until you ask for a result.
  • Transformations (select, filter, withColumn, join, groupBy) return a new DataFrame and run nothing — they extend the plan.
  • Actions (count, collect, show, write, take) force execution. Each action triggers a job.
  • Why it's good: seeing the whole chain lets Catalyst fuse ops, push filters to the scan, and prune columns — impossible if each line ran eagerly.

analogy A shopping list — writing items (transformations) is free; you only walk the aisles (the job) when you go to the store (an action).

5 · Narrow vs wide dependencies (and the shuffle)

  • Narrow: each input partition feeds one output partition (map, filter, select, union). No data moves — Spark pipelines them in one stage.
  • Wide: an output partition depends on many inputs, so rows are redistributed across the network — a shuffle (groupBy, non-broadcast join, distinct, repartition).
  • The shuffle, mechanically: (1) map side partitions output by key and writes shuffle files to local disk; (2) a stage barrier waits for all map tasks; (3) reduce side fetches its slices over the network. That's disk + network + serialize + a wait — the most expensive thing Spark does. spark.sql.shuffle.partitions (default 200 OSS; auto on Databricks) sets the post-shuffle count.

use case events.groupBy("user_id").count() shuffles all events so each user's rows land together — a cluster-killer when one user holds 90% of rows (skew, Lesson 08).

6 · The execution hierarchy: job → stage → task

  1. Action → Job. Each action submits one job.
  2. Job → Stages. Spark cuts a new stage at every shuffle (wide) boundary; narrow ops fuse into the same stage. One shuffle ⇒ two stages.
  3. Stage → Tasks. Each stage runs one task per partition. 200 partitions ⇒ 200 tasks.
  4. Tasks → executor cores in waves (total tasks ÷ total cores).
Why this is the whole track "Too many tiny tasks" → AQE coalesce (Lesson 05). "One task runs 10× longer" → skew (Lesson 08). "A stage with huge Shuffle Read" → an avoidable shuffle (Lessons 02, 11). Learn to read this hierarchy once and every later lesson clicks into place.

The code you'll write (and how to verify it)

Track rule Every technique is paired with how to prove it — the .explain() plan node or the Spark-UI signal. Apply, then verify. Never assume.

Prove lazy evaluation (transformations build, actions fire)

python# None of these run anything — they only extend the logical plan (LAZY).
df  = spark.range(0, 1_000_000)
df2 = df.filter("id % 2 = 0")                 # transformation — no job
df3 = df2.withColumn("bucket", df2.id % 10)   # transformation — no job

# VERIFY: Spark UI > Jobs tab is EMPTY right now. Nothing has executed.
count = df3.count()                          # ACTION — fires exactly one job
# VERIFY: Spark UI > Jobs tab now shows ONE new job for count().

See the plan & stage boundaries (narrow vs wide)

pythonfrom pyspark.sql.functions import col   # used by the col("id") snippets below

# Narrow-only: filter + withColumn fuse into ONE stage, no shuffle.
narrow = spark.range(1_000_000).filter("id > 10").withColumn("x", col("id") * 2)
narrow.explain(mode="formatted")
#   Project +- Filter +- Range     <- no Exchange = no shuffle = single stage ✅

# Wide: groupBy forces a shuffle -> a stage boundary appears.
wide = spark.range(1_000_000).groupBy(col("id") % 100).count()
wide.explain(mode="formatted")
#   HashAggregate
#   +- Exchange hashpartitioning(...)   <- Exchange = the shuffle = a NEW stage ❌(costly)
#      +- HashAggregate +- Range

Uses, edge cases & limitations

Uses

  • The mental model for reading any Spark UI / .explain() — find the Exchange nodes (shuffles = stage boundaries), count tasks per stage (= partitions), watch the task-time spread.
  • Sizing a cluster — match executor cores to partition counts so a stage runs in as few waves as possible.
  • Choosing a deployment mode — cluster mode for scheduled production, client mode for interactive sessions.

Edge cases

  • A "no-op"-looking action still runs a job — show() and take(n) trigger jobs (sometimes partial); even noop runs the full plan and only discards output.
  • Too few partitions → idle cores + huge tasks that spill/OOM. Too many tiny partitions → scheduling overhead dominates. AQE auto-coalesces the over-partitioned case (Lesson 05).
  • spark.sql.shuffle.partitions only affects shuffles — not a plain scan's partition count; use repartition/coalesce for that.
  • Single-node Databricks cluster — the driver node is also the only worker, so driver and executor share one machine; fine for dev, misleading for perf intuition.

Limitations

  • The driver is a single point of failure and a bottleneck — it can't scale out; protect it (Lesson 03).
  • A shuffle cannot be made free, only avoided (broadcast, bucketing) or reduced (AQE coalesce). No flag "turns off" a required shuffle.
  • Lazy evaluation hides cost — a long chain looks instant until the action runs all of it at once. Reason from the plan, not the line count.
  • shuffle.partitions is 200 in OSS Spark; on Databricks it can be auto. State which you mean.

Common mistakes & gotchas

  • Thinking transformations run. They don't — only actions do. A "slow filter" is the action that finally ran the whole chain. Time the action, read the plan.
  • Confusing the two meanings of "partition." Here it's an in-memory Spark partition (a chunk → one task). On-disk Hive/Delta partitions (directories) are different (Lesson 07).
  • collect() to "just check" a big DataFrame. That pulls every row to the driver and risks driver OOM (Lesson 03). Use take(n), show(), or a noop write.
  • Assuming more executors = faster. Parallelism is capped by the partition count — 1000 cores can't help a stage with 8 partitions.
  • Ignoring the Exchange. Every Exchange is a shuffle you're paying for. First question for a slow job: "which Exchange nodes are here, and can I remove one?"
  • Mixing up the cluster manager and the driver. The manager only allocates machines; the driver schedules tasks.
