Spark architecture

The execution model · driver, executors, the shuffle

One line of PySpark becomes a fleet of tasks across a cluster: the driver builds a plan, an action fires a job, shuffle boundaries cut it into stages, and one task runs per partition. Get this one picture and every later lesson clicks into place.

01 · What it is

A small society of processes

A Spark application is one program — your notebook or job — that runs as three kinds of process working together:

  • Driver — the brain. Runs your code, holds the SparkSession, turns DataFrame code into a plan, and schedules the work. There is exactly one per application.
  • Executors — the muscle. JVM processes on the worker nodes that run tasks and hold cached data. There are N of them.
  • Cluster manager — the landlord. Allocates the worker machines (Standalone / YARN / Kubernetes; on Databricks the platform manages it). It allocates machines — it does not schedule your tasks.

Two more choices frame the picture: where you launch from — the edge / gateway node — and where the driver runs — the deployment mode (client vs cluster). The deep dive unpacks each; for now just hold the shape.

The one rule to remember Spark is lazy. Transformations only describe work; an action fires a job; the job is cut at shuffle boundaries into stages; each stage runs as tasks — one per partition. Job → stage → task → partition is how you read any Spark UI.
Driver vs executors

The one driver plans and schedules the work; the N executors run the tasks and hold the data.

Narrow — no shuffle

filter, select, withColumn stay in one stage; no data crosses the network.

Wide — shuffle

groupBy, join, distinct redistribute rows across the network — a new, costly stage.

02 · Why it matters

You can't tune what you can't see

  • Every performance problem is a problem with this picture — too much on the driver, the wrong partition count, or an expensive shuffle between stages. The rest of the track is refinements of this one model.
  • The shuffle is the villain of the whole track. It is the single most expensive thing Spark does, and Lessons 02–11 are about avoiding, repairing, or surviving it. The deep dive is where we take it apart.
  • Interviewers start here. "Walk me through what happens when you call df.count()" separates people who use Spark from people who understand it — the answer is the job → stage → task chain below.

03 · How it works — deep dive

From one line of code to a fleet of tasks

The whole model is a chain: your code builds a plan, an action turns it into a job, shuffle boundaries cut it into stages, and each stage fans out into tasks. Each idea below is explained here once — later sections show it, verify it, and help you decide, but don't re-explain it.

3.1 · The three roles

analogy A restaurant kitchen — the driver is the head chef calling orders and plating the dish; the executors are the line cooks at their stations; the cluster manager is the owner deciding how many cooks you get.

  • Driver. Runs user code, builds the logical → physical plan via Catalyst, cuts it into jobs/stages/tasks, schedules them, and collects results. It also builds the small side of a broadcast join. A single point — overload it and the whole app sinks (Lesson 03).
  • Executors. Long-lived JVMs with fixed cores (task slots) and heap (Lesson 04). They run tasks, cache blocks, and write/read shuffle files. Parallelism is capped by partitions, not by raw core count.
  • Cluster manager. Only allocates machines; it does not schedule tasks — the driver does. On Databricks this is fully managed.

use case A stage with 800 partitions on executors with 4 cores each needs 200 executor-cores to finish in one wave; fewer cores means more waves means longer wall-clock.

3.2 · Edge node & deployment mode — where the driver runs

  • Edge / gateway node. A machine that can submit to the cluster and talk to the cluster manager, but is not in the worker pool. It holds the Spark client libraries and cluster config — the bridge from your laptop/CI host to the cluster.
  • Client mode. Driver runs on the submitting / edge node. YARN: "the driver runs in the client process, and the application master is only used for requesting resources from YARN." If the client dies, the app dies. Best for interactive notebooks / REPL.
  • Cluster mode. Driver runs inside the cluster. YARN: "the Spark driver runs inside an application master process… and the client can go away after initiating the application." Best for production fire-and-forget batch.
  • On Databricks. The notebook front-end plays the edge-node role and the driver runs on the cluster's driver node. On a single-node cluster that node is both master and worker.

3.3 · Lazy evaluation: transformations vs actions

  • Lazy. Spark records what you want but runs nothing until you ask for a result — proven live in Figure A above.
  • Transformations (select, filter, withColumn, join, groupBy) return a new DataFrame and run nothing — they extend the plan.
  • Actions (count, collect, show, write, take) force execution. Each action triggers exactly one job.
  • Why it's good. Seeing the whole chain lets Catalyst fuse ops, push filters to the scan, and prune columns — impossible if each line ran eagerly.

3.4 · Narrow vs wide dependencies — and the shuffle

  • Narrow. Each input partition feeds one output partition (map, filter, select, union). No data moves — Spark pipelines them into one stage.
  • Wide. An output partition depends on many inputs, so rows are redistributed across the network — a shuffle (groupBy, non-broadcast join, distinct, repartition).
  • The shuffle, mechanically. (1) The map side partitions its output by key and writes shuffle files to local disk; (2) a stage barrier waits for all map tasks; (3) the reduce side fetches its slices over the network. Disk + network + serialize + a wait — the most expensive thing Spark does. spark.sql.shuffle.partitions (default 200 in OSS; auto on Databricks) sets the post-shuffle count.

use case events.groupBy("user_id").count() shuffles all events so each user's rows land together — a cluster-killer when one user holds 90% of rows (skew, Lesson 08).

3.5 · The execution hierarchy: job → stage → task

This is the canonical enumeration — the comparison table and gotchas later point back here rather than re-listing it. Read it top-down: each level unfolds into the next.

LevelSplit ruleConcrete example
Action → JobEach action submits exactly one job.1 count() ⇒ 1 job
Job → StagesA new stage is cut at every shuffle (wide) boundary; narrow ops fuse into the same stage.1 shuffle ⇒ 2 stages
Stage → TasksEach stage runs one task per partition.200 partitions ⇒ 200 tasks
Tasks → coresTasks run on cores in waves (total tasks ÷ total cores).200 tasks ÷ 50 cores ⇒ 4 waves
Shuffle ≠ a plain scan's partitioning spark.sql.shuffle.partitions only sets the count after a shuffle. A plain scan's partition count comes from the input files; change that with repartition / coalesce, not this config. Conflating the two is the most common early mistake.

04 · The code — and how to verify it

Four moves, each with its proof

Every technique below is paired with how to prove it worked — the .explain() plan node or the Spark-UI signal. Apply, then verify; never assume.

Simple reading rule When you see a line of PySpark, ask one question first: "Does this line trigger an action (fire a job) or introduce a shuffle — or does it stay narrow and lazy?" An action or an Exchange is where the cost lives; a lazy narrow transformation is nearly free.

Prove lazy evaluation (transformations build, actions fire)

python# None of these run anything — they only extend the logical plan (LAZY).
df  = spark.range(0, 1_000_000)
df2 = df.filter("id % 2 = 0")                 # transformation — no job
df3 = df2.withColumn("bucket", df2.id % 10)   # transformation — no job

# VERIFY: Spark UI > Jobs tab is EMPTY right now. Nothing has executed.
count = df3.count()                          # ACTION — fires exactly one job
# VERIFY: Spark UI > Jobs tab now shows ONE new job for count().

See the plan & stage boundaries (narrow vs wide)

pythonfrom pyspark.sql.functions import col   # used by the col("id") snippets below

06 · Edge cases & limitations

Where it gets subtle

New nuance an interviewer probes — not a re-run of the hierarchy above:

  • A "no-op"-looking action still runs a job. show() and take(n) trigger jobs (sometimes partial); even noop runs the full plan and only discards the output.
  • Too few vs too many partitions. Too few → idle cores and huge tasks that spill/OOM; too many tiny partitions → scheduling overhead dominates. AQE auto-coalesces the over-partitioned case (Lesson 05).
  • Two meanings of "partition." Here it's an in-memory Spark partition (a chunk → one task). On-disk Hive/Delta partitions (directories) are a different thing (Lesson 07).
  • Single-node Databricks cluster. The driver node is also the only worker, so driver and executor share one machine — fine for dev, misleading for perf intuition.

Hard limits worth stating plainly:

  • The driver is a single point of failure and a bottleneck — it can't scale out; protect it (Lesson 03).
  • A shuffle cannot be made free, only avoided (broadcast, bucketing) or reduced (AQE coalesce). No flag "turns off" a required shuffle.
  • Lazy evaluation hides cost — a long chain looks instant until the action runs all of it at once. Reason from the plan, not the line count.
  • shuffle.partitions is 200 in OSS Spark; on Databricks it can be auto. State which you mean.

07 · Common mistakes

Gotchas — the mistake and its one-line fix

MistakeWhy it hurtsBetter move
Thinking transformations runOnly actions execute; a "slow filter" is really the action that finally ran the whole chain.Time the action and read the plan, not the transformation.
Assuming more executors = fasterParallelism is capped by the partition count; 1000 cores can't help a stage with 8 partitions.Match partitions to cores before adding executors.
Ignoring the ExchangeEvery Exchange is a shuffle you're paying for.Ask which Exchange nodes are present and whether one can be removed.
collect() "just to look" at a big DataFramePulls every row to the driver and risks driver OOM (Lesson 03).Use take(n), show(), or a noop write.
Mixing up the cluster manager and the driverThe manager only allocates machines — it does not schedule tasks.Remember the driver schedules tasks; the manager just hands out machines.
