Broadcast variables
& accumulators.

Two ways to share data with your tasks: push a small read-only value out to every executor once, and pull aggregated numbers safely back to the driver — plus the exactly-once trap that catches everyone.

Spark 3.2+ / DBR LTS Shipped once per executor Exactly-once actions only Verified Jun 2026 docs

What it is

Spark gives you two shared variables — values that cross the driver↔executor boundary in a controlled way, instead of being captured ad-hoc by a closure:

  • Broadcast variable — a read-only value you create on the driver; Spark caches it once per executor, so every task on that executor reads the same in-memory copy. bv = sc.broadcast(value); read on workers with bv.value.
  • Accumulator — a write-only-on-workers counter: tasks only .add(); only the driver reads the final .value. acc = sc.accumulator(0); tasks do acc.add(1).

Think of them as opposite-direction pipes: a broadcast variable pushes data out to every executor; an accumulator pulls aggregated numbers back to the driver.

The one rule to remember A broadcast variable is not a broadcast join. The variable is a value you reference inside a UDF/map; the join is a strategy the planner picks via autoBroadcastJoinThreshold / the BROADCAST hint (Lesson 02). Same word, different mechanism.

Why it matters

  • Without a broadcast variable, a lookup dict/set/model referenced in a closure is serialized and shipped with every task — 10,000 tasks = 10,000 copies of the same 50 MB map crossing the network. With a broadcast variable it ships once per executor and is reused by all its tasks: far less network, GC, and duplicated memory.
  • Without an accumulator, getting a number (bad-record count, rows-skipped) out of distributed tasks needs a whole extra pass. An accumulator lets tasks report a number as a side-effect of work you're already doing — no extra job.
  • Interviewers probe both: "How do you ship a 200 MB model to your UDF without re-sending it per task?" (broadcast variable) and "Can you trust a counter incremented inside a map?" (no — exactly-once holds only inside actions).

How it works — deep dive

1 · Broadcast variable — ship a read-only value once per executor

analogy A broadcast variable is like handing every cashier (executor) their own laminated price-list at the start of the shift, instead of faxing it to every customer (task). One copy per till, reused all day.

  • Mechanism: bv = sc.broadcast(value) on the driver serializes the value and ships it to each executor once (torrent-like fan-out); the executor caches it (in Storage memory / the block manager) and every task reads the same bv.value. The docs define it as "a read-only variable cached on each machine rather than shipping a copy of it with tasks."
  • Read-only: workers only read bv.value. Mutating it on a worker has no effect anywhere else — it's broadcast once, not kept in sync.
  • Why it beats a closure variable: a captured closure var is serialized per task (one copy per partition); a broadcast var is serialized once per executor.
  • Lifecycle: bv.unpersist() drops the cached copies (re-broadcast if reused); bv.destroy() removes it permanently (using it afterwards is an error). Release it — it pins Storage memory on every executor.
  • Trade-off: the value must fit on the driver (where it's built) and in each executor's memory; it's static and read-only.

use case A 30 MB country_code → region dict (or a small ML model) referenced in a Python UDF scoring 5 billion rows — broadcast it once per executor instead of shipping it with every task.

python# DRIVER: build the small lookup once, then broadcast it (read-only) to every executor.
lookup = {"US": "NA", "CA": "NA", "DE": "EU", "IN": "APAC"}
bv = sc.broadcast(lookup)                       # shipped ONCE per executor

from pyspark.sql.functions import udf
from pyspark.sql.types import StringType

# WORKER: each task reads the SAME cached copy via bv.value — no per-task re-send.
@udf(StringType())
def to_region(code):
    return bv.value.get(code, "UNKNOWN")    # bv.value is READ-ONLY on the worker

regions = orders.withColumn("region", to_region("country_code"))

# VERIFY (a *variable*, not a plan node — verify in the Spark UI, not .explain()):
#   • Spark UI -> Storage tab: one "Broadcast" block, sized ~your value, NOT x num_tasks.
#   • tasks don't re-serialize the dict per task.
bv.unpersist()   # release when done (re-broadcast if reused) …
# bv.destroy()   # … or remove permanently (using it afterwards raises an error)
Variable vs join — keep them apart sc.broadcast(dict) ships a value you read in code. df.join(broadcast(small_df), …) (Lesson 02) is a join strategy. The DataFrame broadcast() hint takes a DataFrame; sc.broadcast() takes a plain Python object. Don't use a broadcast variable to "speed up a join" — use the join hint.

Same word, two mechanisms

2 · Accumulator — count on workers, read on the driver

analogy An accumulator is like a tally counter at a stadium turnstile — each gate (task) clicks it as people pass, but only the control room (driver) reads the total. Gates can add; they can't read the running total.

  • Mechanism: acc = sc.accumulator(0) creates a shared counter on the driver — PySpark's only accumulator constructor (there is no sc.longAccumulator(name) in PySpark; that's Scala/Java-only). Tasks call acc.add(n) (or acc += n); each executor keeps a local partial and sends its contribution back, where Spark merges them on the driver. Only the driver reads acc.value.
  • Write-only on workers: a task that reads acc.value sees only its own partial — never the global total. The asymmetry is what makes the merge associative and cheap.
  • Lazy: the update only happens when an action runs the transformation containing it. No action → the count stays at the initial value.
  • The exactly-once caveat (the whole interview): the docs guarantee exactly-once only for updates inside actions (e.g. foreach): "In transformations, users should be aware that each task's update may be applied more than once if tasks or stages are re-executed." A map/filter that increments an accumulator may run twice (retried task, recomputed stage, or a second action over an uncached DataFrame) → inflated count.
  • Trade-off: great for cheap side-channel metrics; not a substitute for a real aggregation. For correctness-critical counts use groupBy().count() / filter().count().

use case Counting malformed records while writing a cleaned dataset — bad_rows.add(1) inside the action that writes (foreachPartition), then log bad_rows.value on the driver, with no second pass.

python# DRIVER: sc.accumulator(value) is PySpark's ONLY accumulator constructor — there is no
# sc.longAccumulator(name) in PySpark (Scala/Java-only). It is UNNAMED.
bad_rows = sc.accumulator(0)

def validate_and_write(rows):
    for r in rows:
        if r["amount"] is None:
            bad_rows.add(1)                  # WORKER: only .add() — never read .value here
        # … write/emit the good row …

# ✅ Inside an ACTION (foreachPartition) -> each task's update applied EXACTLY ONCE.
orders.rdd.foreachPartition(validate_and_write)

# DRIVER ONLY reads the merged total — safe because the update ran inside an action.
print("bad rows:", bad_rows.value)

# VERIFY: PySpark's sc.accumulator() is UNNAMED, so it is typically NOT surfaced by name in
#   the Spark UI Stages -> Accumulators view. Confirm the count against a real aggregation:
print("true count:", orders.filter(orders.amount.isNull()).count())   # matches bad_rows.value
python# ❌ Anti-pattern: incrementing inside a TRANSFORMATION (lazy + may re-run).
seen = sc.accumulator(0)
counted = orders.rdd.map(lambda r: (seen.add(1), r)[1])   # transformation — NOT exactly-once
counted.count()        # runs once  -> seen == N …
counted.count()        # runs AGAIN (not cached) -> seen == 2N  ⚠️ inflated
# A retried/speculative task or a recomputed stage can double-count even with one action.
# Fix for a real count: orders.count() / orders.filter(cond).count()  (a true aggregation).

Uses, edge cases & limitations

Uses

  • Broadcast variable: a small static lookup (dict/set), a compiled regex set, or a small ML model referenced inside a UDF/mapInPandas/RDD map over a huge dataset.
  • Accumulator: lightweight side-channel metrics gathered while doing real work inside an action (foreach/foreachPartition) — bad-record counts, rows-dropped, sampled debug tallies.
  • Reach for the DataFrame broadcast() join hint (Lesson 02), not a broadcast variable, when the goal is avoiding the shuffle of a small table in a join.

Edge cases

  • A broadcast value that's almost too big — it must fit on the driver and in each executor's memory; a 2 GB "small" lookup pressures Storage memory and can OOM.
  • Mutating a broadcast value on a worker — read-only; local mutations are silently discarded. To change the lookup, re-broadcast a new variable.
  • Forgetting to release — a long-lived broadcast pins Storage memory (region R, Lesson 04) on every executor; unpersist()/destroy() when done.
  • Accumulator inside a transformation that runs twice — two actions over the same uncached DataFrame recompute and double the count; even one action can double-count on a retried/speculative task or recomputed stage.
  • Reading an accumulator on a worker — illegal/meaningless; read .value on the driver, after the action.

Limitations

  • Both are RDD/SparkContext-level APIs — used via sc.broadcast / sc.accumulator and read inside UDFs/rdd functions. For pure DataFrame work, prefer DataFrame-native tools (the broadcast() join hint; built-in aggregations).
  • Spark Connect / serverless: SparkContext.broadcast and SparkContext.accumulator are not supported on Spark Connect. On Databricks serverless / shared-access (Spark Connect) clusters sc isn't available the same way — use a broadcast join hint and DataFrame aggregations there. They work on classic single-user DBR clusters.
  • No exactly-once inside transformations — by design. Don't build correctness-critical counts on accumulators; use a real aggregation.

Common mistakes & gotchas

  • Confusing a broadcast variable with a broadcast join. sc.broadcast(dict) ships a value you read in a UDF; join(broadcast(df), …) is a join strategy. The #1 trap on this topic.
  • Capturing a big object in a closure instead of broadcasting it. Referencing a 50 MB dict directly in a UDF serializes it per task. Wrap it in sc.broadcast(...) and read bv.value.
  • Trusting an accumulator updated inside a map/filter. Exactly-once holds only inside actions; use foreach/foreachPartition, or a real aggregation.
  • Assuming an accumulator updates without an action. Accumulators are lazy — no action, no increment.
  • Reading acc.value on a worker. Workers are write-only; you get a partial, not the total. Read on the driver.
  • Never releasing a broadcast. It pins executor Storage memory and pressures GC (Lesson 10). unpersist()/destroy() when done.
  • Expecting a PySpark accumulator to show by name in the UI. sc.accumulator(0) is unnamed and PySpark has no sc.longAccumulator(name) (Scala/Java-only), so Python accumulators aren't surfaced by name in Stages → Accumulators. Verify by comparing acc.value to a real aggregation (filter().count()).
