Partition pruning
& dynamic pruning.

The cheapest data to process is the data you never read. Pruning skips whole directories of files — statically when you filter on a partition column, dynamically when you join a partitioned fact to a filtered dimension.

Spark 3.0+ / DBR LTS DPP default on Verified Jun 2026 docs

What it is

Partition pruning is Spark reading only the on-disk directories it needs and skipping the rest, instead of scanning every file in a table. It comes in two flavours:

  • Static partition pruning — a literal filter on a partition column (WHERE country = 'US') lets Spark scan only the matching directory, decided at plan time.
  • Dynamic Partition Pruning (DPP) — when a partitioned fact table is joined to a filtered dimension, Spark pushes the dimension's filter onto the fact's partition column at runtime — pruning fact partitions even though no literal was written against the fact.
The one fact to keep straight "Partition" means two different things in Spark. Pruning is about the on-disk Hive-style table partitions (PARTITIONED BY → one directory per value), not the in-memory Spark partitions you set with spark.sql.shuffle.partitions / repartition().

Why it matters

  • I/O is usually the first bottleneck. Before tuning a join or shuffle, the biggest cheap win is to read less data. A 5-year, daily-partitioned table queried for one day should touch ~1/1825 of the files — if pruning fires.
  • The classic enterprise query is a huge partitioned fact ⋈ a small filtered dimension (sales by order_date ⋈ date_dim WHERE quarter='Q4-2025'). Without DPP, Spark scans all of sales and filters after the join. With DPP, it scans only the Q4 partitions — often a 10×+ reduction in bytes read.
  • Interviewers probe it: "Your query reads the whole fact table even though you only want one region — why, and how do you fix it?" The answer is in PartitionFilters / dynamicpruningexpression in the plan.

How it works — deep dive

1 · Two meanings of "partition" — never confuse them

analogy On-disk partitions are a filing cabinet with one labelled drawer per value — you open only the drawers you need. In-memory partitions are how many people you split a task across once the folders are on the table. Pruning is about which drawers you open, not how many people you hire.

  • (a) On-disk Hive-style partitions — PARTITIONED BY (col) / partitionBy(col). Each distinct value becomes its own directory (/sales/order_date=2025-01-01/…). Pruning skips these.
  • (b) In-memory Spark partitions — the parallelism chunks, governed by spark.sql.shuffle.partitions (200) and repartition()/coalesce(). About parallelism, not file-skipping.

2 · How to create on-disk partitions

  • Mechanism: at write time Spark fans rows into one directory per distinct partition value, so a later filter on that column maps directly to a subset of directories.
  • Why: pre-organises the table by a column you frequently filter on — "scan everything then filter" becomes "scan only matching directories."
  • Trade-off — the cardinality rule: partition on a low-cardinality column (date, region, country). Partitioning a high-cardinality column (user_id, order_id) explodes the table into millions of tiny directories — the small-files problem plus huge driver-side metadata. Aim for partitions in the hundreds-of-MB-to-GB range.

use case events partitioned by event_date — dashboards filter by date, so every query prunes to a handful of day-directories.

3 · Static partition pruning

  • Mechanism: a literal filter on a partition column resolves at plan time; Spark lists only the matching directories and never opens the rest.
  • Why it's free: partition values are encoded in directory paths, so Spark prunes from the file listing before a single task runs.
  • Trade-off / the trap: it only fires when the column is filtered directly with a literal. Wrap it in a function (year(order_date)=2025) or compare it to another query's output and static pruning can't fire — the second case is exactly what DPP exists for.

use case WHERE country = 'US' on a country-partitioned table reads one directory instead of 195.

4 · Why partition pruning (the payoff)

  • The goal of the whole track is to do less work; pruning is the purest form — you don't optimise the scan, you avoid it. Fewer files → fewer tasks → less downstream shuffle, because less data ever entered the pipeline.
  • It composes with the other "read less" tools: column pruning (read only needed columns) and predicate pushdown (push row filters into the reader). Pruning skips whole directories; those two trim what's read inside the surviving files.

5 · Dynamic Partition Pruning (DPP)

analogy DPP is a bouncer who first checks the small VIP list (the filtered dimension), then only opens the cloakroom drawers (fact partitions) for names actually on it — instead of opening every drawer and checking each coat afterwards.

  • The problem: fact JOIN dim WHERE dim.x='Q4'. The filter is on dim, so static pruning can't help — at plan time Spark has no literal to prune the fact with, so it scans the entire fact and filters after the join.
  • Mechanism: with spark.sql.optimizer.dynamicPartitionPruning.enabled = true (default; since Spark 3.0.0), Spark runs the small filtered dimension first, collects the surviving join-key values (a broadcast/subquery result), and pushes them onto the fact's partition column at runtime — so the fact scan reads only matching partitions before the join.
  • Requirements (all must hold): the fact is partitioned on the join key; the filtered dimension is broadcastable; it's an equi-join on the partition column; and it is not a streaming query.
  • Trade-off: when it fires, a massive I/O win for free. When the dimension is too big to broadcast or the fact isn't partitioned on the join key, DPP simply doesn't engage — you fall back to a full fact scan.

use case sales (by order_date, 2 TB) ⋈ date_dim WHERE quarter='Q4-2025' — DPP prunes sales to ~90 day-partitions at runtime instead of scanning all 2 TB.

Reading it in the plan / UI Static → PartitionFilters: [(country = US)] on the Scan node, and "partitions read" drops in the Spark UI SQL tab. DPP → PartitionFilters: [dynamicpruningexpression(...)] + a SubqueryBroadcast / reused broadcast, and the fact Scan reports far fewer files read than the table's total.

The code you'll write (and how to verify it)

Track rule Every technique is paired with how to prove it worked — the .explain() plan line or the Spark-UI signal. Apply, then verify. Never assume.

Create on-disk partitions

python# partitionBy writes one directory per distinct order_date value.
# Pick a LOW-cardinality column (date/region) — never user_id/order_id.
(sales
   .write
   .partitionBy("order_date")                 # ← creates /order_date=.../ directories
   .mode("overwrite")
   .saveAsTable("main.pyspark_perf.sales"))    # UC 3-level name; Delta is the default

# VERIFY the table really is partitioned:
spark.sql("DESCRIBE EXTENDED main.pyspark_perf.sales").show(truncate=False)
#   look for "Partition Columns: [order_date]"

Equivalent Spark SQL — same effect:

sql-- Delta is the default on Databricks — don't write USING DELTA.
CREATE TABLE main.pyspark_perf.sales
PARTITIONED BY (order_date)
AS SELECT * FROM main.pyspark_perf.sales_staging;

Trigger static partition pruning

python# A LITERAL filter on the partition column → plan-time pruning.
one_day = spark.table("main.pyspark_perf.sales").where("order_date = '2025-01-01'")

# VERIFY: the scan node must carry a PartitionFilters line with your literal.
one_day.explain(mode="formatted")
#   (1) Scan ... main.pyspark_perf.sales
#       PartitionFilters: [isnotnull(order_date#..), (order_date#.. = 2025-01-01)]   ✅
#   Spark UI SQL tab: "number of partitions read" = 1 (not the full table)
python# ❌ Anti-pattern: a function on the partition column defeats static pruning.
slow = spark.table("main.pyspark_perf.sales").where("year(order_date) = 2025")
# explain() shows it under Filter / PushedFilters, NOT PartitionFilters → full scan.
# Fix: filter the raw column with a literal/range:
#   order_date >= '2025-01-01' AND order_date < '2026-01-01'

Trigger Dynamic Partition Pruning (fact ⋈ filtered dimension)

pythonfrom pyspark.sql.functions import col

sales    = spark.table("main.pyspark_perf.sales")      # partitioned by order_date (the join key)
date_dim = spark.table("main.pyspark_perf.date_dim")   # small dimension, broadcastable

# Filter lives on the DIMENSION, not the fact — only DPP can prune the fact here.
q = (sales.join(date_dim, sales.order_date == date_dim.d_date)
          .where(col("d_quarter") == "Q4-2025"))         # ← dim filter

# VERIFY: the fact scan's PartitionFilters carries a dynamicpruningexpression.
q.explain(mode="formatted")
#   (n) Scan ... sales
#       PartitionFilters: [dynamicpruningexpression(order_date#.. IN subquery#..)]   ✅
#   ...a SubqueryBroadcast / ReusedExchange feeds it the surviving dates.
#   Spark UI SQL tab: the `sales` Scan reads only the Q4 partitions.
python# DPP is on by default; A/B it by turning it OFF to see the before-state, then reset.
print(spark.conf.get("spark.sql.optimizer.dynamicPartitionPruning.enabled"))  # 'true'
spark.conf.set("spark.sql.optimizer.dynamicPartitionPruning.enabled", "false")  # demo only
# ...re-run q.explain() → dynamicpruningexpression gone, fact reads ALL partitions ❌
spark.conf.set("spark.sql.optimizer.dynamicPartitionPruning.enabled", "true")   # reset

Uses, edge cases & limitations

Uses

  • Date/region/country-partitioned fact tables queried with date-range or region filters → static pruning on every query.
  • Star-schema joins where the filter sits on a dimension (WHERE date_dim.quarter='Q4', WHERE store_dim.region='EMEA') → DPP prunes the partitioned fact at runtime.
  • Combine with column pruning and predicate pushdown (the rest of "read less") before reaching for join/shuffle tuning.

Edge cases

  • A function on the partition column (year(order_date)=2025, lower(country)='us') silently defeats static pruning → full scan. Filter the raw column with literals/ranges.
  • A UDF in the filter is a black box to Catalyst — it can't be pushed into PartitionFilters. Rewrite as native column expressions.
  • DPP needs a broadcastable dimension. If the filtered dimension is too big to broadcast, DPP won't fire — pre-filter/shrink it or accept the full fact scan.
  • Fact not partitioned on the join key → DPP cannot prune (it prunes partition directories, so the join key must be the partition column).
  • Over-partitioning a high-cardinality column gives pruning power you never use plus millions of tiny files and driver-side partition-metadata pressure (Lesson 03) — a net loss.

Limitations

  • DPP is not applied to streaming queries.
  • Pruning skips directories; it does nothing inside a surviving file — that's column pruning + pushdown's job.
  • Static pruning needs the partition column in the predicate; a filter only on a non-partition column gets no partition pruning (though Delta file-stats data-skipping can still help — a different mechanism).
  • spark.sql.optimizer.dynamicPartitionPruning.enabled is a runtime SQL conf (Spark 3.0+, default true); on a pre-3.0 engine DPP doesn't exist.

Common mistakes & gotchas

  • Confusing on-disk partitions with shuffle.partitions. Bumping spark.sql.shuffle.partitions does nothing for file pruning — wrong "partition." Pruning is about PARTITIONED BY directories.
  • Wrapping the partition column in a function and wondering why the whole table scans — year(order_date)=2025 ≠ a literal filter. Use order_date >= '2025-01-01' AND order_date < '2026-01-01'.
  • Expecting static pruning when the filter is on a dimension. A filter on dim can't statically prune fact — that's what DPP is for; make sure its requirements are met.
  • Over-partitioning on a high-cardinality column (user_id) — millions of one-row directories. Partition on low-cardinality columns; use bucketing (Lesson 11) or Z-ORDER / liquid clustering for high-cardinality keys.
  • Assuming DPP fired. It's silent when it doesn't. Confirm dynamicpruningexpression in .explain() and the reduced "partitions read" in the Spark UI.
  • Forgetting the dimension must be broadcastable. If your dimension grew past the broadcast threshold, DPP stops firing and fact scans go full again.
