Cache, persist
& storage levels.

Spark is lazy — by default it recomputes a DataFrame from scratch on every action. Cache and persist pin the computed result so later actions reuse it. Here's where the blocks land, why the first action pays, and when caching hurts instead of helps.

Spark 3.2+ / DBR LTS DataFrame default MEMORY_AND_DISK Verified Jun 2026 docs

What it is

By default a DataFrame holds only a recipe (its lineage of transformations), not its rows. Every action (count, write, collect) re-runs that recipe from the source. Caching stores the computed partitions so the next action reads them instead of recomputing:

  • df.cache() — shorthand for df.persist() with the default storage level.
  • df.persist(level) — same thing, but you pick the storage level (memory / disk / off-heap; replicated or not).
  • df.unpersist() — drop the cached blocks and free the memory. Eager (immediate), unlike cache/persist which are lazy.
The one rule to remember cache() == persist() with the default level. The most-confused fact: that default differs by object — RDD cache() = MEMORY_ONLY, but DataFrame cache() = MEMORY_AND_DISK (the PySpark enum is MEMORY_AND_DISK_DESER). Quote the right one for the right object.

Why it matters

  • Spark recomputes by default. Run three actions on the same DataFrame and Spark executes the full lineage — every read, filter, join, shuffle — three times. If that lineage holds an expensive shuffle, you pay for it on every action.
  • Caching trades memory for compute. Materialize once; later actions read the stored blocks. The classic win is iterative / branching work: ML training loops, repeated exploratory queries, or one intermediate result fed into several writes.
  • But caching is not free. Cached blocks pin Storage memory (the R region from Lesson 04), which can evict execution memory and trigger spill or GC pressure. Caching a DataFrame read once is pure overhead — the wrong instinct ("cache everything") makes jobs slower.

use case You clean and join a 500 GB events table once, then run 6 aggregations off it for a dashboard build — cache the cleaned DataFrame so the expensive join runs once, not six times.

How it works — deep dive

Quick recap: lazy lineage is why caching exists

analogy An uncached DataFrame is a recipe card — every time you want the dish you cook it from raw ingredients. Caching is cooking it once and putting it in the fridge: later you just reheat. The catch is fridge space (memory) is finite, and leaving food you'll never eat (a read-once DataFrame) just wastes the shelf. A DataFrame's transformations are lazy; the plan runs from the source on each action. persist() inserts a checkpoint — the first action computes and stores; later actions read the stored copy.

1 · cache() — the one-liner

  • Mechanism: df.cache() sets the storage level to the default and returns the DataFrame. It is lazy — nothing is stored until the first action.
  • The default level: for a DataFrame/Dataset it's MEMORY_AND_DISK — keep blocks in memory and spill any that don't fit to local disk (a too-big cache degrades to disk reads, not failure). PySpark surfaces this enum as MEMORY_AND_DISK_DESER ("Disk Memory Deserialized 1x Replicated").
  • Trade-off: zero ceremony, but you don't control the level.

2 · persist(level) — you choose where it lands

  • Mechanism: identical to cache() but you pass a StorageLevel encoding four choices: memory? disk? off-heap?, serialized?, replicated to 2 nodes?
  • When to override: DISK_ONLY when it's too big for RAM or you want zero heap/GC pressure; MEMORY_AND_DISK_2/DISK_ONLY_2 to replicate to two nodes so a lost executor doesn't force a recompute; OFF_HEAP to store serialized blocks outside the JVM heap (no GC) — experimental, needs spark.memory.offHeap.enabled=true and a positive spark.memory.offHeap.size.
  • Trade-off: more memory (or disk I/O, or 2× storage for _2) in exchange for fewer recomputes.

use case A cleaned DataFrame that's 2× cluster RAM → persist(StorageLevel.DISK_ONLY) so you reuse it without starving execution memory.

3 · Storage levels — the matrix & the PySpark trap

  • The flags: a level combines memory, disk, off-heap, serialized, and a replication count. Suffix _2 = replicate each block to two nodes (_3 exists for disk).
  • Full Scala/Java set: MEMORY_ONLY(_2), MEMORY_AND_DISK(_2), DISK_ONLY(_2/_3), MEMORY_ONLY_SER, MEMORY_AND_DISK_SER, OFF_HEAP.
  • PySpark nuance (interview trap): "In Python, stored objects will always be serialized with the Pickle library, so it does not matter whether you choose a serialized level." So PySpark doesn't separately expose _SER — the Python set is MEMORY_ONLY(_2), MEMORY_AND_DISK(_2), DISK_ONLY(_2/_3), OFF_HEAP. (DISK_ONLY_3 does exist in Python.)

4 · Lazy materialization — the first action pays

  • Mechanism: persist()/cache() only register the level — nothing is stored until the first action materializes it. That first action is slower (computes and stores); later actions read the cache and are fast.
  • Gotcha: if you cache() and never run an action, nothing is cached — the Storage tab stays empty. Force it with a cheap count() or noop write.

5 · unpersist() — eager cleanup

  • Mechanism: df.unpersist() immediately removes the cached blocks from memory and disk and clears the level. Unlike cache/persist it is eager. Signature: unpersist(blocking=False).
  • Why: cached blocks sit in Storage (R) until evicted or unpersisted — pinning memory and adding GC pressure. Always unpersist when done.
Reading it in the plan / UI A persisted DataFrame shows an InMemoryTableScan over an InMemoryRelation in .explain(mode="formatted") — that's proof the cache is used (the join/scan subtree is gone). The Spark UI Storage tab lists each cached DataFrame, its level, size in memory/on disk, and Fraction Cached (< 100% means it didn't all fit).

Walk the cache lifecycle, step by step

Click each phase. The surprise for most people: cache() stores nothing on its own — the first action pays to compute and store.

Cache lifecycle — lazy in, eager out

A DataFrame on top of a costly join, reused by two downstream actions.

01 df.persist(MEMORY_AND_DISK) — register the level
This sets the storage level on the DataFrame and returns it. It is lazy: no partitions are computed, nothing is stored, and the Spark UI Storage tab is still empty. You've only declared intent to cache.
02 First action (df.count()) — compute + STORE
The first action runs the full lineage (the expensive join/shuffle) and writes the resulting partitions into Storage memory (spilling overflow to disk). This action is slower than normal because it pays the compute and the store. Now the Storage tab shows the cached entry with its size and Fraction Cached.
03 Later actions — reuse the cache
Each subsequent action reads the stored blocks instead of re-running the join. In .explain() you'll see InMemoryTableScan where the SortMergeJoin / FileScan used to be. This is where caching pays off — two reuses turn one expensive join into one compute plus two cheap reads.
04 df.unpersist() — free it (eager)
When you're done, unpersist() immediately drops the blocks from memory and disk, returning Storage memory to the pool so execution and other caches can use it. Skipping this leaves stale blocks pinning R and pressuring GC for the rest of the session.
The whole game lives in step 02 being lazy and step 04 being eager. Cache only when steps 03 happen ≥ 2 times; otherwise step 02's tax buys you nothing.

The code you'll write (and how to verify it)

Track rule Every technique is paired with how to prove it worked — the .explain() plan node or the Spark-UI signal. Apply, then verify. Never assume.

Cache a reused DataFrame and confirm it's used

pythonfrom pyspark.storagelevel import StorageLevel

# expensive: a wide join we don't want to recompute on every downstream action
clean = (events.join(devices, "device_id")        # the costly shuffle/join
                .filter("event_ts > '2026-01-01'"))

clean.persist(StorageLevel.MEMORY_AND_DISK)        # pick the level explicitly
clean.count()                                      # FIRST action: materialize the cache (slow once)

# ...now several downstream actions reuse the stored blocks instead of re-joining:
a = clean.groupBy("country").count()
b = clean.where("is_fraud").count()

# VERIFY: the plan reads from cache, not from the join/source again.
a.explain(mode="formatted")
#   *(n) HashAggregate ...
#   +- InMemoryTableScan [country#..]            <-- reading the CACHE  ✅
#         +- InMemoryRelation [..], StorageLevel(disk, memory, deserialized, 1 replicas)
#   (no SortMergeJoin / no FileScan events here  <-- the join is NOT re-run ✅)

Spark UI → Storage tab: one cached entry, level Disk Memory Deserialized 1x Replicated, Fraction Cached 100%. If it's < 100%, the cache didn't all fit.

Equivalent in Spark SQL

sql-- CACHE TABLE is EAGER by default (materializes immediately) — opposite of df.cache().
-- LAZY makes it match the DataFrame API's lazy behaviour.
CACHE LAZY TABLE clean_events AS
SELECT e.*, d.device_type
FROM   events e JOIN devices d ON e.device_id = d.device_id;

UNCACHE TABLE clean_events;   -- free it when done (eager)
-- VERIFY: EXPLAIN SELECT ... shows InMemoryTableScan; Storage tab lists `clean_events`.

Choose a non-default level (too big for memory)

python# The cleaned DataFrame is 2× cluster RAM — don't fight for heap; cache to disk only.
huge.persist(StorageLevel.DISK_ONLY)   # PySpark also exposes DISK_ONLY_2 / DISK_ONLY_3
huge.count()                           # materialize
# VERIFY: Storage tab shows Size in Memory = 0, Size on Disk > 0, level "Disk Serialized 1x".

Always unpersist when done

pythonclean.unpersist()        # eager: releases Storage memory for execution / other caches
# VERIFY: the Storage tab entry disappears; later .explain() falls back to the source plan.

Contrast: read-once (don't cache) vs reused (do cache)

python# ❌ Anti-pattern: caching a DataFrame used by exactly ONE action — pure overhead.
once = big.filter("active").cache()
once.write.format("noop").mode("overwrite").save()   # only action → cache added cost for nothing

# ✅ Right: cache only when there are ≥ 2 actions, and unpersist after the last one.
reused = big.filter("active"); reused.persist(StorageLevel.MEMORY_AND_DISK); reused.count()
m1 = reused.groupBy("region").count(); m2 = reused.where("vip").count()   # 2 reuses → pays off
reused.unpersist()

Uses, edge cases & limitations

Uses

  • Reuse across ≥ 2 actions: an iterative ML loop, repeated exploratory queries on a cleaned DataFrame, or one intermediate result feeding several writes/branches.
  • Break an expensive lineage: a DataFrame on top of a costly join/shuffle hit multiple times — caching turns N recomputes of the join into one.
  • Fault tolerance on long jobs: _2 levels replicate blocks so losing an executor doesn't force a recompute of an expensive lineage.

Edge cases

  • cache() then no action → nothing is cached. Storage tab stays empty; later actions still recompute. Force it with count() / a noop write.
  • Caching a read-once DataFrame — pure overhead; the first-action "compute + store" tax buys nothing.
  • Cache that doesn't fit: MEMORY_ONLY silently recomputes missing partitions; MEMORY_AND_DISK spills them. Check Fraction Cached.
  • Over-caching evicts execution memory — too many cached blocks in R force shuffles/aggregations to spill or push toward OOM (Lesson 04).
  • Mutating the source after caching — the cache is a snapshot; new rows won't appear until you unpersist() and recompute (on Delta, re-read the table).

Limitations

  • Cache/persist are lazy; only unpersist() is eager. There's no "cache now" without an action.
  • A StorageLevel can't be changed once set without unpersist() first (persist refuses to reassign a level).
  • OFF_HEAP is experimental and needs spark.memory.offHeap.enabled=true + spark.memory.offHeap.size > 0.
  • Caching is not a substitute for pruning (Lesson 07) or fixing a bad join (Lesson 02) — it reuses an expensive result, it doesn't make it cheaper to compute.

Common mistakes & gotchas

  • Quoting the RDD default for a DataFrame. RDD cache() = MEMORY_ONLY; DataFrame cache() = MEMORY_AND_DISK (MEMORY_AND_DISK_DESER).
  • Expecting cache() to be eager. It's lazy — nothing is stored until the first action. df.cache() alone caches nothing.
  • Never calling unpersist(). Stale blocks pin Storage memory and pressure GC for the rest of the session.
  • Caching everything "to be safe." Over-caching evicts execution memory → spill / GC / OOM. The right default is not to cache.
  • Asking for MEMORY_ONLY_SER in PySpark. It isn't separately exposed — Python always serializes (Pickle).
  • CACHE TABLE is eager, df.cache() is lazy. The SQL form materializes immediately unless you write CACHE LAZY TABLE.
  • Assuming cache survives a cluster restart. It doesn't — cache is in-cluster, ephemeral, and snapshot-in-time.
