Sizing Spark Work

PARTITIONS · WAVES · CLUSTER CAPACITY

Performance tuning is partly about making the work the right size: enough partitions to keep cores busy, small enough partitions to avoid spill, and enough memory/overhead for the chosen operators.

shuffle partitionsAQE coalesceDBR LTS

01 · What it is

Partitions create tasks; cores run tasks

A stage runs one task per partition. Executor cores are task slots. If a stage has fewer partitions than cores, the cluster waits idle; if it has millions of tiny partitions, scheduling overhead dominates.

The one rule to rememberPartition count controls parallelism; partition size controls memory pressure. Tune for enough full task waves, not the highest possible partition count.

04 · Uses and gotchas

Sizing does not replace better plans

  • Use it to avoid idle cores, tiny tasks, huge spilling partitions, and tiny output files.
  • Edge case: one huge partition is skew, not a global partition-count problem.
  • Limitation: AQE coalesces post-shuffle partitions; it does not compact existing input files.
  • Gotcha: adding workers to a 20-partition stage will not use 200 cores.
