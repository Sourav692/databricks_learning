Sort-Merge, Shuffle-Hash
& Broadcast joins.

The same join can be a sub-second lookup or a cluster-melting shuffle. Spark picks one of three physical strategies from its size estimate — here's how each works, and how to control which one you get.

Spark 3.2+ / DBR LTS Broadcast threshold 10 MB Verified Jun 2026 docs

What it is

A join combines two DataFrames on a key. Spark can run that one logical join three physically different ways — and the choice decides whether the job moves gigabytes across the network or almost nothing:

  • Broadcast Hash Join (BHJ) — copy the small table to every executor, join locally, no shuffle of the big table.
  • Shuffle Sort-Merge Join (SMJ) — shuffle both tables by key, sort, merge. The default for big-vs-big.
  • Shuffle Hash Join (SHJ) — shuffle both, build an in-memory hash table on one side. A rarely-chosen middle ground.
The one rule to remember Spark picks the strategy from its size estimate of each side. Get the estimate (or the threshold) right and big-vs-small joins broadcast automatically; get it wrong and you pay for a full shuffle you didn't need.

Why it matters

  • A shuffle — Spark moving rows across the network so equal keys land together — is the single most expensive thing a join can do: network I/O + disk spill + serialization + a stage boundary that makes every executor wait for the slowest.
  • The classic enterprise join is a huge fact table ⋈ a small dimension (orders ⋈ countries, events ⋈ device-lookup). If it broadcasts, the fact table is never shuffled — often a 10× win. If it falls back to sort-merge, you shuffle the whole fact table for nothing.
  • Interviewers probe this constantly: "Your join is slow — what do you check?" The answer starts with "which strategy did the planner pick, and why?" — which you read from df.explain() and the Spark UI.

How it works — deep dive

Quick recap: the shuffle is the enemy

analogy A shuffle is like making every shopper in a mall walk to one central desk to be sorted by surname before they can be served — slow, and everyone waits for the longest queue. A join needs matching keys on the same executor: either move the data to the keys (shuffle both sides) or move a copy of the small table to every executor (broadcast). Every strategy is just a different answer to "how do we get matching keys together?"

1 · Broadcast Hash Join (BHJ) — the fast path

  • Mechanism: Spark collects the small side to the driver, ships a read-only copy to every executor (a BroadcastExchange), and each task probes that in-memory hash table while streaming its slice of the big side. The big side is never shuffled.
  • When Spark picks it: automatically when one side's estimated size ≤ spark.sql.autoBroadcastJoinThreshold (10 MB), or when you force it with broadcast() / a BROADCAST hint. AQE can also flip a sort-merge join to broadcast at runtime.
  • Trade-off: blazing fast and shuffle-free, but the small side must fit in the driver's and each executor's memory. Broadcast something too big → driver OOM / maxResultSize exceeded.

use case orders (2 TB) ⋈ country_codes (8 KB) — broadcast the country table; the 2 TB is read once, never shuffled.

2 · Shuffle Sort-Merge Join (SMJ) — the default for big-vs-big

  • Mechanism: both sides shuffled by key (two Exchange nodes), each partition sorted by key, then the two sorted streams merged in lockstep. Sorting is what lets it merge without holding either side fully in memory — so it scales to two enormous tables.
  • When Spark picks it: the default for large-vs-large equi-joins, because spark.sql.join.preferSortMergeJoin = true. Chosen whenever neither side is broadcastable.
  • Trade-off: robust and memory-friendly (streams, sorted), but you pay the full double shuffle + sort.

use case transactions (1 TB) ⋈ accounts (300 GB) on account_id — neither fits in memory, so sort-merge is correct.

3 · Shuffle Hash Join (SHJ) — the rare middle ground

  • Mechanism: both sides shuffled by key, then one side built into a per-partition hash table (no sort) and the other probes it.
  • When Spark picks it: only when neither side is broadcastable and preferSortMergeJoin = false (or a SHUFFLE_HASH hint), and one side fits in memory per partition.
  • Trade-off: skips the sort (can beat SMJ when one side is moderately small), but the hash table must fit in execution memory per partition — risky on skew, which is why Spark prefers sort-merge by default.

How Spark chooses (the decision order)

  1. Is there a hint? Priority: BROADCAST → MERGE → SHUFFLE_HASH → SHUFFLE_REPLICATE_NL.
  2. Else, one side estimated ≤ 10 MB? → Broadcast Hash Join.
  3. Else, equi-join with preferSortMergeJoin = true → Shuffle Sort-Merge Join.
  4. Else (small-ish side, preferSortMergeJoin = false) → Shuffle Hash Join.
  5. No equi-key → nested-loop / Cartesian (slow — usually a bug).
Why you sometimes must force broadcast() A filter or aggregation before the join shrinks a side — but pre-AQE, Spark may not re-estimate it, so it keeps planning a shuffle. That gap is exactly what AQE (Lesson 05) closes at runtime, and why a manual broadcast() is sometimes the fix.

Walk a sort-merge join, phase by phase

Click each phase to see what the engine actually does when neither side can be broadcast.

Sort-merge join execution

transactions ⋈ accounts on account_id — both too big to broadcast.

01 Shuffle both sides by the join key
Each side is repartitioned so all rows with the same account_id land in the same partition number on both sides. This is the expensive part — two Exchange nodes, gigabytes over the network and to local disk. The post-shuffle partition count is spark.sql.shuffle.partitions (200 by default; AQE then coalesces it).
02 Sort each partition by the key
Within every partition, both sides are sorted by account_id. Sorting is what makes the next step cheap: you never have to hold a whole side in memory — you stream two ordered lists.
03 Merge the two sorted streams
Walk both sorted partitions in lockstep like merging two sorted lists; whenever the keys match, emit the joined row. Memory stays flat because it's a streaming merge — this is why SMJ scales to two huge tables where a hash join might OOM.
Cost lives in step 01. Eliminate it by broadcasting a small side, or — for a join you run repeatedly on the same key — by bucketing both tables once on write (Lesson 11), so the shuffle never happens again.

The code you'll write (and how to verify it)

Track rule Every technique is paired with how to prove it worked — the .explain() plan node or the Spark-UI signal. Apply, then verify. Never assume.

Force (or guarantee) a broadcast join

pythonfrom pyspark.sql.functions import broadcast

# orders is huge; country is tiny. broadcast() tells the planner:
# ship `country` to every executor and DO NOT shuffle `orders`.
joined = orders.join(broadcast(country), on="country_code", how="inner")

# VERIFY: BroadcastHashJoin + a BroadcastExchange on the country side,
# and NO Exchange on the orders side.
joined.explain(mode="formatted")
#   *(n) BroadcastHashJoin ... (orders.country_code = country.country_code)
#   :- *(n) <orders scan, no Exchange>          <-- big side NOT shuffled  ✅
#   +- BroadcastExchange ...                    <-- small side broadcast    ✅

Equivalent Spark SQL hint — same effect:

sql-- The hint goes immediately after SELECT. BROADCAST(country) == broadcast(country).
SELECT /*+ BROADCAST(country) */ o.*, c.country_name
FROM   orders o
JOIN   country c ON o.country_code = c.country_code;

Tune the auto-broadcast threshold

python# Raise the threshold so a ~50 MB dimension still auto-broadcasts (default 10 MB).
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", 50 * 1024 * 1024)  # bytes

# Or disable auto-broadcast entirely to FORCE a sort-merge join (to A/B a plan):
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", -1)  # -1 = never auto-broadcast

# VERIFY by re-running .explain() and confirming the join node flipped
# (BroadcastHashJoin <-> SortMergeJoin) — then reset the conf.

Contrast: naive big-vs-small vs broadcast

python# ❌ Naive: if Spark mis-estimates `country` as big, it shuffles the 2 TB orders table.
slow = orders.join(country, "country_code")            # may plan SortMergeJoin → double shuffle

# ✅ Right: force the broadcast so only the tiny side moves.
fast = orders.join(broadcast(country), "country_code")  # BroadcastHashJoin → no big-side shuffle
# Spark UI SQL tab: `fast` has no Exchange on the orders branch; `slow` has two.

Uses, edge cases & limitations

Uses

  • Big fact ⋈ small dimension → broadcast (the highest-leverage join tuning there is).
  • Big ⋈ big on a shared key → sort-merge, and consider bucketing (Lesson 11) to pre-shuffle once and skip it on every future read.
  • Let AQE (Lesson 05) switch sort-merge → broadcast at runtime when the small side only becomes small after a filter.

Edge cases

  • A dimension almost under the threshold: a column projection / filter can shrink it enough — but pre-AQE Spark may not re-estimate, so force broadcast().
  • Broadcasting the wrong (big) side: broadcast() is a hint; broadcasting a 5 GB table OOMs the driver / blows spark.driver.maxResultSize (Lesson 03).
  • Outer joins limit broadcast: you can broadcast only the side allowed to be the build side (e.g. the right side of a LEFT outer join).
  • No equi-key (a.x > b.y): falls back to a nested-loop join — broadcast a tiny side or rewrite the query.

Limitations

  • BHJ needs the small side to fit in driver and executor memory — there's no streamed broadcast. Above your budget, sort-merge (or bucketing) is the only safe choice.
  • The auto-broadcast decision uses an estimate; bad/missing stats → wrong choice. Run ANALYZE TABLE … COMPUTE STATISTICS or rely on AQE's runtime sizes.
  • autoBroadcastJoinThreshold is 10 MB in OSS Spark; on Databricks, AQE also uses a higher runtime switch (spark.databricks.adaptive.autoBroadcastJoinThreshold, 30 MB). State which you mean.

Common mistakes & gotchas

  • Reading the plan wrong. The BroadcastExchange should be on the small side. If it's on your big table, you're broadcasting the wrong one.
  • Forcing broadcast on a table that grows. An 8 MB dimension today becomes an 80 MB OOM next quarter. Prefer AQE; hint only with a known cap.
  • Bumping the threshold globally to fix one join changes every join in the session. Scope it, or use a per-query broadcast()/hint.
  • Assuming a small DataFrame broadcasts. If it's the output of a wide transformation, its estimated size may be huge — check .explain().
  • Confusing a broadcast join with a broadcast variable (Lesson 09). One is a join strategy; the other is a value you reference in a UDF/map.
  • Disabling AQE to "control" joins. Its runtime SMJ→broadcast switch and skew-join split usually beat hand-tuning. Keep it on.
