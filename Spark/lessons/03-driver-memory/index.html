Driver memory
& driver OOM.

There are many executors but only one driver — a single fixed heap. One oversized collect(), one over-broadcast table, or one partition-count explosion can sink a job that 200 healthy executors were running perfectly.

Spark 3.2+ / DBR LTS driver.memory 1g maxResultSize 1g Verified Jun 2026 docs

What it is

Every Spark application has exactly one driver — the JVM that runs your notebook, holds the SparkSession, builds the plan, and schedules tasks. Its heap (spark.driver.memory, default 1g) is fixed and finite. A driver OOM is what happens when something you asked Spark to bring back to the driver doesn't fit.

Four kinds of data live on the driver — each a way to fill that heap:

  • The plan & scheduling state — the logical/physical plan and the bookkeeping for every job, stage, task, and partition.
  • Action results — whatever an action pulls back: collect(), take(), head(), toPandas(), show().
  • The broadcast build — the small side of a broadcast join is collected to the driver first, then shipped to every executor.
  • Partition / file metadata — for a job touching a huge number of partitions or files, the driver holds the listing and per-partition metadata.
The one rule to remember Executors scale out (add nodes for more data); the driver does not. If you move data to the driver — by collect(), by broadcasting, or by exploding partition counts — you are loading one fixed heap. Keep big data on the executors.

Why it matters

  • The driver is a single point of failure. When the driver dies, the whole application dies — every executor's work is thrown away. A slow executor degrades one task; a dead driver kills the job.
  • collect() on a big result is the #1 cause. A display() of millions of rows, a toPandas() on a fact table — each tries to serialize every partition back into one heap. The docs are explicit: results above spark.driver.maxResultSize (default 1g) abort the job, and raising that limit "may cause out-of-memory errors in driver."
  • A mis-sized broadcast hits the driver, not the executor. A broadcast join collects the small side to the driver and ships it out — broadcast a table that's too big and the driver OOMs while building it, before any executor sees it.
  • Interviewers probe the asymmetry: "Ran fine for months, then started failing with no executor-side data growth — where do you look?" Often the driver: a result set, a broadcast, or a partition-count explosion that crept over the limit.

How it works — deep dive

Driver memory management: what the driver holds

analogy The driver is the air-traffic controller in the tower — it doesn't fly the planes (the executors do), but it tracks every flight. Hand the controller a job meant for the whole fleet (collect all the cargo to the tower) and the tower is crushed.

The driver heap (spark.driver.memory, default 1g) is the JVM heap of the driver process. On Databricks it runs on the cluster's driver node; in OSS client mode it runs in the submitting/edge process, in cluster mode inside the cluster. Either way it is one heap, and these compete for it:

  • The plan and the scheduler — Catalyst's plan, the DAG scheduler, task-set state. Cheap normally; expensive when a job fans out to hundreds of thousands of partitions/files.
  • Action results — collect() (all rows), take(n)/head(n) (n rows), toPandas() (all rows, as pandas), show() (a capped preview, usually safe). Serialized on executors, deserialized into the driver heap.
  • The broadcast build — for a broadcast hash join, Spark collects the small side to the driver, builds the broadcast relation, then ships it to every executor. So the broadcast size budget is a driver budget.
The overhead region (a subtle second OOM surface) Cluster-mode drivers also reserve spark.driver.memoryOverhead = driverMemory × spark.driver.memoryOverheadFactor (factor 0.10; 0.40 for non-JVM Kubernetes), floored at spark.driver.minMemoryOverhead = 384m (settable key since Spark 4.0.0; the 384 MB floor predates it). Overhead covers off-heap and the PySpark driver process — so heavy toPandas() / local-Python work pressures overhead, not the JVM heap.

spark.driver.maxResultSize: the guardrail

use case A guardrail so a stray collect() aborts cleanly with a clear error instead of silently hanging the cluster while the heap thrashes and GC pauses pile up.

  • What it is (doc-quoted): "Limit of total size of serialized results of all partitions for each Spark action (e.g. collect). Jobs will be aborted if the total size is above this limit. Having a high limit may cause out-of-memory errors in driver." Default 1g.
  • Mechanism: as task results stream back, Spark sums their serialized sizes; the instant the running total crosses maxResultSize, the job is aborted — a pre-emptive abort, before the heap is necessarily exhausted.
  • Why it exists: it converts a creeping driver OOM (slow, GC-thrashing, hard to diagnose) into a fast, explicit failure with an actionable message.
  • The trap: raising it (or 0 = unlimited) removes the guardrail, so a too-big collect() sails past the abort into a real OutOfMemoryError. Raise it only when you've sized spark.driver.memory to match.

Why & how driver OOM happens — the four paths

  • (a) collect() / toPandas() of too many rows — the classic, doc-grounded path. Every partition is serialized and pulled into one heap. Symptom: the maxResultSize abort, or a driver OutOfMemoryError if the cap was raised. Fix: write it, or pull a bounded take(n)/limit(n).
  • (b) Broadcasting a too-large table — driver OOM during the build. The small side is collected to the driver first; force broadcast() on a 3 GB "dimension" and the driver OOMs before any executor receives it. Look for a BroadcastExchange in the plan. (Operationally well-known; not a doc quote.)
  • (c) A huge partition / file count — metadata OOM. A job that fans out to hundreds of thousands of tasks can exhaust the heap with metadata alone, no result collected. Fix: coalesce, sane shuffle.partitions, compaction, AQE coalesce. (Established practice, not a doc quote.)
  • (d) Large local Python objects — a giant list / pandas frame / model in driver code pressures the PySpark driver process (overhead). Fix: keep heavy data in DataFrames; materialize only summaries.
Driver OOM ≠ executor OOM Driver OOM = data brought to the driver (collect / broadcast-build / metadata). Executor OOM (Lesson 04) = a partition too big to process or spill on a worker. Different node, different log, different fix.

Walk a collect() vs a write(), step by step

Click each phase to see exactly where the rows travel — and why one floods the driver while the other never touches it.

Where the rows go

The same DataFrame, two terminal actions.

01 Executors compute the partitions (both cases)
Every executor runs its tasks and produces output partitions in executor memory/disk. So far the driver only holds the plan and scheduling state — identical for collect() and write().
02a collect(): serialize every partition back to the driver
Each task serializes its result and ships it to the driver, which deserializes all of them into one heap. Spark sums the serialized sizes as they arrive — the moment the total crosses maxResultSize (1g), the job is aborted. If the cap was raised, the heap fills until a real OutOfMemoryError.
02b write(): executors write their own partitions to storage
Each executor writes its partition directly to the table/files in parallel. Nothing is returned to the driver — the driver only collects tiny commit metadata. The driver heap stays flat regardless of result size. write.format("noop") does the same to run a job for timing without writing output.
03 Need rows locally? Bound the pull
If you truly need rows on the driver, never pull the whole thing: take(n) / limit(n).collect() caps it at n rows, and aggregating first (groupBy().agg()) returns a few summary rows instead of the raw table.
Cost lives in step 02a. The fix is to prefer 02b (write / noop) and, when you must pull rows, to bound them (take / limit / aggregate).

The code you'll write (and how to verify it)

Track rule Every technique is paired with how to prove it worked — the .explain() plan node or the Spark-UI signal. Apply, then verify. Never assume.

Replace collect() with a write (or a bounded pull)

python# ❌ Naive: pulls EVERY row of a fact table into the 1g driver heap → maxResultSize abort or OOM.
rows = big_df.collect()                      # all partitions serialized into one heap
pdf  = big_df.toPandas()                     # same hazard — every row, as a pandas frame

# ✅ Right: keep the data on the executors. Write it out…
big_df.write.mode("overwrite").saveAsTable("main.pyspark_perf_demo.result")

# …or, to RUN the job without returning rows (timing / materialize a plan), use the noop sink:
big_df.write.format("noop").mode("overwrite").save()   # full job, nothing pulled to the driver

# ✅ Need a few rows locally? Pull a BOUNDED set, never the whole thing:
sample = big_df.limit(1000).collect()        # at most 1000 rows reach the driver

# VERIFY: Spark UI → Jobs/SQL tab. The collect() job returns a final result task set to the
# driver; the write/noop job returns NOTHING. If collect() is too big, the driver log shows:
#   "Total size of serialized results ... is bigger than spark.driver.maxResultSize"

Aggregate on the cluster, then collect the small summary

pythonfrom pyspark.sql import functions as F

# ❌ Don't collect the raw rows to compute a total on the driver.
total_bad = sum(r["amount"] for r in big_df.collect())     # pulls every row — driver hazard

# ✅ Do the aggregation on the executors; collect only the tiny result.
total_ok = big_df.agg(F.sum("amount").alias("t")).collect()[0]["t"]   # 1 row to the driver

# VERIFY: .explain() shows HashAggregate (partial + final) on the cluster; the action returns
# a single row. Spark UI → SQL tab: the final exchange is tiny, not the whole table.

Guard against a mis-sized broadcast (driver-side OOM)

pythonfrom pyspark.sql.functions import broadcast

# ❌ Forcing broadcast on a table that's actually large OOMs the DRIVER during the build.
risky = fact.join(broadcast(too_big_dim), "k")   # small side collected to driver first → OOM

# ✅ Only broadcast a genuinely small side; otherwise let the planner / AQE choose.
ok = fact.join(small_dim, "k")                    # planner broadcasts only if under the threshold

# Cap auto-broadcast so a creeping dimension can't silently start OOMing the driver:
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", 10 * 1024 * 1024)  # 10 MB (OSS default)

# VERIFY: fact.join(...).explain(mode="formatted") — a BroadcastExchange feeding the SMALL side
# is fine; a BroadcastExchange on a multi-GB side is the warning sign. The OOM, if it comes, is
# on the DRIVER (driver log), not an executor.

Size the limits deliberately, in pairs

python# Read the current budget. These are the levers; size them TOGETHER, deliberately.
print("driver.memory        =", spark.conf.get("spark.driver.memory"))         # '1g' — heap
print("driver.maxResultSize =", spark.conf.get("spark.driver.maxResultSize"))  # '1g' — guardrail

# maxResultSize CAN be set at runtime; raise the GUARDRAIL only with a matching HEAP.
spark.conf.set("spark.driver.maxResultSize", "2g")
# spark.driver.memory is a LAUNCH-TIME JVM flag — set it in the cluster Spark config (restart):
#   spark.driver.memory 8g
# Never set maxResultSize 0 (unlimited) to "make the error go away" — that removes the guardrail.

# VERIFY in the Spark UI → Executors tab: the single 'driver' row shows its memory + GC time.
# Rising driver GC time during a collect() = the heap is under pressure (the OOM warning sign).

Uses, edge cases & limitations

Uses

  • maxResultSize as a safety guardrail so a stray collect() fails fast and clearly instead of silently thrashing the driver heap.
  • Raising spark.driver.memory deliberately for a legitimately large-but-bounded local result (a model object, a moderate aggregation) — paired with a matching maxResultSize.
  • The noop sink to run a job (timing / materializing a plan) with zero data returned to the driver — the safe alternative to collect() for benchmarking.

Edge cases

  • A broadcast that's almost too big. A dimension at 8 MB today, 80 MB next quarter, silently starts OOMing the driver during the build — not the executors. Cap the threshold; let AQE decide.
  • toPandas() hits two limits. It pulls every row into the JVM heap and converts to a pandas frame in the Python driver process (overhead) — it can OOM the container even when the JVM heap looks okay.
  • Partition-count explosion with no big result. repartition(500000) or millions of tiny files can OOM the driver on metadata / scheduling alone, with no collect() in your code.
  • Raised maxResultSize without raised heap. Removing the guardrail just converts a clean abort into a real, harder-to-diagnose driver OOM.

Limitations

  • The driver does not scale out. There is exactly one, with one fixed heap — you cannot "add driver nodes." Above the heap: bring less back or a bigger driver node.
  • spark.driver.memory is launch-time. It's a JVM flag set when the driver starts — set it in the cluster Spark config (a restart), not via spark.conf.set(...) at runtime. maxResultSize can be set at runtime.
  • The broadcast-build and metadata OOM paths aren't verbatim in the docs. Only the collect()/large-result path (via maxResultSize) is doc-quoted; the others are well-established operational practice — true, but stated as such.

Common mistakes & gotchas

  • collect() to "just look at the data." Use df.show(20) or df.limit(20).toPandas() — never collect() / toPandas() on a full fact table.
  • Setting maxResultSize=0 to silence the error. That removes the guardrail and trades a clean abort for a real driver OOM. Size the limit; don't disable it.
  • Blaming the executors for a broadcast OOM. A too-big broadcast() OOMs the driver during the build — check the driver log and look for a BroadcastExchange.
  • Bumping spark.driver.memory at runtime. It's a launch-time JVM flag; spark.conf.set won't take effect until it's set in the cluster config and the cluster restarts.
  • toPandas() on a big DataFrame for "convenience". The riskiest driver call there is — every row, into both the JVM heap and the Python process.
  • Confusing driver OOM with executor OOM (Lesson 04). Driver OOM = data brought to the driver. Executor OOM = a partition too big to process/spill on a worker.
