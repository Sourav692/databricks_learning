Bucketing to
eliminate the shuffle.

When you can't broadcast and you join two big tables on the same key every day, you re-pay the full double shuffle every run. Bucketing pays it once, on write, and removes the Exchange from every future join.

Spark 3.2+ / DBR LTS bucketing.enabled true (2.0+) coalesceBucketsInJoin false (3.1+) Verified Jun 2026 docs

What it is

Bucketing physically pre-shuffles a table by a key once, when you write it, and stores the result as a fixed number of hash-partitioned files ("buckets"). Later, any join or aggregation on that key can skip the shuffle, because the data already sits where the engine would have moved it:

  • You write with df.write.bucketBy(N, "key").sortBy("key").saveAsTable("cat.sch.tbl").
  • Spark hashes every row's key and sends it to bucket hash(key) % N, and (with sortBy) sorts within each bucket.
  • Two tables bucketed by the same key into the same N have matching layouts — corresponding buckets are co-located — so the planner omits the Exchange (shuffle) before the join.
The one rule to remember The expensive part of a shuffle join is the Exchange that moves both sides across the network. Bucketing pays that cost once on write and reuses it on every read — but only if both tables share the same key and the same bucket count. Mismatch either and the shuffle comes right back.

Why it matters

  • A shuffle — Spark moving rows across the network so equal keys land together — is the most expensive thing a wide operation does: network I/O + disk spill + serialization + a stage boundary where every executor waits for the slowest.
  • A broadcast join (Lesson 02) removes the shuffle only when one side is small. When you join big ⋈ big repeatedly on the same key — a daily transactions ⋈ accounts — neither side broadcasts and you re-pay the full double shuffle every run. Bucketing fixes that: shuffle once on write, never again.
  • Interviewers love the contrast: "You can't broadcast and the join is the same every day — what do you do?" The senior answer is "bucket both tables on the join key into matching buckets so the Exchange disappears" — and you prove it in df.explain().

How it works — deep dive

Quick recap: why a shuffle join is expensive

analogy A shuffle join is like two delivery companies each re-sorting all their parcels by postcode at the depot, every morning, before they can be matched. Bucketing is pre-sorting the warehouse shelves by postcode once so the parcels are already grouped — every future match just reads the right shelf. A sort-merge join (Lesson 02) reaches "matching keys in the same partition" by shuffling both sides into spark.sql.shuffle.partitions (200) hash buckets. Bucketing performs that exact hash-partitioning ahead of time and stores it.

1 · Writing a bucketed table (bucketBy + sortBy + saveAsTable)

  • Mechanism: bucketBy(N, "key") hash-partitions the output into exactly N files using HashPartitioning(key). sortBy("key") sorts each bucket, so a later join does an efficient sort-merge inside each bucket with no separate sort stage. saveAsTable(...) stores the bucket spec (key + count) in the metastore / Unity Catalog for every future query.
  • Why it works: when the planner sees a join whose required distribution is HashPartitioning(key, n) and the table already advertises exactly that, it knows the data is already where it needs to be and drops the pre-join Exchange.
  • Trade-off: you pay a full shuffle + sort at write time (and re-pay it on every full rewrite). It only pays off when the table is joined on that key many times.

use case a 2 TB transactions table joined to a 300 GB accounts table on account_id every day — bucket both by account_id into the same N once; every daily join then runs with no Exchange.

2 · Two matching buckets → no Exchange (the core mechanism)

  • Mechanism: for the planner to skip the shuffle, two conditions must hold: (1) the number of partitions on both sides must be exactly the same (same bucket count), and (2) both join operators must use HashPartitioning on the join key — which is exactly what bucketBy produces.
  • Why it works: bucket i of table A and bucket i of table B contain the same set of keys (same hash, same modulus), so they join locally, partition-for-partition, with no data movement.
  • Trade-off: brittle — it holds only when the key and count line up on both sides. Bucket into 256 on one and 200 on the other and the planner shuffles one side to reconcile them.

analogy two decks of cards already sorted into the same 256 numbered trays — to find matching pairs you compare tray-to-tray; you never re-deal.

3 · Bucketing vs partitioning (complementary, not rivals)

  • Partitioning (partitionBy, Lesson 07) → directories, one per value. Great for pruning a low-cardinality filter column (date, region).
  • Bucketing (bucketBy) → a fixed number of files by hash. Great for join/agg keys, often high-cardinality (user id, account id) — exactly the columns you must NOT partition by, or you'd explode into millions of tiny directories.
  • They stack: partitionBy("event_date").bucketBy(256, "user_id").sortBy("user_id") prunes by date and removes the shuffle on user_id joins.

use case an events table partitioned by event_date (one directory per day) and bucketed by user_id (no shuffle on the join to users).

4 · Mismatched bucket counts & coalesceBucketsInJoin

  • Mechanism: different bucket counts → mismatched HashPartitioning → the planner shuffles one side. Since 3.1, spark.sql.bucketing.coalesceBucketsInJoin.enabled lets Spark coalesce the larger side to the smaller count when one count is an exact multiple of the other (e.g. 256 → 64), avoiding the shuffle.
  • Interview gotcha: the default is false (since 3.1.0). It is easy to assume it's on — it is not. And it only helps for multiple-of counts; 200 vs 256 still shuffles.
  • Trade-off: coalescing reads the larger side as fewer, bigger buckets — convenient, but the best practice is to pick one bucket count and use it on both tables.

5 · The classic flag vs the DataSource-V2 flag (don't confuse them)

  • spark.sql.sources.bucketing.enabled = true (default, since 2.0.0) — the classic file-source bucketing this lesson is about.
  • spark.sql.sources.v2.bucketing.enabled = true (since 3.3.0) — a different feature: the DataSource-V2 storage-partitioned join for V2 connectors that report their own partitioning. Same word, different machinery — don't cite the V2 flag for classic bucketing.
Bucketing vs AQE vs broadcast AQE (Lesson 05) coalesces post-shuffle partitions — it never retro-fits bucketing, and a one-sided bucketing still shuffles the other side. Broadcast (Lesson 02) removes the shuffle only when a side is small. Bucketing is the tool for big ⋈ big on the same key, repeatedly.

Walk the write path, phase by phase

Click each phase to see what bucketBy(...).sortBy(...).saveAsTable(...) actually does — and why the cost moves from read time to write time.

What bucketing does on write

Writing transactions bucketed by account_id into N buckets.

01 Hash-partition by the key into N buckets
Every row is sent to bucket hash(account_id) % N — a one-time shuffle at write time, the same hash-partitioning a sort-merge join would do at read time. This is the cost you're moving from "every read" to "once on write".
02 Sort each bucket by the key (sortBy)
Within each bucket the rows are sorted by account_id. A later sort-merge join can then merge bucket-for-bucket with no separate Sort stage — the data is already ordered on disk.
03 Register the bucket spec in the metastore (saveAsTable)
The key and bucket count are stored as table metadata in Unity Catalog. This step is mandatory — bucketing is not supported for save, insertInto, or jdbc; only a registered table carries the spec the planner reads.
04 Every future join inherits the layout — no Exchange
When you join two such tables on account_id with the same N, the planner sees both already satisfy HashPartitioning(account_id, N) and drops the pre-join shuffle. The write-time cost is amortized across all future reads.
The whole point: move the shuffle from read time to write time, then reuse it. Worth it when the table is joined on that key many times — not for a one-off read.

The code you'll write (and how to verify it)

Track rule Every technique is paired with how to prove it worked — the .explain() plan node or the Spark-UI signal. Apply, then verify. Never assume.

Write two tables bucketed by the same key into the same N

python# Bucket BOTH tables by the join key (account_id) into the SAME number of buckets.
# bucketBy(n, key) -> HashPartitioning(account_id) into n files (the pre-shuffle)
# sortBy(key)      -> sort within each bucket so the read-side join skips its Sort too
# saveAsTable(...) -> REQUIRED: bucketing only works through a metastore/UC table
N = 256

(transactions.write
    .bucketBy(N, "account_id").sortBy("account_id")
    .mode("overwrite")
    .saveAsTable("main.pyspark_perf_demo.transactions_bucketed"))

(accounts.write
    .bucketBy(N, "account_id").sortBy("account_id")   # SAME key, SAME N
    .mode("overwrite")
    .saveAsTable("main.pyspark_perf_demo.accounts_bucketed"))

# VERIFY the stored bucket spec:
spark.sql("DESCRIBE EXTENDED main.pyspark_perf_demo.transactions_bucketed").show(50, False)
#   -> Num Buckets: 256   Bucket Columns: [account_id]   Sort Columns: [account_id]

spark.sql.sources.bucketing.enabled is true by default — you don't set it to use bucketing; you'd only set it to false to prove the before-state.

Join the bucketed tables and confirm the Exchange is gone

pythont = spark.table("main.pyspark_perf_demo.transactions_bucketed")

Combine partitioning (prune) with bucketing (no shuffle)

python# Prune by a low-cardinality filter column AND remove the shuffle on the join key.
(events.write
    .partitionBy("event_date")                     # directories -> partition pruning (L07)
    .bucketBy(256, "user_id").sortBy("user_id")     # hash files  -> no shuffle on user_id
    .mode("overwrite")
    .saveAsTable("main.pyspark_perf_demo.events_bpb"))

# VERIFY: the plan shows PartitionFilters on event_date AND no Exchange on the user_id join.

Handle mismatched bucket counts (only when one is a multiple of the other)

python# If two tables are bucketed 256 and 64 on the same key, the counts don't match ->
# by default Spark shuffles one side. Opt in to coalesce the larger side to 64.
spark.conf.set("spark.sql.bucketing.coalesceBucketsInJoin.enabled", True)  # default False (3.1+)

# VERIFY: re-run .explain(); the 256-bucket side now reads as 64 buckets, Exchange gone.
# Best practice is still to pick ONE bucket count for both tables.

Uses, edge cases & limitations

Uses

  • Big ⋈ big on the same key, run repeatedly — the canonical case bucketing exists for. Shuffle once on write, skip it on every read.
  • Repeated aggregations on the same key — a groupBy(key) on a table bucketed by key also avoids the shuffle.
  • Partition + bucket together — partition by a low-cardinality filter column to prune, bucket by the high-cardinality join key to skip the shuffle.

Edge cases

  • Mismatched bucket counts — different N means the layouts don't match, so a shuffle returns. coalesceBucketsInJoin (off by default) only saves you when one count is an exact multiple of the other; 200 vs 256 still shuffles.
  • Different bucket key than the join key — bucketing on customer_id does nothing for a join on account_id.
  • AQE doesn't retro-fit bucketing — AQE coalesces post-shuffle partitions; if only one side is bucketed, the other still shuffles.
  • Tiny tables — if one side broadcasts, broadcast it; bucketing is for the big-vs-big case.
  • Over-bucketing — too many buckets → many tiny files; too few → giant, skew-prone buckets. Size N to the data.

Limitations

  • saveAsTable() only. Bucketing is not supported for DataFrameWriter.save, .insertInto, or .jdbc — you must write a metastore / UC-registered table, not plain files.
  • Both sides must be bucketed the same way to skip the shuffle. One alone doesn't help — the other still shuffles to match.
  • Re-bucketing cost. Changing the key or count means rewriting (re-shuffling) the whole table.
  • Classic ≠ DataSource-V2. spark.sql.sources.bucketing.enabled (this lesson) is not spark.sql.sources.v2.bucketing.enabled (since 3.3.0, the storage-partitioned-join flag).

Common mistakes & gotchas

  • Using save()/insertInto()/jdbc() with bucketBy. Bucketing requires a registered table — it's a no-op (or errors) outside saveAsTable().
  • Bucketing only one side of the join. Both tables must share the same key and the same N, or the unbucketed side shuffles and you've gained nothing.
  • Assuming coalesceBucketsInJoin is on. It defaults to false (since 3.1.0), and even on it only works when one count is a multiple of the other.
  • Mismatched counts. 200 vs 256 buckets on the same key still triggers a shuffle. Pick one number and standardize it.
  • Over- or under-bucketing. Bucket count drives file size — too many → tiny files, too few → giant buckets. Choose N from the table's size and key cardinality.
  • Bucketing a column you never join on. It costs a write-time shuffle for nothing unless the bucket key is the key you actually join / aggregate by, repeatedly.
  • Confusing the classic flag with the V2 flag. spark.sql.sources.bucketing.enabled vs spark.sql.sources.v2.bucketing.enabled are different features.
