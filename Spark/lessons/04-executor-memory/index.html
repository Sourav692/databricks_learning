Executor memory:
unified, spill & OOM.

Where the data actually lives while a task runs — Spark's four heap regions, the borrow-and-evict rules that let execution beat cache, and why the same job either finishes, quietly spills to disk, or dies with an OOM.

What it is

An executor is a JVM process on a worker node that runs your tasks and holds cached data. Its heap is not one undifferentiated pool — Spark's UnifiedMemoryManager carves it into four regions, and which region runs out first decides whether your job is fast, slow (spills to disk), or dead (OOM):

  • Reserved (300 MiB) — a fixed slice carved off first for Spark internals.
  • User memory (~40%) — your data structures, UDF state, Spark metadata; not managed by the unified manager.
  • Unified region M = spark.memory.fraction × (heap − 300 MiB), 0.6 by default — the shared pool for the two consumers that do the work:
    • Execution — shuffle / sort / aggregation / join buffers (transient).
    • Storage — cached/persisted blocks + broadcast data (long-lived).
  • Storage floor R = spark.memory.storageFraction × M, 0.5 by default — the part of cache execution cannot evict.
The one rule to remember Execution and Storage share region M and borrow from each other — but the borrowing is asymmetric. Execution can evict Storage down to R; Storage can never evict Execution. A running task must make progress; a cached block can always be recomputed. Active work beats cached convenience — and that single fact explains most spill/OOM puzzles.

Why it matters

  • Spill and OOM are the two most common "slow / dead job" tickets — both are memory-region problems you can only diagnose if you know which region (execution vs storage) was under pressure.
  • Spill is silent tax. When execution memory is exhausted, Spark spills to local disk and the job still finishes — just much slower (disk I/O + extra GC). Many "slow Spark jobs" are really un-noticed spill.
  • Executor OOM kills the task when even spilling can't keep a partition within the container limit — usually a skewed partition, cache pinned in R, or oversized Python workers.
  • PySpark adds a region the heap diagram doesn't show. Python UDFs run in worker processes outside the JVM heap and count against overhead, not M — the #1 hidden PySpark OOM, where bumping spark.executor.memory does nothing.

How it works — deep dive

1 · Executor memory overview — the four regions

analogy The executor heap is a shared commercial kitchen: a walled-off back office (reserved 300 MiB), your private pantry (user memory), and one big shared counter (region M). On that counter the active cooking workspace (execution) and the shelf of prepped ingredients (storage/cache) share the surface — the cook can shove ingredients aside, but the shelf can never push the cook off.

  • Reserved = 300 MiB (fixed), carved off first.
  • Usable heap = spark.executor.memory − 300 MiB (default spark.executor.memory = 1g).
  • Unified M = 0.6 × usable heap.
  • User memory = the remaining ~40% — user objects, UDF state, metadata, and a safety cushion.

use case On a default 1 GB executor, M ≈ 0.6 × (1024 − 300) ≈ 434 MiB is all that's shared between every shuffle buffer and every cached block — which is why a heavy shuffle and a big cache in the same stage fight for the same few hundred MiB.

2 · Execution vs storage — two consumers, two lifecycles

  • Execution memory — buffers for shuffle, sort, aggregation, join. Transient: grabbed by a task, freed when it finishes. Short → spill.
  • Storage memory — cached/persisted blocks + broadcast. Long-lived: stays until evicted or unpersist()-ed. Over-allocated → starves execution.

analogy Execution is scratch paper you scribble on and toss mid-task; storage is a reference book you keep on the desk all session.

3 · The unified model — borrow & the evict asymmetry

  • Borrow freely: "When no execution memory is used, storage can acquire all the available memory and vice versa." Nothing cached → a shuffle can use all of M; nothing shuffling → cache can fill all of M.
  • Execution may evict Storage — down to R. A task needing execution memory evicts cached blocks above R. Blocks inside R (= 0.5 × M) are immune.
  • Storage may NOT evict Execution. If cache wants more but execution holds it, cache just doesn't get it — the new block spills to disk (per its storage level) or isn't cached.
Why the asymmetry exists A running task must make progress to finish the stage — evicting its buffers would fail or deadlock it. A cached block is only an optimization: if evicted, Spark recomputes it from lineage. So the rule is simply active work beats cached convenience.

use case You cache() a 400 MiB DataFrame then aggregate it; the aggregation evicts cache blocks above R, so the cache ends up smaller than expected and the next read partly recomputes — the asymmetry biting a real job.

4 · Data spilling — the warning sign before death

  • What it is: when execution memory is exhausted during a shuffle/sort/agg/join, Spark spills the in-memory structures to local disk. Docs: "When data does not fit in memory Spark will spill these tables to disk, incurring the additional overhead of disk I/O and increased garbage collection."
  • What it means: the job still finishes — spill is graceful degradation, not a crash — but pays disk I/O + more GC, so a spilling stage can be several times slower.
  • How to see it: Spark UI shows Spill (Memory) and Spill (Disk) per task/stage. Any non-zero spill is a flag.

analogy Spill is overflow parking: the lot (execution memory) is full, so cars (data) park in the muddy field down the road (disk). Everyone still parks — it's just slower to walk back.

5 · Why & how executor OOM — when spilling isn't enough

OOM (java.lang.OutOfMemoryError) happens when even spilling can't keep a partition within the container limit. Common causes:

  • A single huge/skewed partition — one key holds most rows; one task's buffers exceed memory faster than spill drains them (fixed in Lesson 08).
  • Too many cached blocks pinned in R — R is eviction-immune, so a heavily-cached executor leaves little room for execution.
  • Oversized Python workers — grow past the overhead allotment; the cluster manager kills the container (sub-topic 7).
  • Too-small spark.executor.memory / overhead for the partition size.

use case A groupBy where 90% of rows share one value → one task builds a hash aggregation for 90% of the data → spills, spills, then exceeds the container and the executor dies. Fix the skew, don't just add RAM.

6 · Off-heap memory — outside the JVM, no GC

  • What it is: memory outside the JVM heap (Tungsten/Unsafe), so it's not garbage-collected — no GC pauses for that data.
  • Configs: spark.memory.offHeap.enabled = false (default), spark.memory.offHeap.size = 0 (default; must be > 0 when enabled). When on, the docs advise you "shrink your JVM heap size accordingly."
  • Trade-off: less GC pressure on large heaps, but you must hand-size it; mainly a win when GC is the bottleneck (Lesson 10).

7 · PySpark (Python-worker) memory — the region outside the bar

  • Where Python runs: Python UDFs, mapInPandas, Pandas/Arrow UDFs, RDD map in Python run in separate Python worker processes outside the JVM heap — they do not use region M.
  • The default: spark.executor.pyspark.memory is Not set. When unset, Spark does not limit the Python workers and they count against the container's overhead/limit — a hidden OOM cause.
  • Overhead covers it: spark.executor.memoryOverhead = executor.memory × 0.10 (factor; 0.40 for K8s non-JVM), min 384m. Overhead = off-heap + shuffle/network buffers + (when pyspark.memory unset) the Python workers.

use case A Pandas UDF that materializes a big batch per group blows the container while the JVM heap looks fine. The cure is more overhead (or a smaller batch / native function), not a bigger spark.executor.memory.

The code you'll write (and how to verify it)

Track rule Every technique is paired with how to prove it worked — the .explain() plan node or the Spark-UI signal. Apply, then verify. Never assume.

Inspect the unified-memory configs

python# The four numbers that define region M. Read them before tuning anything.
print(spark.conf.get("spark.executor.memory"))          # default "1g" — the JVM heap
print(spark.conf.get("spark.memory.fraction"))          # default "0.6" — M = 0.6 * (heap - 300 MiB)
print(spark.conf.get("spark.memory.storageFraction"))   # default "0.5" — R = 0.5 * M (eviction-immune cache)

# VERIFY (do the arithmetic): on a 1g executor,
#   usable = 1024 - 300 = 724 MiB ; M = 0.6 * 724 ≈ 434 MiB ; R = 0.5 * 434 ≈ 217 MiB
# That ~434 MiB is shared by EVERY shuffle buffer AND every cached block on the executor.

See (and cure) a spill

pythonfrom pyspark.sql import functions as F

# CREATE the condition: a wide aggregation over many groups stresses EXECUTION memory.
big = spark.range(0, 200_000_000).withColumn("k", (F.rand() * 50_000_000).cast("long"))
agg = big.groupBy("k").count()

# VERIFY the spilling operator is in the plan:
agg.explain(mode="formatted")
#   look for: HashAggregate(keys=[k], ...)  +  Exchange hashpartitioning(k, 200)
#   HashAggregate is the execution-memory consumer that spills when M is exhausted.

# Run the job, then READ THE UI:
agg.write.format("noop").mode("overwrite").save()
# Spark UI → Stages → aggregation stage → columns "Spill (Memory)" / "Spill (Disk)".
# Non-zero spill = execution memory ran out. Cure: more, smaller partitions:
spark.conf.set("spark.sql.shuffle.partitions", 800)   # smaller slices → less per-task memory
# Re-run and confirm the spill columns drop toward zero. (Reset the conf afterwards.)

Watch execution evict storage (the asymmetry)

pythonfrom pyspark import StorageLevel

# Cache a DataFrame, then run heavy EXECUTION work on it in the same stage.
cached = big.persist(StorageLevel.MEMORY_AND_DISK)   # DataFrame default; spills cache to disk
cached.count()                                        # materialize (cache is lazy until an action)

heavy = cached.groupBy("k").count()                   # execution now competes with the cache
heavy.write.format("noop").mode("overwrite").save()

# VERIFY in Spark UI → Storage tab: the cached DataFrame may show < 100% "Fraction Cached"
# — execution EVICTED cache blocks above R to make room. Storage could not evict execution.
cached.unpersist()                                    # always release cache when done (frees R)

Enable off-heap memory (when GC is the bottleneck)

python# Off-heap lives OUTSIDE the JVM heap → no GC for that data. Size it explicitly, and
# shrink the heap so you don't double-count RAM (per the tuning docs).
spark.conf.set("spark.memory.offHeap.enabled", "true")
spark.conf.set("spark.memory.offHeap.size", str(2 * 1024 * 1024 * 1024))  # 2 GiB in bytes (>0)

# VERIFY: Spark UI → Executors tab → "Off Heap Memory Used / Total" becomes non-zero, and
# Stages "GC Time" drops for the same workload. Set these at CLUSTER creation in production.

Budget Python-worker memory (the hidden OOM)

python# Python UDFs / Pandas UDFs run OUTSIDE the JVM heap, against the container's OVERHEAD.
# Default: spark.executor.pyspark.memory is NOT set → Spark does not limit Python workers.
print(spark.conf.get("spark.executor.pyspark.memory", "<not set>"))   # → "<not set>" by default

# For a Python-heavy job that OOMs the CONTAINER (not the heap), raise OVERHEAD at cluster
# creation (not runtime-settable per query):
#   spark.executor.memoryOverheadFactor = 0.10   (default; covers off-heap + shuffle + Python)
#   spark.executor.memoryOverhead       = "2g"   (an explicit floor for Python-heavy work)
# VERIFY: container OOM kills (Executors tab "Failed Tasks" / logs) stop after raising overhead
# — whereas bumping spark.executor.memory alone would NOT have helped.

Diagnose an executor OOM, cause by cause

Executor OOM triage

When even spilling can't keep a partition in the container, the executor dies. Which cause is it?

01 A single huge / skewed partition
One key holds most of the rows, so one task's buffers exceed memory faster than spill can drain them. Confirm: Spark UI → Stages → the task-time distribution shows max ≫ median, and that one task has huge Shuffle Read + Spill before it dies. Fix: AQE skew join, then salting (Lesson 08) — split the hot key across many partitions. Adding heap only delays the OOM.
02 Too many cached blocks pinned in R
R = 0.5 × M is immune to eviction, so a heavily-cached executor leaves little room for execution. Confirm: Spark UI → Storage tab shows large cached DataFrames; the OOM stage is a shuffle/agg that needs execution memory. Fix: cache less, use a disk-spilling / serialized level, and unpersist() early (Lesson 06) — not more memory.
03 Oversized Python workers (overhead, not heap)
Python UDFs / Pandas UDFs run outside the JVM heap and spend the container's overhead budget. A big per-group batch blows the container while the heap is idle. Confirm: Executors tab shows the JVM heap healthy but the container is killed (OOM in logs). Fix: raise spark.executor.memoryOverhead / set spark.executor.pyspark.memory, or shrink the batch — not spark.executor.memory.
04 Too-small executor.memory / overhead
The heap or overhead is simply undersized for the partition size. Confirm: no skew (task times even), little cache, no Python — yet every task spills heavily and some die. Fix: add partitions first (smaller slices fit in M); only then raise spark.executor.memory or overhead. This is the only case where "more memory" is the right first move.
The triage order: which region failed? Heap (execution/storage) → fix skew / cache / partitions. Container overhead (Python/off-heap) → raise overhead. The fix is wrong if you don't first name the region.

Uses, edge cases & limitations

Uses

  • Diagnose a slow stage → check Spill (Memory/Disk); spill = execution short. Add partitions / fix skew before adding RAM.
  • Diagnose an executor OOM → decide which region failed: heap (execution/storage) vs container overhead (Python/off-heap). The fix differs completely.
  • Right-size caching → cache lives in M and only R is safe; don't cache so much that execution starves (Lesson 06).
  • Tame GC on big heaps → enable off-heap (Lesson 10) so cached data leaves the heap.

Edge cases

  • Skew AQE can't fully fix — a key so dominant that even split sub-partitions exceed memory; fall back to salting (Lesson 08) or a different key.
  • Cache pinned in R — heavily-cached executors look "fine on heap" but OOM under a shuffle; the cure is less cache.
  • Python-worker blow-up — a huge Pandas UDF batch OOMs the container while the heap is idle; raise overhead.
  • Broadcast in storage — a broadcast relation consumes storage memory on every executor; a near-threshold broadcast (Lesson 02) pressures M cluster-wide.

Limitations

  • The unified model manages only the JVM heap regions (M + user) — not Python workers, off-heap (unless enabled), or the container limit. Those are separate budgets.
  • spark.memory.fraction / storageFraction are cluster/JVM-level; changing them mid-session does not resize a running executor — set them at cluster creation.
  • Spill makes a job slow, not safe forever — a large enough skewed partition still OOMs no matter how much it spills.
  • OSS vs Databricks: the unified-memory math (0.6/0.5/300 MiB/overhead 0.10) is identical; Databricks may layer its own overhead/Photon accounting on top — verify sizing in the Azure Databricks compute docs.

Common mistakes & gotchas

  • "Add more executor.memory" as the reflex fix. If a Python worker or off-heap blows the container, more heap does nothing — raise overhead. If it's skew spill, more heap just delays the OOM.
  • Thinking the split is a static 60/40. It's unified — either can use all of M. The only fixed guarantee is R, the eviction-immune cache floor.
  • Believing storage can evict execution. It can't, ever. Cache yields to active work — this one fact explains most spill/OOM puzzles.
  • Ignoring non-zero spill because "the job finished." Spill is the early warning; a slightly bigger input tomorrow turns it into an OOM.
  • Caching everything. Cached blocks pin storage in R and pressure GC; cache only reused DataFrames, with the right level, and unpersist() when done (Lesson 06).
  • Forgetting Python lives outside the heap. The unified-memory bar does not include Python workers — they're an overhead-budget item, the #1 hidden PySpark OOM.
  • Changing spark.memory.fraction at runtime and expecting it to apply — it's a cluster-launch setting.
