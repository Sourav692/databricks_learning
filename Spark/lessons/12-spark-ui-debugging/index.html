Spark UI Debugging

PLAN NODES · STAGE METRICS · ROOT CAUSE

A slow Spark job is not a mystery. Read the physical plan to see the shape of the work, then use the Spark UI to measure where that work became expensive.

Spark 3.2+DBR LTSverified Jun 2026

01 · What it is

Plan first, metrics second

df.explain(mode="formatted") tells you the operators Spark planned. The Spark UI tells you how those operators behaved at runtime: shuffle, spill, skew, GC, retries, cache, and executor health.

The one rule to rememberExplain shows the shape; Spark UI shows the cost. Use both before changing code, memory, or cluster size.
Plan clue

Exchange, join strategy, scan filters, cache scans, and AQE nodes.

Runtime clue

Task spread, shuffle read/write, spill, GC time, failed tasks, executor health.

Healthy pattern

The plan changed and the matching UI metric improved after your fix.

02 · How it works

The diagnostic loop

analogyThe plan is a blueprint; the UI is the construction bill. A big room on the blueprint is not bad by itself, but the bill tells you which room consumed all the time and material.

query.explain(mode="formatted")
# Look for: Exchange, BroadcastHashJoin, SortMergeJoin, HashAggregate,
# InMemoryTableScan, AdaptiveSparkPlan, AQEShuffleRead.

05 · Uses and gotchas

Use evidence, not guesses

  • Use it for slow joins, skew, spill, GC, failed tasks, and cache validation.
  • Edge case: AQE final plans differ from initial plans; inspect after an action.
  • Limitation: UI history can expire, and logs may still be needed for executor failures.
  • Gotcha: benchmarking with collect() adds driver pressure. Use a real write or noop sink when you want full-plan runtime.
