Query Plans

SQL TO LOGICAL TO PHYSICAL TO EXECUTION

Spark does not run your DataFrame code directly. It parses it, resolves it, optimizes it, chooses a physical plan, generates code where possible, and then runs distributed tasks.

01 · What it is

Read the plan as an assembly line

A Spark query plan is the path from what you wrote to what executors run. The order matters: Spark first understands the query, then improves the logical plan, then chooses a physical strategy, then executes it.

The one rule to rememberThe top of explain(True) shows how Spark understood and optimized your query. The bottom, Physical Plan, shows what Spark will actually run.

06 · Pushdown note

Why a filter can appear twice

If Spark pushes a filter into Parquet/Delta, you may still see a Filter above the scan. That is not necessarily duplicate waste. Spark keeps a physical filter as a correctness check because not every source can fully apply every pushed predicate.

customers.filter(col("age").cast("int") > 50).explain(True)
