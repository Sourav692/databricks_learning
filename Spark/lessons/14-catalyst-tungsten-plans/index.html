01 · What it is

Planning and execution are different layers

Catalyst turns DataFrame/SQL code into optimized logical and physical plans. Tungsten is the execution/memory work that makes the chosen plan run efficiently with compact data and generated JVM code.

The one rule to rememberCatalyst decides what should run; Tungsten helps it run efficiently. Debug by reading Catalyst's physical plan, then verify runtime cost in the Spark UI.

05 · Uses and gotchas

Read the nodes that change decisions

  • Use it to explain why DataFrame/SQL beats RDD/UDF-heavy code for optimizer-visible work.
  • Edge case: UDF internals can block pushdown and pruning because Catalyst cannot inspect them.
  • Limitation: you do not need every node; focus on Exchange, joins, scans, aggregates, cache, and AQE.
  • Gotcha: Catalyst plans the query; executors run tasks. Do not say Catalyst "runs" the job.
