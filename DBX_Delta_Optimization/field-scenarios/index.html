Field scenarios.

These are the recurring layout and maintenance calls an FDE makes on real customer Delta tables — a table drowning in small files, a partition explosion, a filter that scans everything, a maintenance job that never keeps up. Each row is a decision, not a recipe: find where the read (or write) is actually paying, then reach for the smallest layout lever that fits.

00 · How to read this guide

Diagnose the layout, then pick the smallest lever

The FDE reflex Before changing a table's layout, find where the cost actually is. Ask which one it is: too many small files (slow listing, metadata bloat — DESCRIBE DETAIL shows a huge numFiles at a tiny average size), no data skipping (queries scan far more files than they return — the predicate isn't on a clustered/partition column), write-path fragmentation (streaming / MERGE-heavy churn adding tiny files faster than maintenance clears them), or maintenance not keeping up (OPTIMIZE / VACUUM run by hand, or not at all). Then apply the smallest lever — and for any new UC managed table the default answer is liquid clustering + predictive optimization, not partitioning or hand-tuned Z-order. Confirm every change with DESCRIBE DETAIL, DESCRIBE HISTORY, and the query profile; never assume.

The four lever families below map to the four sections. Most day-to-day calls are really about choosing which family the symptom belongs to — the wrong lever (over-partitioning a small table, hand-scheduling OPTIMIZE forever, Z-ordering a liquid-clustered table) wastes compute and leaves the real cost in place.

01 · File sizing & the small-file problem

When the cost is thousands of tiny files

The most common Delta performance call: reads are slow because the table is fragmented. The lever is compaction and right-sizing — but which one depends on whether the fragmentation is being created faster than you can fix it.

ScenarioWhat's really happeningThe call (fix)
A streaming events table has 40,000 files averaging 2 MB by noon; morning dashboards crawl. Every micro-batch writes its own tiny Parquet files; fixed per-file overhead (list, open, footer, task) now dominates every scan. Run OPTIMIZE to bin-pack into fewer right-sized files (idempotent, safe on a live table via snapshot isolation) — schedule daily, or on a UC managed table let predictive optimization run it (Lesson 04, Lesson 11).
Someone added df.coalesce(1) / repartition(200) before a write "to control file count," and it's brittle. Hand-guessing partition count fights the engine; with optimized writes enabled it actively hurts, and it breaks when data volume changes. Remove the manual coalesce/repartition before the write and enable optimized writes (shuffles to ~128 MB files at write time) (Lesson 05).
A MERGE-heavy CDC table keeps fragmenting even though a nightly OPTIMIZE runs. MERGE/UPDATE/DELETE create small files after each commit; a once-a-day compaction can't keep pace with intra-day churn. Rely on auto compaction (always on for MERGE/UPDATE/DELETE) to combine files within partitions post-commit, and keep OPTIMIZE for the deeper consolidation (Lesson 06).
A 20 TB table still has small files even though autotuning is "on." Autotuning grows the target size with table size (up to 1 GB > 10 TB) but does NOT re-optimize existing files — old small files persist. Set an explicit delta.targetFileSize and run OPTIMIZE to rewrite; autotuning only governs future writes (Lesson 07).
You set delta.autoOptimize.autoCompact = true but files still aren't the size you expected. true uses a fixed 128 MB target with no dynamic sizing; auto autotunes the target to the table. Use auto (recommended) so the target adapts to table size, rather than the fixed-128 MB true/legacy value (Lesson 06, Lesson 07).
A batch job writes one giant file per partition and downstream parallelism suffers. Too few, too-large files under-parallelize reads just as many tiny files over-cost them — the target size is the balance. Let optimized writes / autotuning pick the target (128 MB–1 GB by table size), or cap rows with spark.sql.files.maxRecordsPerFile only for a specific need (Lesson 05, Lesson 07).
An external (non-UC-managed) table has no maintenance and grows small files unchecked. Predictive optimization only runs on UC managed tables, so external tables get no automatic OPTIMIZE/VACUUM. Schedule OPTIMIZE (and VACUUM) as a job for the external table, or migrate it to a UC managed table to get predictive optimization (Lesson 04, Lesson 11).

02 · Clustering & data skipping

When the cost is scanning files you didn't need

Right-sized files still lose if the engine can't skip them. These calls are about colocating rows on the columns queries filter — and choosing liquid clustering over the legacy partitioning / Z-order it replaces.

ScenarioWhat's really happeningThe call (fix)
A table is partitioned on a high-cardinality column (customer_id / timestamp) and it's a small-files disaster. One directory per value on a high-cardinality column creates a partition explosion — thousands of tiny partitions, each with tiny files. Migrate to liquid clustering (CLUSTER BY the query keys — no directory explosion, keys redefinable without rewrite); reserve partitioning for low-cardinality columns on tables > 1 TB (Lesson 02, Lesson 08).
Someone wants to partition a 200 GB table "for performance." Databricks says do NOT partition tables under 1 TB; partitions below ~1 GB each just fragment the table. Don't partition — leave it unpartitioned (ingestion-time clustering gives date-like benefits free) or use liquid clustering on the real filter keys (Lesson 02, Lesson 08).
A query filters on a column that isn't the partition column and does a full scan of a 5 TB table. Data skipping only works when the predicate column has stats and rows are colocated; an unclustered filter column can't skip files. Cluster (liquid) or Z-order the table on the common filter columns so per-file min/max stats prune files; confirm files-pruned in the query profile (Lesson 03, Lesson 08).
A filter column sits past column 32 and gets no data skipping. Stats are collected on the first 32 columns by default (UC external); a column beyond that has no min/max to skip on. Reorder so predicate columns lead, or set delta.dataSkippingStatsColumns (DBR 13.3+) then ANALYZE TABLE t COMPUTE DELTA STATISTICS; UC managed tables use predictive optimization for stats with no 32-col limit (Lesson 03).
Someone runs OPTIMIZE … ZORDER BY on a liquid-clustering table. Z-order is incompatible with liquid clustering — LC already colocates by its keys; mixing them is unsupported. Drop the ZORDER; on an LC table plain OPTIMIZE groups by the clustering keys automatically (Lesson 03, Lesson 08).
A team defines 8 clustering keys hoping to cover every query. Liquid clustering allows up to 4 keys; too many keys weaken single-column filters, especially on tables < 10 TB. Pick ≤ 4 keys from the highest-value predicates (or use CLUSTER BY AUTO to let PO choose); more keys is not more skipping (Lesson 08).
Clustering keys were changed, but existing data still filters slowly. Changing keys does not rewrite existing data (the LC benefit); only new writes and incremental OPTIMIZE re-lay-out data. Run OPTIMIZE t FULL (DBR 16.0/16.4 LTS+) once after changing keys to recluster all existing data; plain incremental OPTIMIZE won't (Lesson 08, Lesson 04).
A legacy job Z-orders on the partition column and it does nothing. You cannot Z-order a partition column, and Z-order only colocates within a partition — it can't cross partition boundaries. Z-order on a non-partition high-cardinality predicate column, or migrate off partitioning to liquid clustering entirely (Lesson 03, Lesson 08).

03 · Write path, deletion vectors & maintenance

When the cost is on the write side — or in never cleaning up

Some scenarios aren't about read layout at all: they're about how modifications create files, and whether maintenance (compaction, VACUUM) and history retention are set up correctly.

ScenarioWhat's really happeningThe call (fix)
A GDPR delete of a few rows rewrites entire multi-GB Parquet files and takes forever. Default copy-on-write rewrites the whole file that holds a deleted row — expensive for scattered row-level deletes. Enable deletion vectors (delta.enableDeletionVectors = true) so DELETE/UPDATE/MERGE soft-delete rows (merge-on-read) without rewriting files; Photon uses them for predictive I/O (Lesson 09).
Deletion vectors are on, but the deleted rows must be physically purged for compliance. DVs mark rows as removed but leave the data in the Parquet files until something rewrites them. Run REORG TABLE t APPLY (PURGE) to physically apply soft-deletes, then VACUUM after the retention window to remove the old files (Lesson 09, Lesson 10).
Storage cost keeps climbing on a high-churn table even after deletes. Old, unreferenced data files aren't removed until VACUUM runs past the retention threshold; the log still points readers away from them but they occupy storage. Schedule VACUUM (default 7-day retention); on big tables use VACUUM t LITE (DBR 16.4 LTS+) to skip full file listing, or let predictive optimization VACUUM UC managed tables (Lesson 10, Lesson 11).
A too-aggressive VACUUM RETAIN 1 HOURS is proposed to save storage fast. RETAIN under 7 days is blocked by a safety check because it can delete files still needed by long-running jobs or recent time travel. Keep the default (or a deliberate longer) retention; only lower it with a clear reason, never by disabling retentionDurationCheck casually (Lesson 10).
An analyst needs to query the table "as of last month" but the data is gone. Time travel needs BOTH the log and the underlying data files; VACUUM's 7-day data-file retention caps how far back you can go, regardless of log retention. Raise both delta.deletedFileRetentionDuration and delta.logRetentionDuration BEFORE you need the window (e.g. 30 days); time travel is not a long-term backup (Lesson 10).
An accidental bad MERGE corrupted a production table. Every modification is a new version; the prior good state is still queryable if its files haven't been vacuumed. Inspect DESCRIBE HISTORY, then RESTORE TABLE t TO VERSION AS OF <v> (needs MODIFY; downstream streams may reprocess) (Lesson 10).
Enabling deletion vectors broke an older external reader that can't open the table. DVs upgrade the table protocol (writer v7 / reader v3); clients without DV support can no longer read it. Confirm all readers support DVs before enabling; if you must roll back, ALTER TABLE t DROP FEATURE deletionVectors (DBR 14.1+) to downgrade (Lesson 09).
A big VACUUM is slow and the driver is pegged. VACUUM phase 1 lists files in parallel on workers, but phase 2 deletes from the single driver node — a large delete bottlenecks there. Size the driver up for large VACUUM deletes, or use VACUUM t LITE (DBR 16.4 LTS+) to avoid listing every file (Lesson 10).

04 · Platform & managed layout

When the right answer is to let the platform decide

The most modern set of calls: on Unity Catalog managed tables, layout and maintenance are increasingly the platform's job. These scenarios are about knowing when to stop hand-tuning.

ScenarioWhat's really happeningThe call (fix)
You're designing a brand-new UC managed table and reaching for PARTITIONED BY out of habit. Partitioning is legacy layout; Databricks recommends liquid clustering for all new Delta (and managed Iceberg) tables. Use CLUSTER BY (or CLUSTER BY AUTO) instead of partitioning, and let predictive optimization run OPTIMIZE/VACUUM/ANALYZE (Lesson 08, Lesson 11).
You don't know which clustering keys a new table will actually be queried on. Guessing keys up front often mis-targets real query patterns, and manual key changes need FULL reclustering. Use CLUSTER BY AUTO (DBR 15.4 LTS+, UC managed, requires predictive optimization) — it analyzes query history and picks/changes keys cost-aware (Lesson 08, Lesson 11).
A team hand-schedules OPTIMIZE + VACUUM jobs across hundreds of tables and it's a maintenance burden. Manual maintenance doesn't scale and often runs on the wrong cadence per table. Enable predictive optimization (on by default for accounts since Nov 11 2024; PREMIUM plan, UC managed) — it runs OPTIMIZE, VACUUM, ANALYZE automatically on serverless (Lesson 11).
Predictive optimization is enabled but a Z-ordered table isn't being reclustered. PO's OPTIMIZE does NOT run ZORDER (it ignores Z-ordered files); it drives liquid-clustering key selection instead. Migrate the table to liquid clustering so PO can manage its layout; Z-order is the legacy path PO won't maintain (Lesson 11, Lesson 08).
An unpartitioned append-only table is queried mostly by recent date, and nobody set up any layout. Unpartitioned tables get ingestion-time clustering (DBR 11.3 LTS+) automatically — data lands roughly in ingestion order. Often do nothing: ingestion-time clustering gives date-partition-like skipping for free; add liquid clustering only if queries filter on something other than recency (Lesson 02, Lesson 08).
Predictive optimization was expected to run but the table is skipped. PO runs only on UC MANAGED tables on PREMIUM/supported regions — it does not run on EXTERNAL or OpenSharing recipient tables. Verify with DESCRIBE TABLE EXTENDED (Predictive Optimization field); migrate external tables to managed, or schedule maintenance manually (Lesson 11).
You want longer time travel but you're about to enable predictive optimization (which VACUUMs). Once PO runs VACUUM at the default 7-day retention, older versions become unqueryable — the data files are gone. Raise delta.deletedFileRetentionDuration (and log retention) BEFORE enabling PO if you need a longer time-travel window (Lesson 11, Lesson 10).
A migration to liquid clustering is blocked by old Delta clients that can't read the table. Liquid clustering uses writer v7 / reader v3 (DVs, row tracking, checkpoint V2) — old clients can't read it and you can't downgrade the protocol. Upgrade the reading clients (or DBR) first; confirm compatibility before enabling LC, since the protocol bump is one-way (Lesson 08).
