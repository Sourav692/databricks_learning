Lakebase Postgres

A fully managed, Postgres-compatible OLTP database built into Databricks — low-latency transactions and feature serving that sit right next to your lakehouse.

GA · AWS & Azure Separated storage & compute Postgres 16 / 17 Verified June 2026 docs

In one line

Lakebase is a serverless, Postgres-compatible OLTP database where storage and compute scale independently, with copy-on-write branching and two-way sync to Unity Catalog Delta tables.

Why it matters

The lakehouse is great at analytics but not at millisecond, row-level reads and writes. Lakebase gives you a real transactional Postgres for app backends, online feature stores, and agent state — without standing up and operating a separate database, and without ETL gymnastics to move data between your operational store and the lakehouse.

Core concepts

Managed Postgres OLTP

Standard Postgres wire protocol and SQL (PG 16/17) — connect with psql, psycopg, DBeaver, or any Postgres driver; OAuth token is the password over SSL.

Separated storage & compute

Compute (endpoints, sized in compute units) autoscales 0.5–112 CU and scales to zero; durable storage lives independently, so you pay for compute only while it serves.

Branching

Copy-on-write branches clone a database instantly for dev, testing, or CI/CD — including point-in-time branches for recovery — without copying the data.

Synced tables (lakehouse → Lakebase)

Sync a Unity Catalog Delta table into Postgres for low-latency serving, in snapshot, triggered, or continuous mode via a managed Lakeflow pipeline.

Reverse sync (Lakebase → lakehouse)

Capture Postgres changes back into Unity Catalog Delta tables (CDC) so operational writes land in the lakehouse for analytics — Public Preview.

App & serving use cases

Backs Databricks Apps, online feature stores for ML model serving, and AI agent state — transactional reads/writes co-located with governed lakehouse data.

Minimal example

SQL · standard Postgres against Lakebase-- ordinary Postgres DDL/DML — no Databricks dialect
CREATE TABLE orders (
  id        bigint PRIMARY KEY,
  customer  text NOT NULL,
  amount    numeric(12,2),
  created   timestamptz DEFAULT now()
);

INSERT INTO orders (id, customer, amount)
VALUES (1, 'acme', 42.00);

-- millisecond point lookup — the OLTP read pattern Lakebase is built for
SELECT * FROM orders WHERE id = 1;

Gotchas & limits

  • It is OLTP, not analytics — for big scans and joins, keep data in the lakehouse and serve only the hot subset via synced tables.
  • OAuth credentials expire after ~1 hour; long-lived connections need token refresh, and clients should retry the ~100ms wake from scale-to-zero.
  • Synced tables in Postgres are read-only — only queries, indexes, and DROP are allowed; writing to them breaks the sync pipeline.
  • Reverse sync (Lakebase changes → UC Delta) is Public Preview; per-project limits apply (e.g. 10 unarchived branches, 8 TB logical data per branch).
