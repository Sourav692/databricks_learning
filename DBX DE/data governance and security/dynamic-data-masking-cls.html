Dynamic Data Masking Column-Level Security

Hide sensitive column values from users who aren't authorized — at query time, based on who is asking — without ever copying or altering the underlying data.

The core idea

Dynamic data masking hides the values in a sensitive column from people who shouldn't see them, while still letting them query the table. The row stays; only the cell content changes (a real email becomes ***).

"Dynamic" means masking happens at query time, decided by who runs the query. The data on disk is never changed. Two people can run the same SELECT * and get different results.

Think of it as frosted glass over a column: authorized viewers get a clear pane, everyone else sees the blur — but there's only ever one piece of glass (one table).

The three moving parts

01

Groups

Created under Settings → Identity and Access. Membership decides who sees real data (e.g. developers).

02

Masking function

A reusable SQL UDF: "if caller is in the group, return the value; else return a masked value."

03

Mask attachment

Bind the function to one column with ALTER TABLE … SET MASK.

The key building block is the built-in function is_account_group_member('group'), which returns true/false for the current user. Note: it is a built-in SQL function — used the same way inside dynamic views or column masks.

The code — step by step

Step 1 · Create the masking function

-- Prereqs: Unity Catalog workspace; SQL warehouse OR standard (DBR 12.2 LTS+)
-- OR dedicated (DBR 15.4 LTS+); a group named 'developers'.
CREATE OR REPLACE FUNCTION databricks_orange.silver.dynamic_mask(p_email STRING)
RETURNS STRING
RETURN
  CASE
    WHEN is_account_group_member('developers') THEN p_email
    ELSE '***'
  END;   -- CASE must be closed with END (easy to forget!)

Step 2 · Attach the mask to the column

ALTER TABLE databricks_orange.silver.customers
  ALTER COLUMN email
  SET MASK databricks_orange.silver.dynamic_mask;

Step 3 · Query as normal — masking is automatic

SELECT name, email FROM databricks_orange.silver.customers;
-- Member of 'developers'       -> alice@acme.com
-- NOT a member of 'developers' -> ***

Remove the mask later

ALTER TABLE databricks_orange.silver.customers
  ALTER COLUMN email DROP MASK;

Going deeper · conditional masking with USING COLUMNS

When the mask depends on another column, pass extra columns/literals with USING COLUMNS. The first parameter is always the masked column; everything else comes from USING COLUMNS.

CREATE FUNCTION databricks_orange.silver.mask_address_by_country(
  address STRING, country STRING, group_suffix STRING DEFAULT '_address_viewers')
RETURN IF(
  is_account_group_member(country || group_suffix),  -- e.g. 'US_address_viewers'
  address,
  'REDACTED');

ALTER TABLE databricks_orange.silver.customers
  ALTER COLUMN address
  SET MASK mask_address_by_country USING COLUMNS (country, '_address_viewers');

ApproachHowWhen to use
Dynamic viewsCREATE VIEW with CASE WHEN is_account_group_member()…; grant the view, not the tableOlder but valid. Good when you also reshape/join data
Column masks (used here)CREATE FUNCTION + ALTER TABLE … SET MASKPreferred today. Policy lives on the table; users query the real table name
ABAC (tag-based)Tag columns (e.g. pii), write one policy, apply across many tablesNewest, most scalable for large estates

Common mistakes & gotchas

⏳
Group changes aren't instant. Membership is cached — after adding yourself to a group, results may not change until a short refresh. Wait a minute and re-run.
🖥️
Compute matters. Masks enforce on a SQL warehouse, standard clusters (DBR 12.2 LTS+), or dedicated (DBR 15.4 LTS+). On dedicated DBR 15.3 or below you can't read masked tables. Filtering runs on serverless.
🔻
Drop order is strict. Always ALTER TABLE … DROP MASK before DROP FUNCTION, or the table becomes inaccessible.
♻️
CREATE OR REPLACE TABLE keeps the mask if a same-named column exists — convenient, but it can persist silently and break queries if it references a removed column.
🧬
First parameter type = column type. Mismatches cause errors on INSERT/MERGE/UPDATE.
🐍
Python logic needs a SQL wrapper. You can't attach a Python UDF directly — wrap it in a SQL UDF, or you'll get [ROUTINE_NOT_FOUND].
🔑
Definer vs invoker. Masks run with the definer's rights, but is_account_group_member() and session_user() evaluate as the invoker — which is exactly why per-user masking works.
