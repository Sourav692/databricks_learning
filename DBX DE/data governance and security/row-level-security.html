Row-Level Security (RLS)

Same table, different eyes — each user sees only the rows they're allowed to see. Not masking values inside a column; hiding entire records.

The idea in one sentence

RLS means: same table, but each user only sees the rows they're allowed to see.

You are not hiding values inside a column (that's masking) — you're restricting which whole rows come back. A manager for the East region queries the shared stores table and gets only East rows; the West manager runs the same query and gets only West rows.

Both are built from the same building block — a SQL function (UDF) registered in Unity Catalog. The difference is what the function returns.

How it works (the mechanism)

A row filter is a scalar SQL UDF (a reusable function you register in the catalog) that:

  • Takes one or more columns of the row as input,
  • Returns a BOOLEAN,
  • Is evaluated once per row, at query time.
The core rule: if the function returns FALSE or NULL, the row is filtered out. Only TRUE rows are returned.

To decide "is this user allowed to see this region?", you use a mapping table (an access-control list) that records which user belongs to which region. The filter function looks the user up via current_user() — a built-in that returns the identity of whoever is running the query.

Prerequisites & the "groundwork" order

  • Unity Catalog is required — this is a UC-only feature.
  • Compute: a SQL warehouse, or Databricks Runtime 12.2 LTS+ (standard access mode), or 15.4 LTS+ (dedicated access mode). Older runtimes "fail secure" — they return no data rather than unfiltered data.

Build it in this exact sequence:

  1. Create the mapping table and populate it.
  2. Create the filter function (UDF).
  3. Apply it with ALTER TABLE … SET ROW FILTER.

Step-by-step (runnable Spark SQL)

1 · Create the mapping table

CREATE TABLE databricks_orange.silver.arls_mapping (
  user_email STRING,
  region     STRING
);

2 · Populate it (don't forget VALUES)

INSERT INTO databricks_orange.silver.arls_mapping VALUES
  ('east_owner@example.com',  'East'),
  ('west_owner@example.com',  'West'),
  ('south_owner@example.com', 'South'),
  ('north_owner@example.com', 'North');

3 · Build the intuition first

Before writing the function, prove the lookup works. This returns true/false for whether you are mapped to a region:

SELECT EXISTS (
  SELECT 1 FROM databricks_orange.silver.arls_mapping
  WHERE user_email = current_user()
    AND lower(region) = lower('East')
);

EXISTS(…) collapses "did I find a matching row?" into a clean TRUE/FALSE — exactly what a row filter needs.

4 · Create the filter function

CREATE OR REPLACE FUNCTION databricks_orange.silver.arls_region_filter(p_region STRING)
RETURN EXISTS (
  SELECT 1 FROM databricks_orange.silver.arls_mapping
  WHERE user_email = current_user()
    AND lower(region) = lower(p_region)   -- case-insensitive match
);

The return type (BOOLEAN) is inferred from EXISTS, so you write RETURN EXISTS(…) — not RETURN SELECT …. A scalar function returns a single value, not a result set.

5 · Apply the filter to the table

ALTER TABLE databricks_orange.silver.stores
SET ROW FILTER databricks_orange.silver.arls_region_filter ON (region);

6 · Query the table

SELECT * FROM databricks_orange.silver.stores;
-- East manager -> only East stores; everything else is invisible

What happens at query time

For every row Databricks reads from stores:

  1. It passes that row's region into arls_region_filter(p_region).
  2. The function runs EXISTS(… user_email = current_user() AND region = p_region …) against the mapping table.
  3. User is mapped to that region → TRUE → row returned.
  4. User is not → FALSE → row silently dropped.

The filter runs the moment the row is fetched, so unauthorized rows never reach the result set. To change access, just edit the mapping table — no code changes, no re-applying the filter.

Common mistakes & gotchas

  • Only one row filter per table. Need multiple conditions? Put them all inside the one function.
  • Drop in the right order. Run ALTER TABLE … DROP ROW FILTER before DROP FUNCTION.
  • Case sensitivity bites. If the mapping stores 'East' but the column is 'east', a plain = won't match. Normalize both sides with lower().
  • NULL filters the row out, same as FALSE.
  • Type mismatches can silently become NULL (with ANSI mode off) → rows unexpectedly dropped. Keep parameter types matching column types.
  • REPLACE TABLE keeps the filter — by design.
Inaccessible-table trap: if you drop the function before dropping the row filter, the table becomes unreadable. Fix it with ALTER TABLE <table> DROP ROW FILTER; to clear the orphaned reference.
