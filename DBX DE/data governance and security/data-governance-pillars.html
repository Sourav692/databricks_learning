1 · Data Discoverability

How easily can people find and understand the data they need?

In one sentence: discoverability is locating the right data and understanding it — the rows, the metadata (owner, type, tags), recent updates, its meaning, and where it comes from / flows to. If people can't find or trust data, they re-build it or misuse it.

2 · Data Quality

Is the data correct, complete, and fresh?

In one sentence: bad data quietly produces wrong dashboards and wrong decisions, so catching nulls, zeros, duplicates, and out-of-range values is the engineer's job.

Three layers of quality in Databricks

① Constraints

In the table · write time

NOT NULL / CHECK on a Delta table reject or flag bad rows as they're written.

② Expectations

In pipelines · flow time

EXPECT rules in Lakeflow Declarative Pipelines (formerly DLT) warn, drop, or fail bad rows.

③ Lakehouse Monitoring

In Unity Catalog · after the fact

Continuously profiles a table and alerts when quality metrics drift over time.

Your DLT expectations and table constraints still apply — Lakehouse Monitoring is the Unity Catalog addition for ongoing observability.

Gotcha: a monitor is observability — it tells you quality slipped; it doesn't prevent the bad write. Use constraints/expectations to block bad data, monitoring to detect drift.

3 · Data Collaboration (Delta Sharing)

How do you share data across different metastores — even different orgs or platforms?

The problem: normal grants only reach workspaces on the same metastore. When the other side is on a different metastore, you need Delta Sharing — an open protocol for secure cross-boundary sharing.

Two flavors

  • Open Delta Sharing — share with any recipient, including non-Databricks tools and other clouds.
  • Databricks-to-Databricks (D2D) — secure sharing between two Databricks metastores (our focus).

Try it: the D2D sharing flow

Metastore A · Provider
Workspace A
catalog: main.silver.products
SHARE
Delta Sharing protocol
Metastore B · Recipient
Workspace B
reads a shared catalog
  1. Recipient sends their sharing identifier
    A string in the format <cloud>:<region>:<uuid> (e.g. aws:us-west-2:19a84bee-...) that uniquely names their metastore. They find it via Catalog Explorer or SELECT CURRENT_METASTORE().
  2. Provider creates a recipient
    Using that identifier: CREATE RECIPIENT acme USING ID 'aws:us-west-2:19a84bee-...';
  3. Provider creates a share & adds data
    CREATE SHARE sales_share; then ALTER SHARE sales_share ADD TABLE main.silver.products; A share is a read-only collection of tables/views/volumes/notebooks.
  4. Provider grants the recipient access
    GRANT SELECT ON SHARE sales_share TO RECIPIENT acme;
  5. Recipient queries the shared catalog
    On their side the share appears as a read-only catalog created from the share, which they query like any other catalog.
⚠️ Correction to the lecture: Delta Sharing does not create a "foreign catalog." In current Databricks terminology:
• Delta Sharing → recipient creates a regular catalog from the share (a "shared catalog").
• Foreign catalog → a Lakehouse Federation concept only — a catalog that queries an external database (MySQL, Postgres, Snowflake…) live, without moving data.
Both let you query data you didn't create, but they're different mechanisms.
Gotcha: external sharing must be enabled at the metastore level (off on free/trial workspaces). Shares are read-only — recipients can't modify your data.

4 · Data Access Control

The most used pillar: who can see and do what — and hiding what they shouldn't see.

In one sentence: admins may see everything, but a business analyst should see only what their job needs — with sensitive PII (email, phone, bank/card number, salary) hidden.

Two levels of control

LevelControlsHow
Object-level (coarse)Whole catalog / schema / table / viewGRANT SELECT ON TABLE … TO `analysts`;
Fine-grainedSpecific rows and columnsRow filters & column masks (SQL UDFs)

Try it: see the same table as different roles

A row filter limits which rows you see; a column mask hides sensitive values. Toggle the role:

customer_idnameregionemail PIIsalary PII

How it's implemented

-- Column mask: only admins see the real email
CREATE FUNCTION mask_email(email STRING)
RETURN CASE WHEN is_account_group_member('admins') THEN email
            ELSE '***@masked.com' END;
ALTER TABLE main.silver.customers ALTER COLUMN email SET MASK mask_email;

-- Row filter: non-admins only see their own region
CREATE FUNCTION region_filter(region STRING)
RETURN is_account_group_member('admins') OR region = current_user_region();
ALTER TABLE main.silver.customers SET ROW FILTER region_filter ON (region);

At scale, apply these centrally with ABAC policies (tag a column pii once; a policy masks it everywhere) instead of table-by-table.

Gotcha: grant access by groups, not individuals (update the group when people join/leave). And test as the restricted user — as owner/admin you bypass the masks you're trying to verify.
