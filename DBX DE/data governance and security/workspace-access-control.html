Workspace Access Control

Onboarding a new teammate with the right amount of access — they can log in and run SQL, but can't see your notebooks or change any data.

"Access control" in a Databricks workspace is really two separate layers stacked together. Getting onboarding right means understanding where one ends and the other begins.

Analogy: A workspace entitlement is the building keycard — it gets you through the front door and onto certain floors. Unity Catalog privileges are the keys to individual rooms — which filing cabinets you can open, and whether you may only read the files or also edit them. A keycard does not unlock every room.

The two layers (click to explore)

🪪 Layer 1 — Workspace access Entitlements
Can the person log in, and which features/surfaces can they use?
  • Workspace access → notebooks, jobs, models, pipelines (the feature, not specific files)
  • Databricks SQL access → dashboards, queries, SQL warehouses
  • Admin → full workspace administration (granted deliberately, never by default)

The built-in users group grants Workspace access + Databricks SQL access automatically to every user you add.

↓ then ↓
🔑 Layer 2 — Data access Unity Catalog privileges
Once in, what data can they see and act on?
  • BROWSE → discover objects + metadata only (no data)
  • USE CATALOG / USE SCHEMA → "enter" a catalog / schema (required boundary)
  • SELECT → read data  •  MODIFY → insert/update/delete
  • READ VOLUME → read files  •  EXECUTE → run functions / load models

Granted on the three-level namespace: catalog.schema.table.

Key rule: an entitlement is not a data permission. They are independent.

Why the new user sees no notebooks

When the new teammate logs in, the Workspace folder is empty. The entitlement is not the reason:

  • The Workspace access entitlement only says "you're allowed to use the notebook feature." It grants access to no specific notebook.
  • Notebooks and folders carry their own object-level permissions and are private to their owner by default.
  • So the user can use notebooks as a feature, but the list is empty because nothing was explicitly shared with them — your code lives in your user folder, which they can't access.

Entitlement ≠ object permission. That separation is what hides your workspace content with zero extra effort.

Adding a user (practical steps)

  1. Profile icon → Settings
  2. Open the Identity and access tab
  3. Next to Users, click Manage → Add user, enter the email, confirm

The user gets an invite email (check spam) or can log in directly with the email + one-time code. They're auto-added to the users group.

      How to grant / revoke

      In Catalog Explorer → select the catalog → Permissions: click Grant, pick the principal (user or group), choose a preset or individual privileges, and Confirm. To remove, select the grant and Revoke. Tip: Databricks recommends granting BROWSE to All account users so data is discoverable org-wide — revoke it and the catalog disappears from their view.

      Use groups in real teams

      Granting to one user is great for learning. In production, grant to a group instead:

      • Create a group (e.g., data_analysts)
      • Grant privileges/entitlements to the group once
      • Add each new hire to the group — they inherit everything; remove them to revoke

      Far easier to audit and maintain than per-user grants.

      Common mistakes & gotchas

      Confusing the two layers. Workspace access does not expose your notebooks or grant table access.
      Forgetting USE CATALOG / USE SCHEMA. SELECT alone fails without the USE privileges up the chain — which is exactly why presets exist.
      Over-granting on day one. Don't hand MODIFY, DDL, or ALL PRIVILEGES to new joiners. Start at BROWSE → Data Reader.
      Granting to All account users by accident. Great for BROWSE discovery, risky for anything more.
      Assuming admins are limited. Workspace admin entitlements can't be removed — be deliberate about who gets Admin.
      Expecting instant changes. Users often need to refresh Catalog Explorer for a grant/revoke to show.
