Attribute-Based Access Control

Tag your data once, write the rule once, and protection follows the data automatically — across today's tables and every future one. A complete, interactive walkthrough. Click anything that looks clickable.

Part 1 · Foundations

1 What is ABAC?

Plain: instead of a separate rule per table/column, you label data with tags, write one rule ("mask anything tagged PII"), and Databricks applies it automatically to current and future tagged objects.

Technical: ABAC is a dynamic access-control model in Unity Catalog where access decisions are made by evaluating policies against attributes (governed tags) attached to securables. A single policy attached at a catalog or schema applies automatically to every matching object in scope — it behaves like a future grant for security rules.

GA status: ABAC row-filter & column-mask policies, governed tags, and automated data classification are generally available. ABAC protects tables, materialized views, and streaming tables.

3 Governed tags & policy anatomy

Governed tags

Key–value pairs (pii = ssn), defined once at the account level with allowed values, applied to catalogs/schemas/tables/columns. Reusable across the whole estate (even multiple metastores). Tagging is a security boundary — control who has APPLY TAG.

Policy anatomy — CREATE POLICY

CREATE POLICY <name>
ON <CATALOG | SCHEMA | TABLE> <securable>     -- SCOPE: where attached (+ descendants)
  ROW FILTER <udf>                            -- ACTION (one of these two)
  | COLUMN MASK <udf> ON COLUMN <col>
TO `account users` EXCEPT `admins`       -- PRINCIPALS / exemptions
FOR TABLES                                  -- only supported target today
WHEN has_tag('HR')                         -- TABLE condition (optional)
MATCH COLUMNS has_tag_value('pii','ssn') AS c  -- COLUMN condition (up to 3)
ON COLUMN c
USING COLUMNS (c);                            -- args passed to the UDF
ClauseMeaning
ONScope — catalog/schema/table; applies to it and all descendants.
ROW FILTER / COLUMN MASKWhich protection.
TO … EXCEPTWho it applies to / who's exempt.
WHENTable condition (match tables by tag). Omit = all tables in scope.
MATCH COLUMNSColumn condition; up to 3 expressions, all must match.
USING COLUMNSPasses matched columns / constants to the UDF.

Matching functions: has_tag('key') and has_tag_value('key','value').

Syntax note: snake_case has_tag/has_tag_value is preferred; the camelCase hasTag/hasTagValue still works but is being deprecated for new policies.

5 Row filter vs column mask

The two policy types on profiles. Toggle the policy and switch who's asking.

NamePhoneAddress pii=addressSSN pii=ssn

Row filter — TRUE keeps, FALSE hides

-- hide EU rows ⇒ return FALSE for them
CREATE FUNCTION non_eu_filter(addr STRING)
RETURNS BOOLEAN
RETURN NOT (addr ILIKE '%europe%');

CREATE POLICY hide_eu_customers
ON SCHEMA customers ROW FILTER non_eu_filter
TO `account users` FOR TABLES
MATCH COLUMNS has_tag_value('pii','address') AS a
USING COLUMNS (a);

Column mask — value bound from ON COLUMN

CREATE FUNCTION mask_ssn(s STRING)
RETURNS STRING RETURN '***-**-****';

CREATE POLICY data_mask_ssn
ON SCHEMA customers COLUMN MASK mask_ssn
TO `account users` EXCEPT `admins` FOR TABLES
MATCH COLUMNS has_tag_value('pii','ssn') AS s
ON COLUMN s;
⚠️ Row-filter polarity: the UDF returns TRUE = keep, FALSE = hide. Naming a function "is_eu" and returning TRUE for EU rows would keep them.

6 Unset tags & drop policies

🏷️ Unset a tag — key only

-- name the KEY, never the value
ALTER TABLE customers.profiles
  ALTER COLUMN address UNSET TAGS ('pii');

-- equivalent statement form
UNSET TAG ON COLUMN cat.sch.profiles.address pii;

🛡️ Drop a policy — scope-matched

SHOW POLICIES ON SCHEMA cat.customers;

DROP POLICY data_mask ON SCHEMA cat.customers;
-- drop from the SAME level it was attached
-- (CATALOG / SCHEMA / TABLE)
⚠️ Dropping a tagged column fails while it still carries a governed tag — UNSET first, then drop. You can only unset one column per ALTER TABLE. SQL is the reliable, scriptable, auditable path.

Name mask — HR only

CREATE FUNCTION mask_name(name STRING)
RETURNS STRING
RETURN CASE WHEN is_account_group_member('hr_group')
  THEN name ELSE '***' END;

Salary mask — Finance only

-- return type must cast to DOUBLE
CREATE FUNCTION mask_salary(salary DOUBLE)
RETURNS DOUBLE
RETURN CASE WHEN is_account_group_member('finance_group')
  THEN salary ELSE NULL END;

8 Groups, users & access — two layers

ABAC decides what's masked. It does not grant the right to read the table. Two separate layers.

Layer 1 · Access

Can you open the table at all? Grant to groups, not users:

GRANT USE CATALOG ON CATALOG cat TO `hr_group`;
GRANT USE SCHEMA  ON SCHEMA  cat.customers TO `hr_group`;
GRANT SELECT      ON SCHEMA  cat.customers TO `hr_group`;

Layer 2 · ABAC

Which rows/columns once inside? Governed tags + policies + is_account_group_member() decide masking per group automatically.

⚠️ #1 gotcha: is_account_group_member() checks account-level membership and ignores workspace-local groups. If your groups aren't account-level (and synced), every user falls to the "Other" branch and masking looks broken.

10 Do's & don'ts (UDF performance)

The UDF runs for every row / value, so keep it cheap. Strong guidelines, not hard rules.

✅ Do

  • Keep UDFs simple — basic CASE / boolean logic
  • Reference only target columns (enables predicate pushdown)
  • External table? Keep it small enough to broadcast; partition it
  • Prefer SQL UDFs; mark DETERMINISTIC
  • Use try_cast/try_divide (error-safe)
  • Reuse one mask across columns

❌ Avoid

  • External API calls / cross-DB lookups
  • Complex subqueries / joins on large tables
  • Regex on large text (XML/JSON) fields
  • Multi-level nesting
  • Per-row metadata lookups (information_schema)
  • Non-deterministic funcs (rand(), now())
SecureView barrier: any policy on a table blocks side-effecting predicates from being pushed to storage, so WHERE func(col)=… forces a full scan while WHERE col='x' still prunes. Favor simple equality predicates; EXCEPT removes the barrier for exempt users. Test on ≥1M rows before production.

11 When to use ABAC vs RLS/CLS

Both sit on top of grants (neither grants access). The deciding factor is scale + governance.

Use ABAC when…

  • Many tables need consistent rules
  • Duties are separated (taggers vs policy authors)
  • Estate is growing — new tagged tables auto-covered
  • Admins must enforce rules owners can't bypass
  • CREATE POLICY … ON CATALOG/SCHEMA/TABLE

Use table-level RLS/CLS when…

  • Small, stable set of tables
  • Each table has bespoke logic that doesn't generalize
  • Table owners manage their own filters/masks
  • No central tag system desired
  • ALTER TABLE … SET ROW FILTER / SET MASK

They can coexist (still one filter/table & one mask/column per user). A third option — dynamic views — fits control spanning multiple source tables, but views lack tag-based auditing and the anti-probing barrier. One line: many tables + central governance → ABAC; few, bespoke, stable tables → table-level.

12 Gotchas master list

  • Row-filter polarity: TRUE = keep, FALSE = hide.
  • Columns don't inherit tags — tag each sensitive column directly.
  • Wrong tag key/value = silent miss — no error, data returns unprotected.
  • Unset uses the key only — UNSET TAGS ('pii'), never ('pii'='ssn').
  • Multiple tags on a column = comma tuple, not AND.
  • Untag before dropping a governed-tagged column.
  • Drop a policy from its exact scope; SHOW POLICIES / SHOW EFFECTIVE POLICIES to inspect.
  • is_account_group_member = account-level only; workspace-local membership ignored.
  • ABAC ≠ access — users still need SELECT + USE CATALOG/USE SCHEMA.
  • Mask UDFs must return a cast-compatible type (numeric → NULL/0.0, not a string).
  • One row filter per table, one mask per column per user — conflicts throw UC_ABAC_MULTIPLE_ROW_FILTERS/MULTIPLE_MASKS and block the table.
  • Quotas: 10 policies/catalog · 10/schema · 5/table · 20 principals/policy.
  • Prefer TO`/`EXCEPT over identity checks inside the UDF; CREATE OR REPLACE POLICY to re-run.
