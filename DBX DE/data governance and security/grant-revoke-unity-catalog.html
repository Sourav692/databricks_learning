GRANT & REVOKE in SQL

How to give and take away access to data — and why SELECT alone is never enough. Includes a live access simulator.

The simple idea

Think of your data platform as an office building. GRANT hands someone a key; REVOKE takes it back. You never delete the room — you just control who can walk in and what they can do once inside.

In Databricks these keys are called privileges, and the rooms are securable objects arranged in a hierarchy: catalog → schema → table.

Basic syntax

-- Give a privilege
GRANT <privilege> ON <object_type> <object_name> TO `<principal>`;

-- Take it back
REVOKE <privilege> ON <object_type> <object_name> FROM `<principal>`;

Three slots to fill: the privilege (what they can do), the object (what they can do it on), and the principal (who — a user email, a group, or a service principal). Wrap the principal in backticks because emails contain special characters. The only structural difference between the two statements is TO vs FROM.

The lesson: USE CATALOG and USE SCHEMA are like a building keycard — they let you reach the room but grant no data access. SELECT is the key to the room itself. You need the whole chain. (Shortcut: grant SELECT at the schema or catalog level to cover all current and future tables at once.)

Worked examples

-- Let a group create tables in a schema
GRANT CREATE TABLE ON SCHEMA sales.analytics TO `data-engineers`;

-- Give a single user read access to one table
GRANT SELECT ON TABLE sales.analytics.orders TO `sourav.banerjee@databricks.com`;

-- Give everything on a schema to a group
GRANT ALL PRIVILEGES ON SCHEMA sales.analytics TO `analytics-admins`;

-- Take read access away from one person
REVOKE SELECT ON TABLE sales.analytics.orders FROM `sourav.banerjee@databricks.com`;

Common mistakes & gotchas

“I granted SELECT but the user still can’t query”

Almost always missing USE CATALOG and/or USE SCHEMA. Access is hierarchical — see the simulator above. Run all three grants, or grant SELECT higher up the tree.

“GRANT CREATE ON SCHEMA throws an error”

The bare CREATE privilege is legacy Hive metastore syntax. In Unity Catalog it was split into specific privileges: CREATE TABLE, CREATE FUNCTION, CREATE VOLUME, CREATE MATERIALIZED VIEW, CREATE MODEL. Use GRANT CREATE TABLE ON SCHEMA ….

“What does ALL PRIVILEGES actually cover?”

Everything applicable to the object and its children — but it does not include MANAGE, EXTERNAL USE SCHEMA, or EXTERNAL USE LOCATION. Grant those explicitly when needed.

“REVOKE didn’t remove the user’s access”

Privileges inherit downward. REVOKE only removes a grant made at that level — it can’t cancel out a privilege inherited from a parent object. Revoke at the level where the grant was actually made.

“Should I grant to users or groups?”

Groups are the default — grant once, then add/remove people without rewriting SQL. Use individual users only for genuine one-offs (e.g. a stakeholder doing a quick ad-hoc analysis where a whole group isn’t worth creating).

Inspecting current access

-- See who has what on an object
SHOW GRANTS ON TABLE sales.analytics.orders;

-- See what a specific principal can do
SHOW GRANTS `analysts` ON SCHEMA sales.analytics;
Prerequisites: a Unity Catalog–enabled workspace and a SQL warehouse (or UC-enabled cluster). You can only grant privileges you have authority to manage — object owner, the MANAGE privilege, or metastore admin.
