Triggers, Schedules & Monitoring

How a job starts (Quartz CRON / file arrival / table update / continuous), recovers (retries, partial re-run), and reports (monitoring + notifications + health SLAs). This deep dive pairs each with bundle YAML.

Quartz CRONfile-arrivalretriesnotificationshealth SLA

The one-liner

Triggers start a run (Quartz CRON, file arrival, table update, continuous, periodic, manual); retries & timeouts recover from blips (re-run only failed tasks); monitoring + notifications + health rules tell you what happened (email / webhook / Slack / SLA breach).
⏰ Analogy: triggers are the alarm clock (time) or doorbell (an event); retries are "try the key again if the door sticks"; notifications are the text message telling you it worked or broke.

Continuous — auto-restarts on completion/failure for always-on jobs (compute stays up → cost).
Periodic — fixed interval (trigger.periodic.interval/unit), an alternative to cron.
Manual — "Run now" or via external orchestration / API.

1 · Schedule & triggers as code

schedule:
  quartz_cron_expression: "0 0 2 * * ?"   # 02:00 daily (sec min hr dom mon dow)
  timezone_id: "America/New_York"       # DST-aware; "UTC" avoids DST drift
  pause_status: UNPAUSED                  # or PAUSED
# event-driven instead of cron:
trigger:
  file_arrival: { url: "/Volumes/main/raw/orders/" }
# or: trigger: { periodic: { interval: 6, unit: HOURS } }

2 · Reliability — retries, timeouts, concurrency

timeout_seconds: 7200            # job kill-switch (2h)
max_concurrent_runs: 1           # default 1; raise only if overlap is safe
tasks:
  - task_key: ingest
    max_retries: 2                       # retry transient failures twice
    min_retry_interval_millis: 60000     # wait 60s between retries
    retry_on_timeout: true               # also retry on task timeout
    notebook_task: { notebook_path: ../src/ingest.py }
💡 Retries fix transient failures (flaky API, lost node), not deterministic bugs. On a fix, re-run only the failed tasks (repair run), not the whole job.

3 · Notifications & health (SLA)

email_notifications:
  on_failure: ["oncall@acme.com"]
  on_duration_warning_threshold_exceeded: ["oncall@acme.com"]
webhook_notifications:
  on_failure: [{ id: "${var.slack_webhook_id}" }]   # Slack / PagerDuty
health:
  rules:
    - metric: RUN_DURATION_SECONDS      # SLA: warn if a run exceeds 1h
      op: GREATER_THAN
      value: 3600

Health rules turn "it's slow" into an alert, not a silent miss. Monitor runs in the Runs tab and trend health/cost via system.lakeflow.* system tables.

Why it matters

  • A pipeline that doesn't start automatically, recover from blips, and alert on failure isn't production-ready — this is the ops layer.
  • "Run when files land?" → file-arrival; "daily at 2am?" → Quartz CRON; "alert on failure / SLA?" → notifications + health rules.
🌙 Real-world use case: a gold pipeline runs on Quartz CRON 0 0 2 * * ?, retries transient failures twice, re-runs only failed tasks on manual fix, and emails + Slack-webhooks the on-call if the run fails or breaches its duration SLA.

Uses, edge cases & limitations

  • Uses: scheduled batch, event-driven ingestion, always-on jobs, alerting, partial re-runs after a fix, duration SLAs.
  • Edge cases: default max_concurrent_runs = 1 (a long run can make the next scheduled run skip/queue); Quartz CRON is in the job's timezone (DST/TZ bugs — use UTC); file-arrival watches UC locations only.
  • Limitation: continuous jobs keep compute running (cost); retries fix transient failures, not deterministic bugs.

Common gotchas

  • ❌ Using 5-field Unix cron — Databricks uses 6-field Quartz (0 0 2 * * ?).
  • ❌ Re-running the whole job after one task fails — re-run only the failed tasks.
  • ❌ Wrong CRON timezone → off-by-hours schedules (watch DST).
  • ❌ Expecting overlapping runs by default — max_concurrent_runs is 1 unless raised.
  • ❌ No failure notification / health rule → silent failures nobody notices.
  • ❌ Relying on retries to mask a deterministic bug.
