Lakeflow Connect — Managed & Standard Connectors

Built-in, mostly no-code ingestion from SaaS apps and databases — governed by Unity Catalog, serverless, built on Lakeflow Declarative Pipelines. This deep dive pairs the UI flow with config-as-code.

Managed connectorsCDC + gatewayUC connectionAsset Bundle

The one-liner

Managed connectors = fully-managed, UI-based ingestion from Salesforce, Workday, ServiceNow, SQL Server, MySQL, Postgres… Standard/community connectors reach a wider range of sources from inside your pipelines. Managed = least code, built-in monitoring; reproducible as an Asset Bundle.
✈️ Analogy: managed connectors are direct flights — Databricks runs the whole trip with monitoring & recovery built in. Standard/community connectors are connecting flights you book yourself — more reach, but you wire up more of the journey.

1 · The UI flow (Salesforce example)

  1. Data Ingestion in the sidebar → choose Salesforce.
  2. Create or select a connection (OAuth) — a UC securable storing auth.
  3. Pick destination catalog/schema and the objects (e.g. Account, Opportunity).
  4. Create the ingestion pipeline (a Lakeflow SDP) and set a schedule.
💡 Database connectors add one step: an ingestion gateway + staging schema for CDC before the pipeline.

2 · The same thing as code (CI/CD)

UC connection (databases via SQL; SaaS usually via UI OAuth)

CREATE CONNECTION sqlserver_conn TYPE sqlserver
OPTIONS (host 'db.corp.net', port '1433',
         user secret('kv','sql_user'), password secret('kv','sql_pwd'));

Ingestion pipeline as a Databricks Asset Bundle

# databricks.yml
resources:
  pipelines:
    pipeline_sfdc:
      name: salesforce_ingest
      catalog: main                      # destination catalog
      schema: bronze                     # destination schema
      ingestion_definition:
        connection_name: salesforce_conn  # the UC connection
        objects:
          - table:
              source_schema: objects
              source_table: Opportunity    # Salesforce object → bronze table
              destination_catalog: main
              destination_schema: bronze

Deploy, run & schedule (CLI)

databricks bundle deploy -t prod        # create/update the pipeline
databricks bundle run pipeline_sfdc     # trigger an ingestion run
jobs:
  sfdc_daily:
    trigger: { periodic: { interval: 1, unit: DAYS } }
    tasks:
      - task_key: ingest
        pipeline_task: { pipeline_id: ${resources.pipelines.pipeline_sfdc.id} }

Why it matters

  • Hand-building API ingestion (auth, pagination, CDC, retries, schema) is weeks of brittle work; managed connectors make it point-and-click with monitoring & recovery.
  • "How would you ingest Salesforce data?" → "Lakeflow Connect managed connector" is the modern expected answer.
💼 Real-world use case: finance needs Salesforce opportunities daily. A managed connector ingests them incrementally into bronze — no custom API code, UC-governed, monitored automatically, and captured as a bundle for CI/CD.

Uses, edge cases & limitations

  • Uses: pulling enterprise SaaS/db data (CRM, HR, ITSM, OLTP DBs) into bronze without custom code; CDC from operational databases; CI/CD via Asset Bundles.
  • Edge case: database connectors need an ingestion gateway + staging (heavier than SaaS); sources without a managed connector fall back to standard/community; SaaS auth is OAuth via the UI.
  • Limitation: managed connectors cover a specific (growing) list; community connectors are best-effort. Verify current source availability in the docs before promising it.

Common gotchas

  • ❌ Assuming every SaaS/db has a managed connector — verify the supported list.
  • ❌ Forgetting database connectors require an ingestion gateway.
  • ❌ Hand-rolling API ingestion when a managed connector already exists.
  • ❌ Confusing managed (Databricks-maintained) with community (open-source, best-effort) connectors.
  • ❌ Clicking the pipeline together in the UI for prod and never capturing it as a bundle — you lose CI/CD.
