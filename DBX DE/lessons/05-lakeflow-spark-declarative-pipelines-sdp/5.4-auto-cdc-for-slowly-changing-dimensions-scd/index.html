AUTO CDC — SCD Type 1 & 2

Apply a change feed to build slowly changing dimensions declaratively. SCD1 overwrites; SCD2 keeps versioned history with start/end timestamps. (Replaces APPLY CHANGES INTO.) This deep dive walks each with the code.

AUTO CDCSCD 1 / 2SEQUENCE BYAPPLY AS DELETEparametrized

The one-liner

AUTO CDC applies inserts/updates/deletes to a target by KEYS + SEQUENCE BY, STORED AS SCD TYPE 1 (overwrite) or TYPE 2 (history with __START_AT/__END_AT). It replaces APPLY CHANGES INTO.
🏠 Analogy: tracking a customer's address. SCD1 = a phone contact — editing the address erases the old one. SCD2 = an address-history logbook — each move adds a dated entry.

1 · The mechanism & SCD1 vs SCD2

KEYS = the unique business key; SEQUENCE BY = a sortable column whose highest value wins regardless of arrival order — that's what makes AUTO CDC robust to late/out-of-order events vs a naive MERGE.

SCD Type 1SCD Type 2
Historynone (overwrite)full (versioned rows)
Extra columns—__START_AT, __END_AT
Use for"current value only" dimsaudited / point-in-time joins
Table growthflatgrows (row per change per key)

2 · SQL — AUTO CDC INTO

CREATE OR REFRESH STREAMING TABLE dim_customers;

CREATE FLOW cdc_flow AS AUTO CDC INTO dim_customers
FROM stream(customers_changes)
KEYS (customer_id)
APPLY AS DELETE WHEN operation = 'DELETE'      -- handle deletes explicitly
SEQUENCE BY change_ts                            -- ordering; late events resequenced
COLUMNS * EXCEPT (operation)
STORED AS SCD TYPE 2                             -- or SCD TYPE 1 (overwrite)
TRACK HISTORY ON * EXCEPT (last_seen_ts);        -- skip versioning noisy cols

3 · Python — create_auto_cdc_flow

from pyspark import pipelines as dp
from pyspark.sql.functions import col, expr

dp.create_streaming_table("dim_customers")

dp.create_auto_cdc_flow(                          # replaces legacy dlt.apply_changes
    target="dim_customers",
    source="customers_changes",
    keys=["customer_id"],
    sequence_by=col("change_ts"),
    stored_as_scd_type=2,                          # 1 or 2
    apply_as_deletes=expr("operation = 'DELETE'"))

Sources that arrive as full snapshots (Python-only)

dp.create_auto_cdc_from_snapshot_flow(
    target="dim_customers", source="daily_snapshot",
    keys=["customer_id"], stored_as_scd_type=2)

5 · Pipeline modes & monitoring — click each

Triggered (batch)▸
Runs to completion, then stops. Cheaper — compute spins down between runs. Use for scheduled batch updates.
Continuous (streaming)▸
Keeps running for low-latency updates. Higher cost (compute stays up). Use when freshness matters.
Monitoring▸
The pipeline UI shows the DAG, per-dataset row counts, expectation (data-quality) metrics, and an event log for debugging failures.

Why it matters

  • Hand-coding SCD2 MERGE logic (close old row, open new, handle late data) is notoriously error-prone — AUTO CDC does it declaratively.
  • "Implement SCD Type 2" is a classic interview task — on Databricks the answer is AUTO CDC … STORED AS SCD TYPE 2.
🗺️ Real-world use case: a customers change feed builds dim_customers as SCD2 — every address change is versioned with start/end timestamps, so historical orders join to the address current at the time.

Uses, edge cases & limitations

  • Uses: dimension tables from CDC feeds; SCD1 for "current only", SCD2 for audited history; create_auto_cdc_from_snapshot_flow to derive changes by comparing snapshots; parametrized pipelines for multi-env reuse.
  • Edge cases: SEQUENCE BY must be sortable (else wrong winner on out-of-order data); deletes need APPLY AS DELETE WHEN or they're missed; TRACK HISTORY ON * versions on every column change — exclude noisy columns.
  • Limitation: triggered vs continuous trades latency for cost; SCD2 tables grow over time; FROM SNAPSHOT is Python-only.

Common gotchas

  • ❌ Using SCD1 when the business needs history → use SCD2.
  • ❌ Bad/omitted SEQUENCE BY → late events overwrite newer data.
  • ❌ Forgetting APPLY AS DELETE WHEN → deletes in the feed are ignored.
  • ❌ Running continuous when triggered would do → wasted cost.
  • ❌ Re-implementing SCD2 with manual MERGE, or using legacy dlt.apply_changes — prefer AUTO CDC / create_auto_cdc_flow.
