Idempotency & Schema Evolution

Exactly-once ingestion, and what happens when a file shows up with new columns — compare rescue vs addNewColumns side by side.

1. Idempotency — "exactly once"

Every file is processed once and only once. As files are discovered, Auto Loader records each one in RocksDB, a key-value store inside the checkpoint location. Each run consults it, skips processed files, and reads only what's new — which is why row counts grow incrementally (2 → 4 → 5) instead of reprocessing everything.

Gotcha: this memory lives in the checkpoint. Delete the checkpoint and Auto Loader forgets what it processed — it re-reads from the beginning. (That's exactly what "drop the folder / use destination_two" did in the demo: a deliberate reset.)

2. Schema Evolution — the fork in the road

The trigger is always the same: a new file contains a column not in Auto Loader's cached schema. What happens next is decided entirely by cloudFiles.schemaEvolutionMode. Toggle the two modes below and drop the evolved file.

4. Why only addNewColumns needs mergeSchema

addNewColumns literally changes the destination Delta table's column set, so the sink must be allowed to evolve → .option("mergeSchema","true"). rescue never changes the table's columns (everything lands in the pre-existing _rescued_data string), so there is nothing to merge.

5. Reading back _rescued_data

It is a STRING column holding a JSON blob (the rescued fields + the source file path) — it only looks like a struct. Parse it with from_json and a schema you define, then pull out the fields you need:

from pyspark.sql.functions import from_json, col from pyspark.sql.types import StructType, StructField, DoubleType, StringType rescued_schema = StructType([ StructField("discount", DoubleType(), True), StructField("payment_method", StringType(), True), ]) df_out = (df .withColumn("r", from_json(col("_rescued_data"), rescued_schema)) .withColumn("discount", col("r.discount")) .withColumn("payment_method", col("r.payment_method")) .drop("r"))

You only do this when you actually need those columns — the appeal of rescue is that nothing unexpected pollutes your table.

Three corrections from the walk-through: the exception is UnknownFieldException (not "unknown file exception"); _rescued_data is a STRING of JSON, not a struct (hence from_json); and the "can be called only on streaming DataFrame" break in the rescue demo was a notebook-state slip — pure rescue mode does not stop the stream.
