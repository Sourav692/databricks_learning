Auto Loader — Concept & Complete Architecture

Incremental, exactly-once file ingestion from cloud storage — built on Spark Structured Streaming, with the bookkeeping handled for you.

1. What Auto Loader Is

Auto Loader is a Databricks feature that incrementally and efficiently ingests files from cloud storage (S3, ADLS, GCS) into Databricks as they arrive — without re-reading files it has already processed. You invoke it through the cloudFiles source format.

In one technical sentence: it is built on top of Spark Structured Streaming (Spark's engine for processing data continuously as it arrives), and it automates the state, checkpoints, and new-file tracking you would otherwise have to manage by hand. You worry about the data; Auto Loader manages everything else.

2. The Three Pillars

Master these three ideas and the code becomes easy.

Pillar 1 · Idempotency

Exactly once

Each file is processed once and only once. New-file metadata is tracked in RocksDB, a fast key-value store kept inside the checkpoint location. Each run consults it and skips processed files.

Pillar 2 · Schema Inference

Sample then store

On first run it infers the schema by sampling the first 50 GB or 1,000 files, whichever comes first, and saves it in a _schemas folder at cloudFiles.schemaLocation.

Pillar 3 · Schema Evolution

Adapt, don't drop

When a new column appears, Auto Loader infers it, updates the schema location, then stops with UnknownFieldException. A restart reloads the schema and continues.

4. End-to-End Flow (in words)

Day 1 — one file. Auto Loader reads it, infers the schema and saves it to _schemas, writes rows to the destination, and records the file in RocksDB inside the checkpoint.

Day 2 — a second file. RocksDB shows file 1 is done, so it is skipped. Only file 2 is read; schema matches; rows are written; file 2 is recorded.

Day 3 — a file with a new column. Earlier files are skipped. Auto Loader infers the new column, appends it to the schema location, then compares against its in-memory cached schema, finds a mismatch, and stops with UnknownFieldException. You restart; it reloads the updated schema and continues normally.

The key insight: Auto Loader updates the schema location before it throws the error — that is exactly why the restart succeeds.

5. Schema Evolution Modes

Set via cloudFiles.schemaEvolutionMode.

ModeBehavior when a new column appears
addNewColumnsDefault when no schema is supplied. Adds the column, stops the stream with UnknownFieldException; restart to continue.
rescueNever fails. Unknown columns are captured in _rescued_data instead of evolving the schema.
failOnNewColumnsStream fails and does not update the schema; fix the schema or hints manually.
noneIgnores new columns entirely; nothing rescued, nothing evolved.
addNewColumns is not allowed when you explicitly provide a schema (then the default becomes none) — but it does work if you supply your schema as a schema hint.

6. Two Locations People Confuse

Checkpoint = what's been processed. Schema location = what the data looks like.

7. Common Mistakes / Gotchas

  • Deleting the checkpoint → Auto Loader forgets processed files and reprocesses everything (breaks exactly-once).
  • Expecting the stream never to stop on addNewColumns → it stops once per new column; design for automatic restart (jobs / Lakeflow pipelines do this for you).
  • Providing a fixed schema and expecting evolution → the default mode becomes none; use schema hints if you want evolution plus type control.
  • Assuming columns get typed → inferred columns default to string; use cloudFiles.inferColumnTypes or schema hints for real types.
  • Forgetting _rescued_data → it silently catches mismatched / extra data; check it or you may "lose" records you didn't realize were rescued.
