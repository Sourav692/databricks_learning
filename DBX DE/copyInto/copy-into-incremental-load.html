Incremental Load
with COPY INTO

A SQL command that loads files from cloud storage into a Delta table — and remembers what it already loaded, so re-runs pick up only the new files.

Idempotent  re-runnable Delta  target table SQL  native Docs updated 2026-05-08

01The plain-English idea

COPY INTO loads files from a folder in cloud storage into a Delta table. Its superpower: it keeps a logbook of which files it has already loaded. Run it again and it skips the old files, ingesting only what's new.

Picture a mailroom clerk with a logbook. Each time letters (files) arrive, the clerk checks the book, delivers only the unlogged letters, and updates the book. Run it every morning and no letter is ever delivered twice. That repeatable, no-duplicates behavior is what we call an incremental load.

Idempotent means running the command many times has the same result as running it once. Already-loaded files are skipped — even if the file was modified after it was first loaded.

Note: COPY INTO only appends. With force=true it reloads files it already loaded, which is why the row count can grow past the number of unique source rows — that's how duplicates happen.

03How it works & the syntax

Each run does four things: list the source files → compare against the load history in the Delta log → read & append only the unseen files → record the newly loaded files. The target must already exist as a Delta table; COPY INTO appends into it.

-- Re-run this on a schedule; only new files get loaded.
COPY INTO databricksansh.bronze.orders_copyinto
  FROM '/Volumes/databricksansh/bronze/autovol/copyinto_demo'
  FILEFORMAT = PARQUET            -- CSV|JSON|AVRO|ORC|PARQUET|TEXT|BINARYFILE
  [ VALIDATE [ ALL | n ROWS ] ]   -- dry-run (DBR 10.4 LTS+)
  [ FILES = (...) | PATTERN = 'glob' ]
  [ FORMAT_OPTIONS ('reader option' = 'value') ]
  [ COPY_OPTIONS ('force' = 'false', 'mergeSchema' = 'false') ]
FORMAT_OPTIONS vs COPY_OPTIONS. FORMAT_OPTIONS are passed to the Spark data-source reader (how each file is parsed). COPY_OPTIONS control COPY INTO itself (its load behavior). They are not interchangeable.

04The options that matter

Option / clauseDefaultWhat it does
COPY_OPTIONS('force')falseSet true to disable idempotency — reload all matching files, ignoring history. Appends, so it can duplicate rows.
COPY_OPTIONS('mergeSchema')falseSet true to evolve the target table schema (add new columns) as incoming data changes.
FORMAT_OPTIONS('mergeSchema')falseSpark reader option that reconciles schemas across the source files (e.g. Parquet part files that differ).
PATTERN = 'glob'—Load only files matching a wildcard (e.g. '*.parquet'). Can't combine with FILES.
FILES = (...)—Explicit list of file names, up to 1000.
VALIDATE [n ROWS]ALLDry-run (DBR 10.4 LTS+): checks parsing, schema & constraints and previews data without writing.
SELECT ... FROM '<path>'—Transform on the way in: cast, rename, add columns, or capture _metadata.file_path.

05COPY INTO vs Auto Loader

COPY INTOAuto Loader (cloudFiles)
StyleSQL command, batchStructured Streaming source
Best forPeriodic batches, up to ~thousands of filesContinuous ingestion; millions of files over time
Tracks "new files" viaDelta load history on the tableCheckpoint (+ optional file-notification queue)
Schema evolutionmergeSchema optionBuilt-in, richer (schemaEvolutionMode)
Mental model"Run it on a schedule""Leave it running / trigger it"

Rule of thumb: COPY INTO for simple scheduled batch loads; switch to Auto Loader when file volume gets large or you want near-real-time, hands-off ingestion.

06Common gotchas

GotchaWhat to know
Two mergeSchema optionsOne evolves the target table (COPY_OPTIONS), the other reconciles source files (FORMAT_OPTIONS). Different jobs; sometimes you need both.
force=true duplicates rowsCOPY INTO only appends; it never de-duplicates. Truncate first, or use MERGE for upserts.
Target must existIt must already be a Delta table — COPY INTO won't create it for you.
Modified files are still skippedIdempotency is by file identity, not contents. Re-saving the same path won't reload it — use force.
Root paths need a trailing slashe.g. s3://my-bucket/.
