What it is

Zerobus Ingest is a push-based ingestion API that writes data directly into Unity Catalog Delta tables. A serverless connector that automatically scales to handle incoming connections — no partitions to configure, no brokers to manage.

Sits under Lakeflow Connect alongside managed connectors (Salesforce, SQL Server, etc.) and Auto Loader — but it's the only one designed for app-emitted, high-volume events rather than pulling from a source.

The problem it solves

The classic pattern to land app events in the lakehouse is:

App → Kafka / Kinesis / EventHub → Structured Streaming or DLT → Delta

That's three systems to operate, scale, and bill for, just to get bytes into a table. Zerobus collapses the message bus and the streaming job into a single serverless service. Your app talks to Databricks directly.

Mental model

Your "scaling strategy" with Zerobus is to open more streams. One stream guarantees ordering. Many streams give you horizontal throughput.

How it works (4 steps)

  1. Client opens a stream to the Zerobus Ingest API and specifies a target Delta table.
  2. Client constructs a message that matches the table schema and pushes it.
  3. Service buffers, validates schema, durably persists the message, and acks the client.
  4. Service materializes the data in the Delta table in optimized batches.

Ordering guarantee

Ordering is guaranteed per stream, not across streams. If your client round-robins messages across multiple streams, ordering is lost.

Schema safety net — Durable Fallback Location

If a breaking schema change happens to the target table while Zerobus has data in flight, that data is written as Parquet to _zerobus/table_rejected_parquets/ under the table root. Nothing is lost; it stays inside the table's storage boundary and inherits its access controls.

Interfaces

gRPC (with SDKs in Python, Java, Rust, Go, TypeScript), REST (Beta), and OpenTelemetry / OTLP (Beta — not billed at this time).

Monitoring

Two system tables: Zerobus Ingest system table for usage, and billable usage system table filtered by billing_origin_product = 'LAKEFLOW_CONNECT'.

When an Architect Picks It

Pick Zerobus

  • App telemetry, clickstream, IoT, observability going directly to the lakehouse
  • High fan-in from many producers with variable throughput
  • OpenTelemetry pipelines (traces, logs, metrics via OTLP)
  • You want the simplest path from app → Delta — no message bus
  • The lakehouse is the only real-time destination
  • Greenfield event pipeline with no Kafka already in place

Don't Pick Zerobus

  • You need a true pub/sub with multiple independent consumers
  • Source is an existing DB or SaaS — use Lakeflow Connect managed connector (CDC)
  • Source is files in cloud storage — that's Auto Loader's job
  • You already run Kafka happily — don't rip it out
  • You need replay or time-travel on the stream itself
  • Single-digit ms end-to-end latency required (trading, fraud)

Why Zerobus ≠ Pub/Sub

Zerobus is a one-way pipe: producer → Delta table. Once data lands in Delta, it's at rest. It is not a streaming bus that multiple services can subscribe to in real time.

Kafka, Kinesis, EventHubs, Pub/Sub are brokers. They sit between producers and N consumers and let each consumer:

  • Read the same event independently (different consumer groups / shards)
  • Maintain its own offset / cursor — replay from any point
  • Apply backpressure without affecting other consumers
  • Receive the event in milliseconds, often before anyone else has seen it

Zerobus deliberately removes the broker. The price: there's nothing for other consumers to subscribe to.

Concrete scenarios where you still need Kafka/Kinesis

1

Multiple real-time consumers

An order_placed event feeds fraud detection, inventory, notifications, and the lakehouse. With Kafka all four subscribe; with Zerobus only the lakehouse gets the data.

2

Event-driven microservices

Services communicate via events on a bus. The lakehouse is one subscriber among many — often added later as an analytics tap. Kafka is the system of record for the event itself.

3

Replay / time-travel

Kafka retains events for days/weeks. A new service can deploy and replay history. Zerobus has no replay API — data sits in Delta but it's not a rewindable stream.

4

Cross-cloud / cross-region fan-out

Kafka MirrorMaker, Kinesis cross-region replication — these solve "the same event must reach four regions and two clouds." Zerobus terminates in one Unity Catalog table.

5

Sub-100ms end-to-end latency

Zerobus is P95 ≤ 500 ms to Delta durability. Trading or fraud systems reacting to events need single-digit ms broker→consumer. That's Kafka territory.

6

Existing Kafka ecosystem

Schema Registry, ksqlDB, Flink jobs, existing topics, ops muscle. Ripping that out for Zerobus saves nothing and breaks every non-lakehouse consumer.

Rule of thumb

If more than one system needs the event in real time, you need a broker. Zerobus is for "the lakehouse is the only real-time consumer."

Interview Framing

The one-liner
"Zerobus is Databricks saying: if the only reason you're running Kafka is to get app events into Delta, you don't need Kafka anymore — push straight to a Unity Catalog table over gRPC, serverless, with sub-second durability."
The pub/sub clarifier
"Zerobus replaces Kafka only when Kafka was being used solely as a transport to land events in the lakehouse. If Kafka is also serving real-time microservices, replay, multi-consumer fan-out, or cross-region distribution, those are pub/sub responsibilities Zerobus does not take on. In those cases the right pattern is Kafka and Zerobus side-by-side, or Kafka with a Structured Streaming bronze job — Zerobus doesn't enter the picture."

Three questions to ask a customer before recommending Zerobus

  1. Who else, besides analytics, needs this event stream in real time?
  2. Do you control the producer (your app) or is this coming from a SaaS / DB / file source?
  3. Is sub-100ms latency a hard requirement, or is sub-second to Delta good enough?
